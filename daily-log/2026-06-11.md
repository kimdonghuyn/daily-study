## 모범 답안

### 메인 질문 답변

**AOP가 없었을 때의 상황:**

AOP 없이 반복적인 작업을 처리하려면 각 메서드마다 수동으로 코드를 삽입했어야 합니다.

```java
// AOP 없이 각 메서드에 반복적으로 작성
public class UserService {
    
    public User createUser(User user) {
        // 1. 로깅
        logger.info("Creating user: {}", user.getName());
        
        // 2. 보안 검사
        if (!securityManager.hasPermission("CREATE_USER")) {
            throw new UnauthorizedException();
        }
        
        // 3. 트랜잭션 시작
        transaction.begin();
        
        try {
            // 실제 비즈니스 로직 (5%)
            User savedUser = userRepository.save(user);
            
            // 4. 트랜잭션 커밋
            transaction.commit();
            return savedUser;
        } catch (Exception e) {
            // 5. 트랜잭션 롤백
            transaction.rollback();
            throw e;
        }
    }
    
    public User updateUser(User user) {
        // 동일한 코드 반복... (DRY 원칙 위반)
        logger.info("Updating user: {}", user.getId());
        if (!securityManager.hasPermission("UPDATE_USER")) {
            throw new UnauthorizedException();
        }
        transaction.begin();
        // ...
    }
    
    public void deleteUser(Long id) {
        // 또 반복... (유지보수의 악몽)
        logger.info("Deleting user: {}", id);
        if (!securityManager.hasPermission("DELETE_USER")) {
            throw new UnauthorizedException();
        }
        transaction.begin();
        // ...
    }
}
```

**발생하는 문제들:**

1. **코드 중복 (Code Duplication)**
   - 같은 로직이 100개 메서드에 100번 반복
   - 유지보수 난제: 로깅 형식 변경 → 100곳 수정 필요

2. **관심사 분리 위반 (Separation of Concerns)**
   - 비즈니스 로직 5줄 vs. 횡단 관심사 코드 20줄
   - 메서드의 실제 목적이 흐려짐

3. **오류 가능성 증가**
   - 어느 메서드는 보안 검사 빠뜨림
   - 어느 메서드는 로깅 형식이 다름
   - 버그 고정 시 일관성 없음

4. **가독성 악화**
   - 비즈니스 로직을 찾기 어려움
   - 메서드가 너무 길어짐

### 꼬리 질문 답변

**AOP 사용 시 코드 관리 이점:**

```java
// AOP 적용 후: Aspect에 한 번만 정의
@Aspect
@Component
public class LoggingAspect {
    
    @Around("@annotation(Loggable)")
    public Object logExecution(ProceedingJoinPoint joinPoint) throws Throwable {
        logger.info("Executing: {}", joinPoint.getSignature());
        long startTime = System.currentTimeMillis();
        
        try {
            Object result = joinPoint.proceed();
            long duration = System.currentTimeMillis() - startTime;
            logger.info("Completed in {}ms", duration);
            return result;
        } catch (Exception e) {
            logger.error("Exception: {}", e.getMessage());
            throw e;
        }
    }
}

@Aspect
@Component
public class SecurityAspect {
    
    @Before("@annotation(RequireAuth)")
    public void checkSecurity(JoinPoint joinPoint) {
        if (!securityManager.hasPermission()) {
            throw new UnauthorizedException();
        }
    }
}

@Aspect
@Component
public class TransactionAspect {
    
    @Around("@annotation(Transactional)")
    public Object manageTransaction(ProceedingJoinPoint joinPoint) throws Throwable {
        transaction.begin();
        try {
            Object result = joinPoint.proceed();
            transaction.commit();
            return result;
        } catch (Exception e) {
            transaction.rollback();
            throw e;
        }
    }
}

// 이제 비즈니스 메서드는 깔끔함
@Service
public class UserService {
    
    @Loggable
    @RequireAuth
    @Transactional
    public User createUser(User user) {
        // 순수 비즈니스 로직만!
        return userRepository.save(user);
    }
    
    @Loggable
    @RequireAuth
    @Transactional
    public User updateUser(User user) {
        return userRepository.update(user);
    }
}
```

**관리 측면의 이점:**

| 측면 | AOP 없음 | AOP 사용 |
|------|---------|---------|
| **변경점** | 100개 메서드 수정 필요 | 1개 Aspect만 수정 |
| **일관성** | 휴먼 에러 가능 | 자동으로 통일됨 |
| **가독성** | 비즈니스 로직 찾기 어려움 | 비즈니스 로직만 명확 |
| **테스트** | 횡단 관심사 테스트 복잡 | 분리된 테스트 가능 |
| **확장성** | 새로운 기능 추가 시 기존 코드 수정 필요 | 새 Aspect 추가만으로 확장 |

## 핵심 개념 정리

### 📌 횡단 관심사 (Cross-Cutting Concerns)
- **정의**: 여러 메서드/클래스에 걸쳐 반복되는 기능
- **예시**: 로깅, 보안, 트랜잭션, 캐싱, 성능 모니터링
- **특징**: 비즈니스 로직과 무관하게 기술적으로 필요한 것들

### 📌 AOP의 핵심 용어

```
┌─────────────────────────────────────┐
│  Aspect: 횡단 관심사를 구현한 클래스  │
├─────────────────────────────────────┤
│  JoinPoint: 메서드 실행 시점         │
│  Pointcut: JoinPoint를 선택하는 조건 │
│  Advice: 실제 실행될 코드            │
│    - @Before: 메서드 실행 전         │
│    - @After: 메서드 실행 후          │
│    - @Around: 전후 모두 제어         │
│    - @AfterReturning: 정상 반환 후   │
│    - @AfterThrowing: 예외 발생 시    │
└─────────────────────────────────────┘
```

### 📌 Spring AOP vs AspectJ

| 항목 | Spring AOP | AspectJ |
|------|-----------|---------|
| **처리 방식** | 프록시 기반 (런타임) | 바이트코드 조작 (컴파일) |
| **성능** | 약간 느림 | 빠름 |
| **학습곡선** | 쉬움 | 어려움 |
| **Spring 통합** | 완벽 | 별도 설정 필요 |
| **사용** | Spring 권장 | 복잡한 경우 |

### 📌 DRY 원칙 (Don't Repeat Yourself)
- AOP는 DRY 원칙의 완벽한 구현
- 변경점을 최소화 → 버그 가능성 감소 → 유지보수 비용 절감
