## 💬 내 답변

1. 잔액은 우선 private로 두고 직접적으로 수정이 불가능하게 하고, 해당 잔액을 조회 할땐 getter 메서드를 public 으로 두고 조회를 하도록 하며 만약 수정을 한다면 수정 메서드를 public 으로 두되 정확한 검증을 통과해야지만 되도록 할 것 같다.
2. 수정 권한을 제한하는거지 조회 권한을 제한하는게 아니라서 그냥 getter 메서드를 통해서 조회하면 된다.

## 🤖 AI 피드백

✅ 잘한 점 (구체적으로)
- 캡슐화의 핵심 개념을 정확히 이해하고 있습니다. private으로 필드를 보호하고, public 메서드로 제어하는 기본 원칙을 올바르게 설명했습니다.
- getter/setter 패턴의 필요성과 검증 로직의 중요성을 인식하고 있습니다. "정확한 검증을 통과해야지만"이라는 표현에서 비즈니스 로직 보호 의식이 보입니다.
- 꼬리 질문에 대해 조회와 수정 권한을 구분하는 논리적 사고를 보여주었습니다.

⚠️ 보완할 점 (구체적으로)
- 구체적인 검증 내용이 부재합니다. "정확한 검증"이 무엇인지 예시가 필요합니다. 예를 들어, "잔액은 0 이상이어야 한다", "거래액이 현재 잔액을 초과할 수 없다", "트랜잭션 기록이 남아야 한다" 등을 언급했다면 더 깊이 있었을 것입니다.
- 접근 제어자의 세분화가 부족합니다. private/public만 언급했는데, protected나 package-private(default) 같은 다른 수준의 접근 제어와 실무에서의 사용 사례를 언급했으면 좋았습니다.
- 불변성(Immutability) 관점이 없습니다. getter에서 원본 객체를 그대로 반환하면 안 되는 경우들(예: List, Date 같은 mutable 객체)에 대한 고려가 부족합니다.

💡 핵심 포인트 (놓친 부분)
- 실무 수준의 답변이 되려면: 검증 예시 + 트랜잭션 로깅 + 예외 처리 + 동시성 고려(synchronized, volatile) 같은 추가 요소를 언급했으면 시니어 개발자답습니다.
- 예제 코드 제시: 구두 설명보다 간단한 코드(withdraw/deposit 메서드에서의 검증 로직)를 보여주면 훨씬 설득력이 있습니다.
- 비즈니스 영향: "잔액이 음수가 되는 것"뿐만 아니라 "금융 규정 준수", "감사 추적", "동시 거래 처리"까지 언급했으면 시니어 관점입니다.

# 캡슐화와 접근 제어자 - 모범 답안

## 모범 답안

### 1단계: 문제 상황 인식
먼저 이 문제의 핵심은 **데이터 무결성 보호**입니다. 사용자 잔액은 비즈니스 로직상 중요한 데이터이므로, 임의의 변경을 방지해야 합니다.

### 2단계: 구체적 해결 방안

```java
// ❌ 잘못된 예: 데이터가 노출됨
public class User {
    public double balance;  // 누구나 직접 접근 가능
}

// ✅ 올바른 예: 캡슐화를 통한 접근 제어
public class User {
    // 1. private으로 데이터 보호
    private double balance;
    
    // 2. 읽기만 허용 (getter)
    public double getBalance() {
        return balance;
    }
    
    // 3. 검증 로직이 포함된 쓰기 (setter)
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("입금액은 0보다 커야 합니다");
        }
        this.balance += amount;
    }
    
    public boolean withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("출금액은 0보다 커야 합니다");
        }
        if (amount > this.balance) {
            throw new InsufficientBalanceException("잔액이 부족합니다");
        }
        this.balance -= amount;
        return true;
    }
}
```

### 3단계: 접근 제어자 설정 전략

| 접근자 | 대상 | 이유 |
|--------|------|------|
| **private** | `balance` (필드) | 직접 접근/수정 금지 |
| **public** | `getBalance()` | 조회 목적으로 누구나 조회 가능 |
| **public** | `deposit()`, `withdraw()` | 비즈니스 로직을 통한 통제된 수정 |

### 4단계: 꼬리 질문 답변

## 핵심 개념 정리

### 📌 캡슐화 (Encapsulation)
- **정의**: 객체의 내부 상태와 구현 세부사항을 숨기고, 필요한 기능만 노출
- **목적**: 데이터 무결성 보호, 외부 변경 영향 최소화, 유지보수성 향상
- **원칙**: "구현은 숨기고, 인터페이스만 공개"

### 📌 접근 제어자 4가지

```
┌─────────────┬──────────┬──────────┬──────────┬──────────┐
│ 제어자      │ 같은 클래스 │ 같은 패키지 │ 상속 클래스 │ 전체 공개 │
├─────────────┼──────────┼──────────┼──────────┼──────────┤
│ private     │    ✅    │    ✗     │    ✗     │    ✗     │
│ (default)   │    ✅    │    ✅    │    ✗     │    ✗     │
│ protected   │    ✅    │    ✅    │    ✅    │    ✗     │
│ public      │    ✅    │    ✅    │    ✅    │    ✅    │
└─────────────┴──────────┴──────────┴──────────┴──────────┘
```

### 📌 Getter/Setter의 역할
- **Getter**: 외부에서 데이터 읽기 (읽기 권한 제어)
- **Setter**: 외부에서 데이터 수정 (검증 로직, 쓰기 권한 제어)
- **특징**: 메서드이므로 추가 로직 삽입 가능

### 📌 불변성 (Immutability)
```java
// 생성 후 변경 불가능하게 설계
public final class ImmutableUser {
    private final double balance;
    private final String name;
    
    public ImmutableUser(String name, double balance) {
        this.name = name;
        this.balance = balance;
    }
    
    // getter만 제공, setter 없음
}
```
