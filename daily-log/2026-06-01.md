## 📚 모범 답안 & 개념 정리

### 1. 📖 개념 설명

# DIP(의존성 역전 원칙)의 변경 강인성

## 모범 답안

DIP는 **변경의 파급 범위를 최소화**함으로써 코드를 변경에 강하게 만듭니다. 

구체적으로 설명하면:

**DIP 없는 경우 (변경에 약함)**
```python
class PaymentProcessor:
    def process(self, payment_type: str, amount: float):
        if payment_type == "credit_card":
            # 신용카드 로직
            pass
        elif payment_type == "bank_transfer":
            # 계좌이체 로직
            pass
        elif payment_type == "crypto":
            # 암호화폐 로직
            pass
        # 새로운 결제수단 추가 시 이 함수를 수정해야 함 ❌
```

**문제점:**
- 새 결제수단 추가 시 `PaymentProcessor` 수정 필요
- 기존 코드를 건드리므로 회귀 테스트 필요
- 결제 로직 전체가 위험에 노출
- OCP(Open-Closed Principle) 위반

**DIP 적용한 경우 (변경에 강함)**
```python
from abc import ABC, abstractmethod

# 추상화: 고수준의 개념에만 의존
class PaymentMethod(ABC):
    @abstractmethod
    def validate(self) -> bool:
        pass
    
    @abstractmethod
    def execute(self, amount: float) -> dict:
        pass

# 저수준의 구체화들
class CreditCardPayment(PaymentMethod):
    def validate(self) -> bool:
        return self._check_card_validity()
    
    def execute(self, amount: float) -> dict:
        return {"status": "approved", "amount": amount}

class CryptoPayment(PaymentMethod):
    def validate(self) -> bool:
        return self._check_wallet_balance()
    
    def execute(self, amount: float) -> dict:
        return {"status": "confirmed", "amount": amount}

# 고수준 모듈: 추상화에만 의존
class PaymentProcessor:
    def __init__(self, payment_method: PaymentMethod):
        self.payment_method = payment_method
    
    def process(self, amount: float) -> dict:
        if not self.payment_method.validate():
            raise ValueError("결제 수단 검증 실패")
        return self.payment_method.execute(amount)

# 사용
processor = PaymentProcessor(CreditCardPayment())
result = processor.process(10000)

# 새로운 결제 수단 추가 시 PaymentProcessor 수정 불필요 ✅
class ApplePayPayment(PaymentMethod):
    def validate(self) -> bool:
        return self._check_apple_pay()
    
    def execute(self, amount: float) -> dict:
        return {"status": "success", "amount": amount}

processor = PaymentProcessor(ApplePayPayment())  # 그냥 대입하면 됨
```

**DIP의 변경 강인성 효과:**

| 측면 | 변경 범위 | 영향도 |
|------|---------|-------|
| **신규 결제수단 추가** | 새 클래스만 작성 | 기존 코드 무영향 ✅ |
| **기존 결제수단 수정** | 해당 클래스만 수정 | 다른 수단에 무영향 ✅ |
| **결제 로직 변경** | 필요 없음 | 기존 코드 안전 ✅ |
| **테스트** | 새 수단만 테스트 | 회귀 테스트 불필요 ✅ |

## 핵심 개념 정리

### 1️⃣ **의존성 역전의 의미**
- **역전 전:** 고수준 모듈 → 저수준 모듈 (직접 의존)
- **역전 후:** 고수준 모듈 ← 추상화 → 저수준 모듈 (추상화에만 의존)
- 의존 방향이 하위에서 상위로 역전됨

### 2️⃣ **변경 강인성 3가지 메커니즘**

**① 변경 범위 제한**
```python
# DIP 덕분에 변경이 로컬화됨
- PaymentProcessor: 변경 없음 (변경 구역 제외)
- CreditCardPayment: 수정 시에도 다른 결제수단 무영향
- 영향도: O(1) → 변경에 아주 강함
```

**② 계약 기반 설계**
```python
# 추상 계약이 변경을 흡수
class PaymentMethod(ABC):
    # 이 계약만 지키면 어떤 구현도 가능
    @abstractmethod
    def execute(self, amount: float) -> dict:
        pass
```

**③ 확장성**
```python
# 새로운 요구사항도 기존 코드 수정 없이 확장
class InstallmentPayment(PaymentMethod):
    def execute(self, amount: float) -> dict:
        return self._process_installment(amount)
```

### 3️⃣ **LSP와의 관계**
```python
# DIP가 있어야 LSP를 안전하게 적용 가능
payment_methods: list[PaymentMethod] = [
    CreditCardPayment(),
    CryptoPayment(),
    ApplePayPayment()  # 모두 PaymentMethod를 완벽히 대체 가능
]

for method in payment_methods:
    # LSP: 어떤 구현이든 안전하게 사용
    processor = PaymentProcessor(method)
    result = processor.process(10000)  # 모두 예상대로 동작
```

### 4️⃣ **ISP와의 시너지**
```python
# ISP: 클라이언트가 필요한 것만 구현하도록
class PaymentMethod(ABC):
    @abstractmethod
    def execute(self, amount: float) -> dict:
        pass

class RefundablePayment(PaymentMethod):
    @abstractmethod
    def refund(self, transaction_id: str) -> dict:
        pass

# 환불 가능한 결제만 이 인터페이스 구현
class CreditCardPayment(RefundablePayment):
    def refund(self, transaction_id: str) -> dict:
        return {"status": "refunded"}

# 환불 불가능한 결제
class CryptoPayment(PaymentMethod):  # RefundablePayment 구현 안 함
    pass
```

## 실무 포인트

### 🔧 **변경 강인성을 수치화하면**

```
변경 비용 = Σ(수정해야 할 클래스 수) × (테스트 범위) × (회귀 위험도)

DIP 없음:
- 결제수단 1개 추가 → PaymentProcessor 수정 필요
- 수정 클래스: 3~5개 (관련 코드 다수)
- 영향도: 높음 (결제는 핵심 기능)

DIP 있음:
- 결제수단 1개 추가 → 새 클래스만 작성
- 수정 클래스: 1개 (신규 클래스)
- 영향도: 낮음 (기존 코드 무영향)

변경 비용 감소: 약 70~80%
```
