## 📌 캡슐화 관련 질문

### 모범 답안

**캡슐화의 수준 결정 기준:**

캡슐화는 "정보 은닉"이 목표이지, "무조건 접근을 막는 것"이 아닙니다. 이 문제의 핵심은 **책임과 의도의 명확성**에 있습니다.

```java
// ❌ 나쁜 예: 과도한 메서드 증식
public void updatePassword(String newPassword) { ... }
public void resetPassword() { ... }
public void setEncryptedPassword(String encrypted) { ... }
public void setTemporaryPassword() { ... }
// → 메서드 증식, 책임이 분산됨

// ✅ 좋은 예: 책임 있는 setter + 내부 검증
public class User {
    private String password; // private 유지
    
    // 단일 진입점, 모든 변경이 여기를 통과
    public void changePassword(String rawPassword) {
        if (!isValidPassword(rawPassword)) {
            throw new InvalidPasswordException("규칙 미충족");
        }
        this.password = hashPassword(rawPassword); // 암호화
        this.lastPasswordChangedAt = LocalDateTime.now(); // 감사추적
    }
    
    // 관리자 초기화 - 의도가 명확한 별도 메서드
    public void resetPasswordByAdmin(String tempPassword) {
        this.password = hashPassword(tempPassword);
        this.requiresPasswordChange = true; // 다음 로그인 시 강제 변경
    }
    
    // 내부용: 이미 해시된 값을 설정 (테스트, 마이그레이션용)
    void setPasswordHash(String hash) {
        this.password = hash;
    }
}
```

**메서드 증식이 정말 나쁜 설계인가?**

아닙니다. **목적이 명확하다면** 메서드 증식은 건강한 신호입니다:
- `changePassword()` - 사용자가 스스로 변경
- `resetPasswordByAdmin()` - 관리자가 초기화
- `setPasswordHash()` - 내부 유틸리티 (package-private)

각 메서드는 **다른 검증 규칙과 부수효과(side effects)**를 가집니다.

**@Transient와 @Formula의 역할:**

```java
public class User {
    @Column
    private String passwordHash;
    
    // ✅ @Transient: DB에 저장하지 않음
    // 캡슐화 ○, 성능 ○
    @Transient
    private String rawPassword; // 메모리상에만 존재
    
    // ❌ @Formula: 쿼리 시마다 자동 계산
    // 캡슐화는 충족하지만, 성능 문제 발생 가능
    @Formula("CASE WHEN last_login > NOW() - INTERVAL 7 DAY THEN 1 ELSE 0 END")
    private boolean activeThisWeek;
}
```

**결론:** 이들은 **캡슐화를 우회하는 것이 아니라 지원하는 도구**입니다. 진정한 캡슐화는 "어떤 데이터를 어떻게 변경할 것인가"의 의도를 명확히 하는 것입니다.

### 핵심 개념 정리

#### 캡슐화(Encapsulation)
- **정보 은닉 ≠ 접근 차단**: 불필요한 세부사항만 숨기기
- **변경의 의도를 명확히**: 메서드명으로 목적 표현 (changePassword vs resetPassword)
- **불변식(Invariant) 보호**: 객체가 항상 유효한 상태 유지
  ```java
  private LocalDateTime lastPasswordChangedAt;
  // password 변경 시 반드시 함께 업데이트 → 불변식 보호
  ```

#### 검증 계층화
| 레벨 | 책임 | 예시 |
|------|------|------|
| **Public** | 비즈니스 검증 | `changePassword()` - 정책 적용 |
| **Protected/Package** | 내부 접근 | `setPasswordHash()` - 테스트 |
| **Private** | 내부 유틸리티 | `hashPassword()` - 구현 |

#### 접근 제어자 전략
```java
// 계층적 접근 제어
public changePassword()        // 외부 사용자
protected resetPasswordByAdmin() // 같은 패키지 서비스
package-private setPasswordHash() // 테스트, 마이그레이션
private hashPassword()         // 구현 세부사항
```

#### ✅ DO: 캡슐화의 올바른 사용
```java
@Entity
public class User {
    private String password;
    
    // 단일 책임, 명확한 의도
    public void changePassword(String currentPassword, String newPassword) {
        validateCurrentPassword(currentPassword);
        validateNewPasswordPolicy(newPassword);
        validateNotReusedInLast3Months(newPassword);
        
        this.password = PasswordEncoder.encode(newPassword);
        this.lastPasswordChangedAt = LocalDateTime.now();
        this.passwordChangeCount++;
    }
}

// 사용
user.changePassword("oldPwd", "newSecurePwd123!");
```

#### ❌ DON'T: 캡슐화 위반
```java
// 위험: setter가 검증 없이 평문 저장

// 위험: 너무 많은 변형 메서드

user.setPassword();
user.resetPassword();
```

#### 🎯 트레이드오프 판단 기준
```
결정: changePassword() 하나 vs 여러 메서드

메서드 많음 (+)          메서드 하나 (+)
├─ 의도 명확             ├─ 심플함
├─ 각자 검증 로직        └─ 모든 변경점 추적 용이
└─ 의도치 않은 호출 방지

→ 결론: 의도가 다르면 메서드를 나누되,
```

현재 코드는 **4가지 OOP 원칙 모두를 위반**하고 있습니다:

| 원칙 | 위반 형태 | 영향 |
|------|---------|------|
| **추상화** | 결제 수단의 공통 인터페이스 부재 | 각 수단별 로직이 산재됨 |

| **OCP 위반** | 확장에 닫혀있고 수정에 열려있음 | 회귀테스트 범위 확대, 버그 리스크 증가 |

### 2️⃣ 해결책: Strategy + Template Method 패턴 적용

```java
// 1단계: 추상화 - 결제 수단의 공통 계약 정의
public abstract class PaymentStrategy {
    protected String paymentType;
    
    public abstract PaymentResult process(Payment payment);
    public abstract RefundResult refund(String transactionId, BigDecimal amount);
    public abstract SettlementResult settle(LocalDate date);
    public abstract void validate(Payment payment) throws InvalidPaymentException;
    
    // Template Method Pattern: 공통 흐름 정의
    public final PaymentResult executePayment(Payment payment) {
        validate(payment);
        return process(payment);
    }
}

// 2단계: 구체적 구현 - 각 결제 수단 전담 클래스
public class CardPaymentStrategy extends PaymentStrategy {
    @Autowired
    private CardGateway cardGateway;
    
    @Override
    public PaymentResult process(Payment payment) {
        CardPaymentRequest request = CardPaymentRequest.from(payment);
        CardPaymentResponse response = cardGateway.authorize(request);
        
        return PaymentResult.builder()
            .transactionId(response.getTransactionId())
            .status(PaymentStatus.SUCCESS)
            .build();
    }
    
    @Override
    public RefundResult refund(String transactionId, BigDecimal amount) {
        return cardGateway.refund(transactionId, amount);
    }
    
    @Override
    public SettlementResult settle(LocalDate date) {
        return cardGateway.settle(date);
    }
    
    @Override
    public void validate(Payment payment) throws InvalidPaymentException {
        if (payment.getCard() == null) {
            throw new InvalidPaymentException("Card information is required");
        }
        if (!payment.getCard().isValid()) {
            throw new InvalidPaymentException("Invalid card");
        }
    }
}

public class PointPaymentStrategy extends PaymentStrategy {
    @Autowired
    private PointGateway pointGateway;
    @Autowired
    private PointLedgerRepository ledgerRepo;
    
    @Override
    public PaymentResult process(Payment payment) {
        // 포인트 차감 트랜잭션
        PointDeductionResult result = pointGateway.deduct(
            payment.getMemberId(), 
            payment.getAmount()
        );
        
        // 장부 기록 (감시 추적용)
        ledgerRepo.save(PointLedger.ofDeduction(payment, result));
        
        return PaymentResult.builder()
            .transactionId(result.getTransactionId())
            .status(PaymentStatus.SUCCESS)
            .build();
    }
    
    @Override
    public RefundResult refund(String transactionId, BigDecimal amount) {
        // 포인트 환급 (여기서 발생했던 버그!)
        PointRefundResult result = pointGateway.refund(transactionId, amount);
        ledgerRepo.save(PointLedger.ofRefund(transactionId, result));
        return result;
    }
    
    @Override
    public void validate(Payment payment) throws InvalidPaymentException {
        BigDecimal balance = pointGateway.getBalance(payment.getMemberId());
        if (balance.compareTo(payment.getAmount()) < 0) {
            throw new InvalidPaymentException("Insufficient points");
        }
    }
}

@Service
public class PaymentService {
    @Autowired
    private Map<String, PaymentStrategy> paymentStrategies;
    
    @Autowired
    private PaymentTransactionRepository transactionRepo;
    
    // 전략 등록 (Spring DI로 자동화)
    @Configuration
    public static class PaymentStrategyConfig {
        @Bean
        public Map<String, PaymentStrategy> paymentStrategies(
            CardPaymentStrategy cardStrategy,
            PointPaymentStrategy pointStrategy,
            TransferPaymentStrategy transferStrategy) {
            
            Map<String, PaymentStrategy> strategies = new HashMap<>();
            strategies.put("CARD", cardStrategy);
            strategies.put("POINT", pointStrategy);
            strategies.put("TRANSFER", transferStrategy);
            return strategies;
        }
    }
    
    // 단순화된 메인 메서드
    public PaymentResult process(String paymentType, Payment payment) {
        PaymentStrategy strategy = getStrategy(paymentType);
        
        try {
            PaymentResult result = strategy.executePayment(payment);
            transactionRepo.save(PaymentTransaction.from(payment, result));
            return result;
        } catch (InvalidPaymentException e) {
            return PaymentResult.failure(e.getMessage());
        }
    }
    
    public RefundResult refund(String paymentType, String transactionId, BigDecimal amount) {
        PaymentStrategy strategy = getStrategy(paymentType);
        return strategy.refund(transactionId, amount);
    }
    
    public SettlementResult settle(String paymentType, LocalDate date) {
        PaymentStrategy strategy = getStrategy(paymentType);
        return strategy.settle(date);
    }
    
    private PaymentStrategy getStrategy(String paymentType) {
        PaymentStrategy strategy = paymentStrategies.get(paymentType);
        if (strategy == null) {
            throw new UnsupportedPaymentMethodException(
                "지원하지 않는 결제 수단: " + paymentType
            );
        }
        return strategy;
    }
}
```

```java
    public BigDecimal calculatePrice(OrderContext context) {
        // 실물상품: 기본가 + 배송비 + 무게기반 추가비
        BigDecimal basePrice = this.getPrice();
        BigDecimal shippingFee = calculateShippingFee();
        BigDecimal weightFee = weight.multiply(BigDecimal.valueOf(0.5));
        return basePrice.add(shippingFee).add(weightFee);
    }
    
    @Override
    public void processInventory(Long quantity) {
        // 실물상품: 창고에서 차감
        Warehouse warehouse = warehouses.stream()
            .filter(w -> w.getQuantity() >= quantity)
            .findFirst()
            .orElseThrow(() -> new InsufficientInventoryException());
        
        warehouse.decreaseQuantity(quantity);
    }
    
    @Override
    public ShippingInfo getShippingInfo() {
        return new PhysicalShippingInfo(
            this.weight,
            this.dimensions,
            ShippingMethod.PARCEL
        );
    }

        // 무게 기반 배송비 계산
        if (weight.compareTo(BigDecimal.valueOf(5)) < 0) {
            return BigDecimal.valueOf(3000);
        }
        return BigDecimal.valueOf(5000);
    }
```
