## 1️⃣ 캡슐화 관련 질문

### 모범 답안

**핵심 관점: "캡슐화는 접근 제어가 아니라 책임 관리"**

이 딜레마는 **"정보 은닉의 본질이 무엇인가"**에 대한 질문입니다.

```java
// ❌ 나쁜 예: 접근 제어만 강조
public class User {
    private String password;
    // setter 완전 차단
}

// ❌ 또 다른 나쁜 예: 검증 없는 노출
public class User {
    public String password; // 평문 위험
}

// ✅ 올바른 접근: 책임을 명확히 하는 캡슐화
public class User {
    private EncryptedPassword password; // 값 객체로 감싸기
    
    // 단일 책임: password 변경만 담당
    public void changePassword(String rawPassword) {
        validatePasswordPolicy(rawPassword);
        this.password = EncryptedPassword.encrypt(rawPassword);
        this.passwordChangedAt = LocalDateTime.now();
    }
    
    // 관리자만 사용
    public void resetPassword(String tempPassword) {
        this.password = EncryptedPassword.encrypt(tempPassword);
        this.requiresPasswordChange = true;
    }
    
    // 내부 도메인 로직
    public boolean verifyPassword(String rawPassword) {
        return this.password.matches(rawPassword);
    }
    
    private void validatePasswordPolicy(String password) {
        if (password.length() < 8) throw new InvalidPasswordException();
        // 정책 검증
    }
}
```

**메서드가 여러 개인 것이 문제가 아닙니다.**
- `changePassword()` - 사용자 자신이 호출
- `resetPassword()` - 관리자/시스템이 호출
- `verifyPassword()` - 인증 로직에서 호출
- `forceChangePassword()` - 보안 정책에서 호출

→ 각 메서드는 **서로 다른 비즈니스 시나리오**를 표현합니다.

**@Transient와 @Formula에 대한 판단:**

```java
// @Transient: 캡슐화 우회 ⚠️
public class User {
    @Transient
    private String decryptedPassword; // DB에 저장 안 함
    
    // 문제: 이건 그냥 접근 제어 우회
}

// @Formula: 캡슐화 정책 표현 ✅
public class Order {
    @Formula("(SELECT COUNT(*) FROM order_items WHERE order_id = id)")
    private int itemCount;
    
    // 적절한 사용: 계산된 파생값을 읽기 전용으로 제공
}
```

### 핵심 개념 정리

- **정보 은닉 ≠ 접근 제어**
  - 접근 제어: private/public 키워드 수준
  - 정보 은닉: 내부 구현을 외부로부터 보호하는 설계 의도

- **값 객체(Value Object) 패턴**
  ```java
  public class EncryptedPassword {
      private final String encryptedValue;
      
      private EncryptedPassword(String encrypted) {
          this.encryptedValue = encrypted;
      }
      
      public static EncryptedPassword encrypt(String raw) {
          // 암호화 로직은 여기서만 관리
          return new EncryptedPassword(hashPassword(raw));
      }
      
      public boolean matches(String raw) {
          return this.encryptedValue.equals(hashPassword(raw));
      }
  }
  ```

- **메서드 폭발(Method Explosion) vs 책임 분리**
  - 메서드가 많다 = 비즈니스 케이스가 다양하다 (OK)
  - 메서드가 모호하다 = 단일 책임 위반 (NOT OK)

- **접근 제어 전략**
  | 수준 | 사용 시점 | 예시 |
  |------|---------|------|
  | private | 내부 구현 | 검증 로직, 헬퍼 메서드 |
  | package-private | 도메인 같은 패키지 | 엔티티 간 통신 |
  | protected | 상속 계층 | 공통 템플릿 메서드 |
  | public | 공개 계약 | 도메인 행동 |

```java
public class User implements AggregateRoot {
    private EncryptedPassword password;
    private PasswordChangeHistory history;
    
    // 도메인 이벤트 발행
    public void changePassword(String oldPassword, String newPassword) {
        if (!password.matches(oldPassword)) {
            throw new InvalidPasswordException();
        }
        validatePasswordPolicy(newPassword);
        
        EncryptedPassword newEncrypted = EncryptedPassword.encrypt(newPassword);
        this.password = newEncrypted;
        this.history.add(newEncrypted, LocalDateTime.now());
        
        // DDD 패턴: 도메인 이벤트 발행
        addDomainEvent(new PasswordChangedEvent(this.id, LocalDateTime.now()));
    }
}

// 스프링 예제
@Service
public class UserService {
    @Transactional
    public void changePassword(Long userId, String oldPassword, String newPassword) {
        User user = userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException());
        
        // User 엔티티가 자신의 책임을 다함
        user.changePassword(oldPassword, newPassword);
        
        // 저장만 함 (도메인 로직은 User에서)
        userRepository.save(user);
    }
}
```

### 해결책: OOP 4대 원칙 적용한 리팩토링

```java
// 1️⃣ 추상화: 결제 수단의 공통 인터페이스 정의
public interface PaymentMethod {
    PaymentResult process(PaymentRequest request);
    RefundResult refund(String transactionId, BigDecimal amount);
    SettlementResult settle(LocalDate date);
    boolean validate(PaymentRequest request);
}

// 2️⃣ 캡슐화: 각 결제 수단별 비즈니스 로직을 자체 클래스에 캡슐화
public class CardPayment implements PaymentMethod {
    private CardGateway gateway;
    private CardValidator validator;
    
    @Override
    public PaymentResult process(PaymentRequest request) {
        // 카드 결제 로직 캡슐화
        if (!validator.validate(request)) {
            return PaymentResult.fail("Invalid card");
        }
        
        CardTransaction transaction = gateway.authorize(
            request.getCardNumber(),
            request.getAmount()
        );
        
        return PaymentResult.success(transaction.getId());
    }
    
    @Override
    public RefundResult refund(String transactionId, BigDecimal amount) {
        // 카드 환불은 카드 전문가만 알고 있음
        return gateway.refund(transactionId, amount);
    }
    
    @Override
    public SettlementResult settle(LocalDate date) {
        return gateway.dailySettle(date);
    }
}

public class PointPayment implements PaymentMethod {
    private PointService pointService;
    private PointValidator validator;
    
    @Override
    public PaymentResult process(PaymentRequest request) {
        // 포인트 결제 로직
        if (!validator.hasEnoughPoint(request.getMemberId(), request.getAmount())) {
            return PaymentResult.fail("Insufficient points");
        }
        
        String transactionId = pointService.deductPoint(
            request.getMemberId(),
            request.getAmount()
        );
        
        return PaymentResult.success(transactionId);
    }
    
    @Override
    public RefundResult refund(String transactionId, BigDecimal amount) {
        // 포인트 환불: 차감했던 포인트 복구
        pointService.restorePoint(transactionId, amount);
        return RefundResult.success();
    }
    
    @Override
    public SettlementResult settle(LocalDate date) {
        return pointService.settlementReport(date);
    }
}

public abstract class PaymentMethodAdapter implements PaymentMethod {
    protected Logger logger = LoggerFactory.getLogger(getClass());
    protected MetricsCollector metrics;
    
    // 공통 로깅, 모니터링 기능
    protected void logTransaction(String transactionId, BigDecimal amount) {
        logger.info("[{}] Transaction: {}, Amount: {}", 
            getClass().getSimpleName(), transactionId, amount);
        metrics.recordTransaction(getClass().getSimpleName(), amount);
    }
    
    // 공통 재시도 로직
    protected <T> T executeWithRetry(Supplier<T> operation, int retries) {
        // 공통 재시도 로직
        return null;
    }
}

        this.paymentMethods = new HashMap<>();
        // 결제 수단 등록 (팩토리 패턴 활용)
        paymentMethods.put(PaymentType.CARD, new CardPayment(cardGateway));
        paymentMethods.put(PaymentType.TRANSFER, new TransferPayment(transferGateway));
        paymentMethods.put(PaymentType.POINT, new PointPayment(pointService));
        paymentMethods.put(PaymentType.MOBILE, new MobilePayment(mobileGateway));
    }
    
    // 다형성: 구체적 결제 수단에 신경 쓰지 않음
    public PaymentResult process(PaymentRequest request) {
        PaymentMethod method = paymentMethods.get(request.getType());
        
        if (method == null) {
            throw new UnsupportedPaymentMethodException(request.getType());
        }
        
        return method.process(request);
    }
    
    public RefundResult refund(PaymentRequest request) {
        PaymentMethod method = paymentMethods.get(request.getType());
        return method.refund(request.getTransactionId(), request.getAmount());
    }
    
    public SettlementResult settle(PaymentType type, LocalDate date) {
        PaymentMethod method = paymentMethods.get(type);
        return method.settle(date);
    }
}
```

// ✅ 수정된 버전
public class PointPayment implements PaymentMethod {
    
    @Override
    public RefundResult refund(String transactionId, BigDecimal amount) {
        // 1. 원래 거래 조회 및 상태 검증
        PointTransaction originalTransaction = pointService
            .getTransaction(transactionId)
            .orElseThrow(() -> new TransactionNotFoundException(transactionId));
        
        if (!originalTransaction.canRefund()) {
            return RefundResult.fail("Cannot refund already refunded transaction");
        }
        
        // 2. 환불 트랜잭션 생성 (원자성 보장)
        try {
            String refundId = pointService.createRefundTransaction(
                transactionId,
                amount,
                RefundStatus.PENDING
            );
            
            // 3. 포인트 복구 실행
            pointService.restorePoint(originalTransaction.getMemberId(), amount);
            
            // 4. 환불 상태 완료로 업데이트
            pointService.updateRefundStatus(refundId, RefundStatus.COMPLETED);
            
            logger.info("Point refund completed: {} -> {}", transactionId, refundId);
            return RefundResult.success(refundId);
            
        } catch (Exception e) {
            logger.error("Point refund failed: {}", transactionId, e);
            return RefundResult.fail("Refund processing error: " + e.getMessage());
        }
    }
}

| 항목 | JOINED | SINGLE_TABLE | 선택 근거 |
|------|--------|--------------|----------|
| **쿼리 성능** | JOIN 필요 (느림) | 단일 테이블 (빠름) | 정규화 중요 |
| **데이터 정규화** | 우수 | 나쁨 (NULL 많음) | 무결성 중요 |
| **스토리지** | 효율적 | 비효율적 | 저장공간 절약 |
| **새 타입 추가** | 쉬움 | 쉬움 | 유지보수 편함 |
| **타입별 컬럼** | 테이블 분리 | 모두 한 테이블 | 논리적 분리 |
| **복잡한 조회** | 복잡도 증가 | 간단 | 현실적 고려 |
