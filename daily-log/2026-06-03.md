# SOLID 원칙 - LSP, ISP, DIP 모범 답안

## 모범 답안

### 메인 질문: LSP(리스코프 치환 원칙)

**핵심 설명:**

LSP는 "부모 클래스의 인터페이스를 지키면서 자식 클래스가 그 역할을 대체할 수 있어야 한다"는 원칙입니다. 단순히 상속받는 것뿐 아니라, **계약(Contract)을 지켜야 한다**는 의미입니다.

**왜 중요한가:**
- 다형성(Polymorphism)을 안전하게 사용 가능
- 런타임에 예측 불가능한 버그 방지
- 코드의 유연성과 확장성 확보

**문제 발생 예시:**

```java
// ❌ LSP 위반 사례
class Bird {
    public void fly() {
        System.out.println("날아갑니다");
    }
}

class Penguin extends Bird {
    @Override
    public void fly() {
        throw new UnsupportedOperationException("펭귄은 날 수 없습니다");
    }
}

// 클라이언트 코드
public void letBirdFly(Bird bird) {
    bird.fly(); // Penguin이 들어오면 런타임 에러!
}
```

**문제점:**
- Bird 타입으로 메서드 호출했는데 예상과 다른 동작 발생
- 클라이언트가 각 자식 클래스의 특수 상황을 일일이 처리해야 함
- 새로운 Bird 추가 시마다 클라이언트 코드 수정 필요

**✅ LSP를 준수한 해결책:**

```java
// 1단계: 인터페이스 분리
interface Flyable {
    void fly();
}

interface Swimmable {
    void swim();
}

// 2단계: 적절한 구현
class Eagle implements Flyable {
    @Override
    public void fly() {
        System.out.println("독수리가 날아갑니다");
    }
}

class Penguin implements Swimmable {
    @Override
    public void swim() {
        System.out.println("펭귄이 헤엄칩니다");
    }
}

// 3단계: 클라이언트는 해당 인터페이스만 기대
public void letFly(Flyable bird) {
    bird.fly(); // 모든 Flyable 객체는 안전하게 fly() 가능
}
```

### 꼬리 질문: ISP(인터페이스 분리 원칙) vs LSP

**차이점 정리:**

| 구분 | LSP | ISP |
|------|-----|-----|
| **초점** | 자식이 부모 역할을 제대로 수행 | 클라이언트가 필요 없는 메서드 강제 금지 |
| **문제 해결** | 치환 가능성 보장 | 불필요한 의존성 제거 |
| **관점** | 행동의 정확성 | 인터페이스 설계 |
| **비유** | "자식은 부모처럼 행동해야 함" | "클라이언트가 원하는 것만 제공" |

**ISP 위반 → LSP 문제 발생 메커니즘:**

```java
// ❌ ISP 위반으로 인한 LSP 붕괴
interface Worker {
    void work();
    void eat();  // 모든 Worker가 eat()이 필요하지는 않음
}

class Human implements Worker {
    @Override
    public void work() { System.out.println("일합니다"); }
    
    @Override
    public void eat() { System.out.println("먹습니다"); }
}

class Robot implements Worker {
    @Override
    public void work() { System.out.println("작동합니다"); }
    
    @Override
    public void eat() {
        throw new UnsupportedOperationException("로봇은 먹지 않습니다");
    }
}

// LSP 위반 발생
public void feedWorker(Worker worker) {
    worker.eat(); // Robot 입력 시 에러!
}
```

**개선된 설계 (ISP 적용):**

```java
// ✅ 인터페이스 분리
interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

class Human implements Workable, Eatable {
    @Override
    public void work() { System.out.println("일합니다"); }
    
    @Override
    public void eat() { System.out.println("먹습니다"); }
}

class Robot implements Workable {
    @Override
    public void work() { System.out.println("작동합니다"); }
}

// 이제 LSP가 자동으로 보장됨
public void executeWork(Workable worker) {
    worker.work(); // 모든 Workable은 work() 가능
}
```

## 핵심 개념 정리

### LSP (Liskov Substitution Principle)
- **정의**: 부모 클래스의 메서드 계약을 자식도 따라야 함
- **체크포인트**:
  - ✅ 사전조건(Precondition)을 약하게 만들지 말 것
  - ✅ 사후조건(Postcondition)을 강하게 만들지 말 것
  - ✅ 불변식(Invariant) 유지
  - ✅ 예외 처리 일관성

### ISP (Interface Segregation Principle)
- **정의**: 클라이언트가 필요하지 않은 메서드에 의존하지 않도록 설계
- **핵심**:
  - 역할별로 인터페이스 분리
  - 한 번에 여러 인터페이스 구현 가능
  - "지방을 피하라" (Fat Interface 제거)

### DIP (Dependency Inversion Principle)
- **정의**: 고수준 모듈이 저수준 모듈에 의존하지 말고, 추상화에 의존
- **패턴**:
  - Dependency Injection
  - Service Locator
  - Factory Pattern

### 세 원칙의 관계
```
ISP (인터페이스 설계)
  ↓
LSP (치환 가능성 보장)
  ↓
DIP (추상화 의존)
  → 전체적인 유연하고 확장 가능한 구조
```

```java
// ❌ 잘못된 설계
@Service
public class PaymentService {
    public void processPayment(PaymentGateway gateway, Order order) {
        if (gateway instanceof CreditCardGateway) {
            ((CreditCardGateway) gateway).charge(...);
        } else if (gateway instanceof PayPalGateway) {
            ((PayPalGateway) gateway).sendPayment(...);
        }
    }
}

// ✅ 올바른 설계 (DIP + LSP)
interface PaymentProcessor {
    void process(Order order);
}

@Service
public class CreditCardProcessor implements PaymentProcessor {
    @Override
    public void process(Order order) { ... }
}

@Service
public class PayPalProcessor implements PaymentProcessor {
    @Override
    public void process(Order order) { ... }
}

@Service
public class PaymentService {
    private final PaymentProcessor processor;
    
    @Autowired
    public PaymentService(PaymentProcessor processor) {
        this.processor = processor; // 추상화에 의존
    }
    
    public void processPayment(Order order) {
        processor.process(order); // LSP 보장됨
    }
}
```
