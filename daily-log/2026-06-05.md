## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# IoC와 DI — 왜 직접 객체를 만들지 않나

## 모범 답안

### 문제점 분석
당신의 상황은 **강한 결합도(Tight Coupling)** 문제입니다. 주문 처리 클래스가 SMS 객체 생성에 직접 의존하고 있어서, 알림 방식이 추가될 때마다 주문 처리 코드를 수정해야 합니다.

### DI 적용 후의 개선점

```python
# ❌ DI 없이 (강한 결합)
class OrderProcessor:
    def __init__(self):
        self.notifier = SMSNotifier()  # 직접 생성
    
    def process_order(self, order):
        # SMS만 가능, 카톡/푸시 추가 시 이 코드를 수정해야 함
        self.notifier.send(order)

# ✅ DI 적용 (느슨한 결합)
from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, message: str):
        pass

class SMSNotifier(Notifier):
    def send(self, message: str):
        print(f"SMS 전송: {message}")

class KakaoNotifier(Notifier):
    def send(self, message: str):
        print(f"카톡 전송: {message}")

class PushNotifier(Notifier):
    def send(self, message: str):
        print(f"푸시 전송: {message}")

class OrderProcessor:
    def __init__(self, notifiers: list[Notifier]):  # 외부에서 주입받음
        self.notifiers = notifiers
    
    def process_order(self, order):
        for notifier in self.notifiers:
            notifier.send(f"주문 {order.id} 접수")  # 코드 수정 불필요!

# 사용
if __name__ == "__main__":
    # 어떤 알림 조합도 가능 - OrderProcessor는 건드릴 필요 없음
    notifiers = [SMSNotifier(), KakaoNotifier(), PushNotifier()]
    processor = OrderProcessor(notifiers)
    processor.process_order(order)
```

### DI의 핵심 이점

| 항목 | 문제상황 | DI 적용 |
|------|---------|---------|
| **코드 수정 범위** | Notifier 추가 시 OrderProcessor 수정 필요 | OrderProcessor 수정 불필요 |
| **테스트** | 실제 SMS 발송으로 테스트 (느림, 비용) | Mock 객체 주입으로 테스트 (빠름) |
| **확장성** | 새로운 알림 방식 추가 어려움 | 새 클래스만 추가하면 자동 지원 |
| **결합도** | 높음 (구체적 클래스에 의존) | 낮음 (추상화에 의존) |

## 핵심 개념 정리

### 1. **제어의 역전(IoC: Inversion of Control)**
- **기존**: 객체가 자신이 필요한 의존성을 직접 생성·관리 (객체가 제어권 보유)
- **IoC**: 객체 생성·관리를 외부(컨테이너)에 맡김 (제어권을 역전)
- **효과**: 객체는 "비즈니스 로직"에만 집중, "관계 관리"는 외부에 위임

### 2. **의존성 주입(DI: Dependency Injection)**
- IoC를 구현하는 구체적인 패턴
- **3가지 방식**:
  ```python
  # 1️⃣ 생성자 주입 (권장)
  OrderProcessor(notifiers)
  
  # 2️⃣ 메서드 주입
  processor.set_notifiers(notifiers)
  
  # 3️⃣ 속성 주입
  processor.notifiers = notifiers
  ```

### 3. **의존성 역전 원칙(DIP)**
```python
# 나쁜 예: 구체화에 의존
class OrderProcessor:
    def __init__(self):
        self.sms = SMSNotifier()  # 구체 클래스에 의존

# 좋은 예: 추상화에 의존
class OrderProcessor:
    def __init__(self, notifier: Notifier):  # 추상 인터페이스에 의존
        self.notifier = notifier
```

### 4. **느슨한 결합(Loose Coupling)의 이점**
- 각 클래스가 인터페이스만 알고 구현은 모름
- "변경의 영향 범위"가 최소화됨
- 새로운 요구사항에 대해 "확장"만 하고 "수정"은 없음 (Open/Closed Principle)

## 실무 포인트

### 1. **Spring/FastAPI에서의 DI 컨테이너**
```python
# FastAPI 예제 (의존성 주입)
from fastapi import FastAPI, Depends

app = FastAPI()

def get_notifiers() -> list[Notifier]:
    return [SMSNotifier(), KakaoNotifier()]

@app.post("/orders")
def create_order(
    order: Order,
    notifiers: list[Notifier] = Depends(get_notifiers)  # 자동 주입
):
    processor = OrderProcessor(notifiers)
    return processor.process_order(order)
```

### 2. **테스트 코드에서의 이점**
```python
# Mock 객체를 주입하여 실제 SMS 발송 없이 테스트
class MockNotifier(Notifier):
    def __init__(self):
        self.sent_messages = []
    
    def send(self, message: str):
        self.sent_messages.append(message)

def test_order_processing():
    mock = MockNotifier()
    processor = OrderProcessor([mock])
    processor.process_order(order)
    
    assert len(mock.sent_messages) == 1
    assert "주문" in mock.sent_messages[0]
```

### 3. **실제 프로젝트 체크리스트**
- ✅ 외부 의존성(DB, API, 메시지 큐)을 생성자로 주입받는가?
- ✅ 추상 인터페이스를 통해 구현체를 교체 가능하게 했는가?
- ✅ 테스트 코드에서 Mock/Stub 객체를 쉽게 주입할 수 있는가?
- ✅ DI 컨테이너(Spring, Pydantic, etc.)를 제대로 활용하고 있는가?

### 4. **주의사항**
- **과도한 DI**: 간단한 프로젝트에 DI는 오버엔지니어링
- **순환 의존성**: A→B→A 형태의 순환 참조 피하기
- **가독성**: 주입받는 의존성이 너무 많으면 리팩토링 신호

## 결론

**"왜 DI를 하나?"** → 코드 수정 없이 동작을 변경하기 위해

당신의 상황에서 DI를 적용하면:
- SMS 추가 시: OrderProcessor는 그대로, NotifierImpl만 추가
- 카톡 추가 시: OrderProcessor는 그대로, NotifierImpl만 추가
- 푸시 추가 시: OrderProcessor는 그대로, NotifierImpl만 추가

**"확장에는 열려있고, 수정에는 닫혀있다"** — Open/Closed Principle의 핵심입니다.
