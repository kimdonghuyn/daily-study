## 모범 답안

### 메인 질문: 메모리 부족 상황과 GC의 역할

**"메모리가 가득 차면 어떻게 될까?"**

```
메모리가 가득 차면 OutOfMemoryError가 발생해서 프로그램이 비정상 종료됩니다.
이를 사전에 방지하기 위해 Garbage Collection이 존재합니다.
```

**"GC가 정확히 뭘 하는 건가?"**

```java
// GC의 역할 - 메모리 자동 관리
public class GCExample {
    public static void main(String[] args) {
        // 1. 객체 생성 (Heap 메모리에 할당)
        User user1 = new User("John");
        User user2 = new User("Jane");
        
        // 2. 참조 제거 (user1이 더 이상 필요없음)
        user1 = null;  // 이 객체는 더 이상 접근 불가능
        
        // 3. GC가 자동으로 감지
        // → user1이 가리키던 객체를 Heap에서 제거
        // → 메모리 회수
    }
}
```

**GC의 3가지 주요 역할:**
1. **Heap의 사용 가능한 객체 추적** - 어떤 객체가 여전히 필요한지 판단
2. **불필요한 객체 제거** - 참조되지 않는 객체를 메모리에서 해제
3. **메모리 재정렬** - Fragmentation을 줄이고 연속된 메모리 공간 생성

### 꼬리 질문: GC 동작 중 프로그램 상태 (Stop-The-World)

**"사용자가 버튼을 누를 때 GC가 돌아가면?"**

```
→ 프로그램이 멈춥니다 (Stop-The-World, STW)
→ 사용자의 입력이 일시적으로 반응하지 않습니다
→ 이것이 GC 알고리즘 선택의 핵심 고민점입니다
```

**시각적 비교:**

```
[Serial GC] - 단일 스레드
═════════════════════════════════════════════
|----GC 중(STW)----|   |----GC 중(STW)----|
애플리케이션 중단 → GC → 애플리케이션 중단 → GC
(일반적으로 수백ms ~ 수초)

[Parallel GC] - 멀티 스레드
═════════════════════════════════════════════
|--GC(멀티--스레드)-|   |--GC(멀티--스레드)-|
STW 시간 단축, 처리량 증가

[G1 GC] - 영역 기반
═════════════════════════════════════════════
|GC|        |GC|        |GC|
짧은 STW, 예측 가능한 지연

[ZGC] - 거의 무중단
═════════════════════════════════════════════
|--STW <10ms--|        |--STW <10ms--|
거의 인지 불가능한 지연
```

## 핵심 개념 정리

### 1️⃣ GC의 기본 원리 (Mark-Sweep-Compact)

```
Phase 1: Mark (표시)
├─ GC Root에서 시작 (스택, static, JNI 참조)
├─ 도달 가능한 모든 객체 추적
└─ 도달 불가능한 객체 표시

Phase 2: Sweep (청소)
├─ 표시되지 않은 객체를 메모리에서 제거
└─ 메모리 공간 회수

Phase 3: Compact (압축)
├─ 조각난 메모리를 정리
└─ 연속된 메모리 공간 확보
```

### 2️⃣ 세대별 GC 개념 (Generational Hypothesis)

```
Heap 구조
┌─────────────────────────────────┐
│       Old Generation (오래된)      │  ← 거의 변하지 않음
│   (Major GC가 필요할 때만)        │
├─────────────────────────────────┤
│  Young Generation (새로운)        │  ← 자주 변함
│  ┌──────────┬──────┬──────┐      │
│  │  Eden    │ S0   │ S1   │      │
│  └──────────┴──────┴──────┘      │
│  (Minor GC가 자주 발생)           │
└─────────────────────────────────┘

특징: 대부분의 객체는 Young에서 금방 쓸모없어짐
전략: Young은 자주, Old는 가끔만 GC 수행
```

### 3️⃣ GC 알고리즘 비교표

| 특성 | Serial | Parallel | G1 | ZGC |
|------|--------|----------|-----|------|
| **동작 방식** | 단일 스레드 | 멀티 스레드 | 영역 기반 | 동시 처리 |
| **STW 시간** | 수백ms~수초 | 수십~수백ms | 수ms~수십ms | <10ms |
| **대상 힙 크기** | ~100MB | ~GBs | ~GBs | ~TBs |
| **처리량** | 낮음 | 높음 | 중간~높음 | 약간 낮음 |
| **예측성** | 예측 불가능 | 예측 불가능 | **예측 가능** | **매우 예측 가능** |
| **사용 사례** | 개발/테스트 | 배치 작업 | **일반적인 웹서버** | **초저지연 시스템** |

## Serial GC (단일 스레드)

### 특징
```java
// JVM 옵션
java -XX:+UseSerialGC MyApp

// 동작 원리
┌─────────────────────────┐
│ Young GC(Minor GC)      │ ← 자주 발생 (몇 초마다)
│ STW: ~50-100ms          │
└─────────────────────────┘
         ↓ (오래된 객체 증가)
┌─────────────────────────┐
│ Old GC(Major GC)        │ ← 가끔 발생 (GC 알고리즘에 따라)
│ STW: 수백ms ~ 수초      │
└─────────────────────────┘
```

### 언제 사용?
```
✅ 개발 환경
✅ 테스트 환경  
✅ 1GB 이하의 작은 힙 사이즈
✅ 싱글 코어 CPU

❌ 프로덕션 멀티코어 서버 환경에서는 부적합
```

## Parallel GC (멀티 스레드)

### 특징
```java
// JVM 옵션 (Java 8 기본값)
java -XX:+UseParallelGC MyApp

// 특징: 여러 스레드가 병렬로 GC 수행
┌─────────────────────────────────┐
│ Minor GC (여러 스레드 동시 수행) │
│ STW: ~20-50ms (더 짧음!)        │
│                                 │
│ Thread1: [████] Mark            │
│ Thread2: [████] Mark            │
│ Thread3: [████] Sweep           │
└─────────────────────────────────┘
```

### 언제 사용?
```
✅ 배치 처리 작업 (처리량 중요)
✅ 멀티코어 서버
✅ 실시간성이 중요하지 않은 경우

❌ 응답시간이 중요한 경우 (STW가 여전히 긴 편)
❌ 매우 큰 힙 사이즈 (STW가 선형적으로 증가)
```
