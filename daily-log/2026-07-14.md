## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 음식 배달 앱의 상속과 다형성 문제 해결

## 모범 답안

### 문제의 본질
현재 상황은 **코드 중복(DRY 원칙 위반)** 과 **확장성 부족** 이라는 두 가지 문제를 가지고 있습니다. 상속과 다형성은 이를 해결하는 핵심 메커니즘입니다.

### 해결 방안

```python
# 1단계: 기본 클래스(추상 클래스) 정의 - 공통점 추출
from abc import ABC, abstractmethod

class Food(ABC):  # 모든 음식의 청사진
    def __init__(self, name: str, price: int):
        self.name = name
        self.price = price
    
    @abstractmethod
    def get_cooking_time(self) -> int:
        """각 음식마다 구현해야 하는 메서드"""
        pass
    
    @abstractmethod
    def get_ingredients(self) -> list:
        """음식 특성에 따른 재료"""
        pass
    
    def get_order_info(self) -> str:
        """공통 로직 - 상속받은 모든 음식이 사용"""
        return f"{self.name} - 가격: {self.price}원, 조리시간: {self.get_cooking_time()}분"


# 2단계: 구체적인 음식 클래스들 - 각각의 특성 구현
class Pizza(Food):
    def get_cooking_time(self) -> int:
        return 15
    
    def get_ingredients(self) -> list:
        return ["밀가루", "토마토소스", "치즈"]


class Chicken(Food):
    def get_cooking_time(self) -> int:
        return 20
    
    def get_ingredients(self) -> list:
        return ["닭다리", "튀김가루", "양념"]


class KoreanFood(Food):
    def get_cooking_time(self) -> int:
        return 10
    
    def get_ingredients(self) -> list:
        return ["쌀", "된장", "고추"]


# 🆕 새로운 음식 추가 - 기존 코드 수정 없음 (OCP)
class JapaneseFood(Food):
    def get_cooking_time(self) -> int:
        return 12
    
    def get_ingredients(self) -> list:
        return ["쌀", "생선", "와사비"]


# 3단계: 주문 시스템 - 음식 종류 상관없이 처리
class OrderSystem:
    def __init__(self):
        self.orders: list[Food] = []  # 다형성의 핵심: Food 타입으로 모두 처리
    
    def add_order(self, food: Food) -> None:
        """어떤 Food 타입이든 추가 가능"""
        self.orders.append(food)
    
    def get_total_cooking_time(self) -> int:
        """각 음식의 get_cooking_time() 메서드가 동적으로 호출됨"""
        return sum(food.get_cooking_time() for food in self.orders)
    
    def print_orders(self) -> None:
        """다형성: 각 음식의 get_order_info()가 자동으로 호출"""
        for food in self.orders:
            print(food.get_order_info())  # 실제로는 Pizza, Chicken 등의 메서드가 실행됨


# 4단계: 실제 사용
if __name__ == "__main__":
    order_system = OrderSystem()
    
    # 서로 다른 타입의 음식들을 같은 방식으로 처리
    order_system.add_order(Pizza("페페로니", 18000))
    order_system.add_order(Chicken("후라이드", 16000))
    order_system.add_order(KoreanFood("비빔밥", 12000))
    order_system.add_order(JapaneseFood("초밥", 20000))
    
    # 다형성의 효과: 하나의 메서드로 모두 처리
    print("=== 주문 내역 ===")
    order_system.print_orders()
    print(f"\n총 조리 시간: {order_system.get_total_cooking_time()}분")
```

**출력 결과:**
```
=== 주문 내역 ===
페페로니 - 가격: 18000원, 조리시간: 15분
후라이드 - 가격: 16000원, 조리시간: 20분
비빔밥 - 가격: 12000원, 조리시간: 10분
초밥 - 가격: 20000원, 조리시간: 12분

총 조리 시간: 57분
```

### 핵심 개선사항 정리

| 문제 | 해결 방법 | 효과 |
|------|---------|------|
| **코드 중복** | 공통 로직을 기본 클래스에 작성 | `get_order_info()`는 1번만 구현 |
| **새 음식 추가 어려움** | 기본 클래스 상속 후 필수 메서드만 구현 | 기존 코드 수정 없음 (OCP 준수) |
| **음식별 처리 필요** | 다형성으로 `Food` 타입 하나로 통일 | `OrderSystem`은 음식 종류를 알 필요 없음 |

## 핵심 개념 정리

### 🏗️ **상속(Inheritance)**
- **정의**: 기본 클래스의 속성과 메서드를 자식 클래스가 물려받음
- **목적**: 코드 중복 제거, 공통 인터페이스 제공
- **이 문제에서**: `Food` 기본 클래스가 모든 음식의 공통 구조 정의
  - 공통 속성: `name`, `price`
  - 공통 메서드: `get_order_info()`
  - 강제할 메서드: `get_cooking_time()`, `get_ingredients()`

### 🔄 **다형성(Polymorphism)**
- **정의**: 같은 메서드명이 객체마다 다르게 동작하는 특성
- **종류**:
  - **컴파일 타임 다형성(Method Overloading)**: 같은 메서드명, 다른 매개변수
  - **런타임 다형성(Method Overriding)**: 자식 클래스가 부모 메서드를 재구현
- **이 문제에서**: `get_cooking_time()`
  ```python
  pizza = Pizza("마르게리타", 15000)
  pizza.get_cooking_time()  # 15분 반환 (Pizza의 구현)
  
  chicken = Chicken("양념", 16000)
  chicken.get_cooking_time()  # 20분 반환 (Chicken의 구현)
  
  # 같은 메서드명인데 다른 결과! 이것이 다형성
  ```

### 📋 **추상 클래스(Abstract Class)**
```python
from abc import ABC, abstractmethod

class Food(ABC):  # ABC를 상속받으면 추상 클래스가 됨
    @abstractmethod  # 이 메서드는 반드시 자식에서 구현해야 함
    def get_cooking_time(self) -> int:
        pass

# Food() 직접 생성 불가능 → 반드시 자식 클래스를 만들어야 함
# food = Food()  # TypeError!
```

### 🎯 **SOLID 원칙과의 연관**
- **O (Open/Closed Principle)**: 확장에는 열려있고, 수정에는 닫혀있음
  - ✅ 새 음식 추가 시: `Food` 상속만으로 가능
  - ✅ 기존 코드: 수정 불필요
- **L (Liskov Substitution Principle)**: 자식은 부모 역할을 완벽히 대체 가능
  ```python
  def process_food(food: Food):  # Food 타입으로 선언
      print(food.get_cooking_time())
  
  process_food(Pizza(...))      # 어떤 자식이든 문제없음
  process_food(Chicken(...))
  process_food(JapaneseFood(...))
  ```
