### 1. 🎤 면접 질문

# 캡슐화와 접근 제어자 - 모범 답변

## 모범 답안

### 1단계: 문제 상황 설명
"은행 계좌 시스템에서 잔액을 직접 수정할 수 있다면 심각한 문제들이 발생합니다:

- **무결성 위반**: 고객이 의도적으로 잔액을 1000000원으로 조작할 수 있습니다
- **비즈니스 로직 무시**: 출금 시 잔액 확인, 수수료 차감 같은 필수 규칙들을 우회할 수 있습니다
- **감시 불가**: 언제 누가 잔액을 변경했는지 추적할 수 없습니다
- **보안 위협**: 외부 공격자가 더 쉽게 시스템을 조작할 수 있습니다"

### 2단계: 접근 제어자의 필요성
"접근 제어자는 **데이터 보호와 제어 흐름 강제**를 위해 필수입니다:

```java
public class BankAccount {
    private double balance;  // 직접 접근 불가
    
    // 유일한 수정 경로: 비즈니스 로직이 포함된 메서드
    public void withdraw(double amount) {
        if (amount > balance) {
            throw new InsufficientFundsException("잔액 부족");
        }
        if (amount <= 0) {
            throw new InvalidAmountException("금액은 0보다 커야 함");
        }
        balance -= amount;
        recordTransaction("출금", amount);  // 감시 기록
    }
    
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new InvalidAmountException("금액은 0보다 커야 함");
        }
        balance += amount;
        recordTransaction("입금", amount);
    }
    
    // 조회는 읽기 전용으로 제공
    public double getBalance() {
        return balance;
    }
}
```

**이를 통해**:
- 모든 수정이 검증된 메서드를 거침
- 비즈니스 규칙을 강제할 수 있음
- 모든 변경사항을 로깅할 수 있음"

### 3단계: 꼬리 질문 해결 (읽기 접근성)
"잔액 조회는 **public getter 메서드**로 제공합니다:

```java
public class BankAccount {
    private double balance;
    
    // public이므로 누구나 조회 가능
    public double getBalance() {
        return balance;  // 읽기만 가능, 쓰기 불가
    }
}

// 사용
BankAccount account = new BankAccount();
double myBalance = account.getBalance();  // ✅ 가능
account.balance = 1000000;  // ❌ 컴파일 에러
```

**핵심**: 접근 제어자는 차단이 아니라 **제어된 접근**입니다. 
- `private`: 내부에서만 수정
- `public getter`: 외부에서 조회 가능
- `public method`: 검증 로직이 있는 메서드로만 수정 가능"

## 핵심 개념 정리

### 📌 접근 제어자 4가지

| 제어자 | 같은 클래스 | 같은 패키지 | 상속 클래스 | 외부 |
|--------|----------|----------|----------|-----|
| **private** | ✅ | ❌ | ❌ | ❌ |
| **default** (기본) | ✅ | ✅ | ❌ | ❌ |
| **protected** | ✅ | ✅ | ✅ | ❌ |
| **public** | ✅ | ✅ | ✅ | ✅ |

### 📌 캡슐화(Encapsulation)의 3가지 효과

1. **데이터 무결성**
   - 유효하지 않은 값으로부터 보호
   - 일관된 상태 유지

2. **제어된 인터페이스**
   - 내부 구현 변경해도 외부 API 유지 가능
   - 변경의 영향 범위 제한

3. **검증 및 로깅**
   - 모든 접근/수정에 검증 적용
   - 감시 기록(audit log) 생성

### 📌 실제 예시: 점수 관리 시스템

```java
// ❌ 나쁜 예: public 필드
public class Student {
    public int score;  // 누구나 마음대로 수정
}

student.score = -100;  // 유효성 검증 없음!

---

// ✅ 좋은 예: private + getter/setter
public class Student {
    private int score;
    
    public void setScore(int score) {
        if (score < 0 || score > 100) {
            throw new IllegalArgumentException("점수는 0~100 사이여야 함");
        }
        this.score = score;
    }
    
    public int getScore() {
        return score;
    }
}

student.setScore(-100);  // ❌ 예외 발생 - 보호됨!
```

    private OrderStatus status;

    @PrePersist

            throw new IllegalStateException("주문 금액은 0보다 커야 함");
        }

    public void cancel() {

        }
        this.status = OrderStatus.CANCELLED;
    }
}
