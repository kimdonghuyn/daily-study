## 모범 답안

### 메인 질문 답변

네, 좋은 비유입니다. 캡슐화와 접근 제어자는 정확히 그 은행 시스템이 작동하는 방식을 프로그래밍에 적용한 것입니다.

**은행 비유로 설명하면:**
- **공개되면 안 되는 것**: 통장 잔액 데이터 (private)
- **제어된 방식으로만 접근**: 출금/입금 같은 메서드 (public)
- **검증 로직**: 은행원이 거래 전 신원 확인, 잔액 확인 등을 하는 것처럼, 메서드 내에서 데이터 유효성 검사

```java
// ❌ 캡슐화가 없는 경우 (은행이 개방된 경우)
class BankAccount {
    public double balance = 1000000;  // 누구나 직접 수정 가능
}

// 사용자가 직접 수정
account.balance = -999999;  // 말도 안 되는 거래 가능!

// ✅ 캡슐화된 경우 (은행이 관리하는 경우)
class BankAccount {
    private double balance = 1000000;  // 직접 접근 불가
    
    public void withdraw(double amount) {
        // 1단계: 유효성 검사
        if (amount <= 0) {
            throw new IllegalArgumentException("출금액은 양수여야 합니다");
        }
        
        // 2단계: 잔액 확인 (비즈니스 로직)
        if (amount > balance) {
            throw new InsufficientFundsException("잔액이 부족합니다");
        }
        
        // 3단계: 감사 로그
        logger.info("출금: {} 원", amount);
        
        // 4단계: 실행
        balance -= amount;
    }
    
    public double getBalance() {
        return balance;
    }
}
```

**핵심**: 접근 제어자는 **신뢰 관계의 경계**를 정의합니다. 내부 상태는 보호하고, 정해진 인터페이스(메서드)를 통해서만 상호작용하도록 강제하는 것입니다.

// 실제 사례: 이커머스 시스템
class ShoppingCart {
    public List<Item> items = new ArrayList<>();  // public
}

// 결제 시스템을 거치지 않고 직접 조작 가능
ShoppingCart cart = new ShoppingCart();
cart.items.add(expensiveItem);
cart.items.add(expensiveItem);
// 프로모션 할인, 재고 확인, 결제 검증 등을 모두 우회!

// ✅ 개선된 코드
class ShoppingCart {
    private List<Item> items = new ArrayList<>();
    private int promotionCode;
    private Inventory inventory;
    
    public void addItem(Item item) {
        // 1. 재고 확인
        if (!inventory.isAvailable(item.getId())) {
            throw new OutOfStockException();
        }
        
        // 2. 재고 감소 처리
        inventory.reserve(item.getId());
        
        // 3. 가격 계산 (프로모션 적용)
        item.applyDiscount(promotionCode);
        
        // 4. 추가
        items.add(item);
    }
    
    public List<Item> getItems() {
        return Collections.unmodifiableList(items);  // 외부에서 수정 불가
    }
}
```

**4️⃣ 멀티스레드 환경에서의 혼란**
```java
// ❌ public 필드
class Account {
    public double balance = 10000;
}

// 스레드 1
Thread t1 = new Thread(() -> {
    account.balance -= 5000;  // 잔액 읽고 → 계산 → 수정
});

// 스레드 2
Thread t2 = new Thread(() -> {
    account.balance -= 3000;  // 동시에 접근하면?
});

// 결과: 기댓값 2000 이지만, 실제로는 5000 또는 7000이 될 수 있음
// (Race Condition)

// ✅ synchronized 메서드로 보호
class Account {
    private double balance = 10000;
    
    public synchronized void withdraw(double amount) {
        if (amount > balance) throw new Exception();
        balance -= amount;  // 원자적 연산
    }
}
```

## 핵심 개념 정리

### 📌 접근 제어자 (Access Modifiers)

| 제어자 | 범위 | 언제 사용 |
|--------|------|----------|
| **private** | 같은 클래스 내부만 | 내부 구현 세부사항 |
| **package-private** (default) | 같은 패키지 | 패키지 내 관련 클래스들 |
| **protected** | 상속 관계 + 같은 패키지 | 하위 클래스가 확장할 부분 |
| **public** | 모든 곳 | 공개 API, 사용자가 사용할 인터페이스 |

### 🎯 캡슐화의 핵심 원칙

1. **데이터 은닉 (Information Hiding)**
   - 내부 상태는 `private`으로 보호
   - 필요한 정보만 getter로 제공

2. **제어된 접근 (Controlled Access)**
   - 데이터 변경은 `setter` 메서드를 통해
   - setter에서 유효성 검사 수행

3. **불변성 보장 (Invariant Protection)**
   - 객체의 일관된 상태 유지
   - 논리적으로 맞지 않는 상태 방지

### 📋 Java에서의 캡슐화 베스트 프랙티스

```java
// 1. 필드는 항상 private
private String name;
private int age;

// 2. 필드별로 getter/setter 제공
public String getName() {
    return name;
}

public void setName(String name) {
    if (name == null || name.trim().isEmpty()) {
        throw new IllegalArgumentException("이름은 필수입니다");
