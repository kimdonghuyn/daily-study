### 캡슐화와 접근 제어자의 역할

**개념:**
캡슐화는 **데이터와 메서드를 하나의 단위로 묶고, 외부 접근을 제한**하는 원칙입니다. 접근 제어자는 이를 구현하는 메커니즘입니다.

```java
// ✅ 좋은 예: 캡슐화 적용
class BankAccount {
    private double balance;  // 내부 상태 보호
    
    public BankAccount(double initialBalance) {
        if (initialBalance >= 0) {
            this.balance = initialBalance;
        } else {
            throw new IllegalArgumentException("초기 잔액은 0 이상이어야 합니다");
        }
    }
    
    // 외부에서 접근 가능한 통로를 제한적으로 제공
    public void withdrawal(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("출금액은 양수여야 합니다");
        }
        if (amount > balance) {
            throw new IllegalArgumentException("잔액 부족");
        }
        balance -= amount;
    }
    
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("입금액은 양수여야 합니다");
        }
        balance += amount;
    }
    
    public double getBalance() {
        return balance;  // 읽기만 가능
    }
}

// 사용 예
BankAccount account = new BankAccount(1000);
account.withdrawal(500);    // ✅ 유효한 검증 통과
// account.withdrawal(-100);  // ❌ 예외 발생
// account.balance = -1000;    // ❌ 컴파일 에러
```

### 꼬리 질문 답변: private + public 메서드가 안전한 이유

**5가지 핵심 이유:**

1. **단일 진입점(Single Entry Point)**
   ```java
   // 모든 변경이 withdrawal() 메서드를 통해서만 가능
   // → 검증 로직이 한 곳에 집중되어 통제 가능
   ```

2. **불변식(Invariant) 보장**
   ```java
   // balance는 항상 >= 0이라는 조건을 보장할 수 있음
   // withdrawal()과 deposit()에서만 변경 가능하므로
   ```

3. **비즈니스 로직 강제**
   ```java
   // 잔액 부족 확인, 금액 검증 등을 반드시 거쳐야 함
   // 이를 우회할 수 없음
   ```

4. **내부 구현 변경 가능**
   ```java
   class BankAccount {
       // 나중에 balance를 List<Transaction>으로 변경해도
       // public 메서드는 동일하게 동작
       private List<Transaction> transactions = new ArrayList<>();
       
       public void withdrawal(double amount) {
           // 내부 구현이 변경되어도 외부 코드는 영향 없음
       }
   }
   ```

5. **감시와 추적 가능**
   ```java
   public void withdrawal(double amount) {
       // 로깅, 모니터링, 감사 추적 추가 가능
       logger.info("출금: {} → 잔액: {}", amount, balance);
       // ...
   }
   ```

## 핵심 개념 정리

### 📌 접근 제어자 (Access Modifiers)

| 제어자 | 범위 | 사용 목적 |
|--------|------|---------|
| **public** | 모든 곳에서 접근 가능 | 외부 인터페이스 제공 |
| **protected** | 같은 패키지 + 상속 클래스 | 상속 구조에서 공유 |
| **default** | 같은 패키지 내에서만 | 패키지 내부 협력 |
| **private** | 해당 클래스 내에서만 | 내부 세부사항 숨김 |

### 📌 캡슐화의 4가지 목표

1. **데이터 보호** - 잘못된 상태 변경 방지
2. **복잡도 감소** - 사용자는 인터페이스만 알면 됨
3. **유지보수성 향상** - 내부 변경이 외부에 영향 없음
4. **보안** - 민감한 정보 접근 제한

### 📌 Getter/Setter 패턴

```java
// ❌ 피해야 할 패턴: getter/setter 남발
private double balance;
public double getBalance() { return balance; }
public void setBalance(double balance) { this.balance = balance; }
// → 이렇게 하면 캡슐화 효과 없음!

// ✅올바른 패턴: 비즈니스 의도가 있는 메서드
public void withdrawal(double amount) { /* 검증 */ }
public void deposit(double amount) { /* 검증 */ }
public double getBalance() { return balance; }  // 읽기만 가능
```

### 📌 객체 지향 5대 원칙 (SOLID) 중 관련 개념

- **Single Responsibility** - 한 클래스는 한 책임만
- **Open/Closed** - 확장에는 열려있고 수정에는 닫혀있음
- **Liskov Substitution** - 하위 타입은 상위 타입 계약 준수

## 실무 포인트

### 🏢 실제 적용 사례

#### 1. JPA Entity 설계
```java
@Entity
@Table(name = "accounts")
public class Account {
    @Id
    private Long id;
    
    @Column(nullable = false)
    private BigDecimal balance;
    
    // private 생성자 + Builder 패턴
    private Account() {}
    
    public static AccountBuilder builder() {
        return new AccountBuilder();
    }
    
    // 비즈니스 메서드만 노출
    public void withdraw(BigDecimal amount) {
        if (amount.compareTo(balance) > 0) {
            throw new InsufficientFundsException();
        }
        this.balance = balance.subtract(amount);
    }
    
    // 읽기만 허용
    public BigDecimal getBalance() {
        return balance;
    }
}
```

#### 2. API Response DTO
```java
@Data
@Builder
public class AccountResponse {
    private Long accountId;
    private BigDecimal balance;
    private LocalDateTime lastTransaction;
    
    // 민감한 정보는 DTO에 포함하지 않음
    // (예: 비밀번호, 내부 상태)
}
```

#### 3. Service Layer에서의 캡슐화
```java
@Service
public class AccountService {
    
    // 비즈니스 로직이 한 곳에 집중
    public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
        Account from = accountRepository.findById(fromId)
            .orElseThrow(() -> new AccountNotFoundException());
        Account to = accountRepository.findById(toId)
            .orElseThrow(() -> new AccountNotFoundException());
        
        // 원자성 보장
        from.withdraw(amount);
        to.deposit(amount);
        
        accountRepository.saveAll(List.of(from, to));
        transactionLog.record(fromId, toId, amount);
    }
}
```
