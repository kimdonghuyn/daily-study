## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# JVM 메모리 구조와 OutOfMemoryError 분석

## 모범 답안

당신의 OutOfMemoryError와 선배의 조언은 **서로 다른 메모리 영역의 문제**입니다.

### 상황 분석

**당신의 상황 (Heap 메모리 부족)**
```java
List<UserData> users = new ArrayList<>();
while (true) {
    users.add(new UserData()); // 계속 객체 생성
    // GC가 제거할 수 없는 참조 유지 → Heap 고갈
}
// Exception in thread "main" java.lang.OutOfMemoryError: Java heap space
```

- **원인**: 사용자 데이터 객체들이 Heap에 계속 쌓임
- **특징**: 프로그램이 천천히 메모리 사용량 증가
- **해결책**: 데이터 저장 방식 재검토, GC 튜닝, 메모리 누수 제거

**선배의 조언 (Stack 메모리 부족)**
```java
void recursiveFunction(int n) {
    // 깊은 재귀 호출 시 각 함수 호출이 Stack에 쌓임
    recursiveFunction(n + 1);
}
// Exception in thread "main" java.lang.StackOverflowError
```

- **원인**: 함수 호출 스택이 한계 도달
- **특징**: 보통 갑작스럽고 깊은 재귀에서 발생
- **해결책**: 재귀 깊이 제한, 반복문으로 변경

## 핵심 개념 정리

### 1️⃣ **Heap 메모리**
- **용도**: 동적으로 생성된 **객체(인스턴스)** 저장
- **생명주기**: 명시적 삭제 또는 GC가 참조 없음을 감지할 때까지
- **크기**: 시작 시 `-Xms`, 최대 `-Xmx` 옵션으로 설정
- **멀티스레드**: 모든 스레드가 공유 → 동기화 필요
- **성능**: 할당/해제가 느림, GC 오버헤드 발생

```
[Heap 구조]
Young Generation (Eden, S0, S1)  ← 새로운 객체
                    ↓ (Minor GC)
Old Generation                    ← 오래된 객체
                    ↓ (Major GC)
Permanent/Metaspace              ← 클래스 정보
```

### 2️⃣ **Stack 메모리**
- **용도**: 함수 호출 시 **지역변수, 메서드 참조, 반환값** 저장
- **생명주기**: 함수 실행 종료 시 자동 해제
- **크기**: 스레드당 고정 (보통 1MB)
- **멀티스레드**: 각 스레드가 독립적 Stack 소유
- **성능**: 할당/해제가 매우 빠름 (LIFO)

```
[Stack 호출 스택]
main() 호출
├─ functionA() 호출
│  ├─ functionB() 호출
│  │  └─ functionC() 호출  ← 현재 위치
│  │     [지역변수 c, 임시값]
│  └─ [지역변수 a, b]
└─ [지역변수 m]
```

### 3️⃣ **Method Area (Metaspace)**
- **용도**: 클래스 구조, 메서드 코드, 상수 풀, static 변수
- **공유**: 모든 스레드 공유
- **특징**: Java 8 이후 Native Memory에서 관리 (이전: Heap의 PermGen)

## 실무 포인트

### 🚨 **OutOfMemoryError 대응 체크리스트**

```java
// 1. 메모리 누수 탐지
// 불필요한 참조 유지 제거
List<Data> cache = new ArrayList<>(); // ❌ 제한 없음
List<Data> cache = new LinkedHashMap<>(16, 0.75f, true) {
    protected boolean removeEldestEntry(Map.Entry eldest) {
        return size() > 1000; // ✅ 크기 제한
    }
};

// 2. 대용량 데이터 처리 최적화
// ❌ 전체 로드
List<User> users = database.loadAll();

// ✅ 스트리밍 처리
try (Stream<User> stream = database.stream()) {
    stream.filter(u -> u.isActive())
          .forEach(this::processUser);
}

// 3. WeakReference 활용
Map<String, WeakReference<BigObject>> cache = new WeakHashMap<>();
```

### 🔍 **문제별 진단 전략**

| 문제 | 증상 | 원인 | 해결책 |
|------|------|------|--------|
| **Heap 부족** | 천천히 증가 후 OutOfMemoryError | 메모리 누수, 무제한 캐싱 | GC 로그 분석, 프로파일링 |
| **Stack 부족** | StackOverflowError | 깊은 재귀, 상호 재귀 호출 | 재귀 깊이 제한, BFS 변경 |
| **Metaspace 부족** | OutOfMemoryError: Metaspace | 동적 클래스 로딩 과다 | ClassLoader 누수 제거 |

### 💡 **면접 추가 질문 대비**

**Q. GC는 왜 Heap만 정리하고 Stack은 안 정리하나요?**
```
A. Stack은 LIFO 구조로 자동 정리되므로 GC가 불필요합니다.
   함수 종료 = 자동 메모리 해제이기 때문입니다.
```

**Q. Heap과 Stack 크기를 크게 늘리면 안 되나요?**
```
A. Heap은 GC 오버헤드 증가, Stack은 스레드 수 제한으로
   전체 성능 악화 및 응답 시간 증가가 발생합니다.
```

## 결론

> **Heap 문제 = "데이터를 어디에 저장할까?"** (저장소 전략)  
> **Stack 문제 = "함수를 어떻게 호출할까?"** (호출 구조)

두 영역을 이해하면 메모리 문제의 **원인 파악 → 진단 → 해결**이 훨씬 명확해집니다.
