## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 상속과 다형성을 활용한 게임 직업 시스템 설계

## 모범 답안

### ❌ 문제 있는 기존 코드 (if-else 지옥)

```java
// 직업 선택에 따라 매번 확인해야 함
if (jobType == "KNIGHT") {
    attackWithSword();
} else if (jobType == "WIZARD") {
    castFireball();
} else if (jobType == "ARCHER") {
    shootArrow();
}
// 새 직업 추가 시마다 여기를 수정해야 함!
```

**문제점:**
- 새로운 직업 추가 시 기존 코드 수정 필요 (OCP 위반)
- 직업 추가할 때마다 if-else 문이 늘어남
- 코드가 복잡해지고 버그 발생 위험 증가

---

### ✅ 상속과 다형성으로 해결

```java
// 1️⃣ 부모 클래스: 모든 직업의 공통 틀
abstract class Job {
    abstract void attack();  // 각 직업이 자신의 방식으로 구현
}

// 2️⃣ 자식 클래스들: 각 직업별 구현
class Knight extends Job {
    @Override
    void attack() {
        System.out.println("칼로 베기!");
    }
}

class Wizard extends Job {
    @Override
    void attack() {
        System.out.println("파이어볼 발사!");
    }
}

class Archer extends Job {
    @Override
    void attack() {
        System.out.println("화살 쏘기!");
    }
}

// 3️⃣ 플레이어 클래스
class Player {
    private Job job;  // 부모 타입으로 선언
    
    public Player(Job job) {
        this.job = job;
    }
    
    // ⭐️ 핵심: if-else 없이 다형성으로 처리
    public void performAttack() {
        job.attack();  // 실제 객체의 attack()이 실행됨
    }
}

// 4️⃣ 사용 코드 (가장 깔끔함!)
Player player1 = new Player(new Knight());
Player player2 = new Player(new Wizard());
Player player3 = new Player(new Archer());

player1.performAttack();  // "칼로 베기!" 출력
player2.performAttack();  // "파이어볼 발사!" 출력
player3.performAttack();  // "화살 쏘기!" 출력
```

---

### 🆕 새로운 직업 추가 (기존 코드 수정 없음!)

```java
// 새로운 직업 추가 - 기존 코드는 건드릴 필요 없음!
class Paladin extends Job {
    @Override
    void attack() {
        System.out.println("신성한 빛으로 공격!");
    }
}

// 그냥 추가하면 끝!
Player player4 = new Player(new Paladin());
player4.performAttack();  // 자동으로 작동
```

**차이점:**
- ✅ 새 직업 추가 시 **새 클래스만 만들면 됨**
- ✅ 기존 코드 수정 불필요 (OCP 준수)
- ✅ 플레이어의 performAttack() 메서드는 영원히 변하지 않음

## 핵심 개념 정리

### 📌 상속 (Inheritance)
- **정의**: 부모 클래스의 속성과 메서드를 자식 클래스가 물려받는 것
- **역할**: 공통된 특징을 한 곳에서 관리
- **예제**: 모든 Job은 attack() 메서드를 가짐
- **장점**: 코드 중복 제거, 유지보수 용이

### 📌 다형성 (Polymorphism)
- **정의**: 같은 메서드명이지만 **실제 실행되는 코드는 객체의 실제 타입에 따라 결정**
- **메커니즘**: 컴파일 타임에는 Job 타입으로 인식, 런타임에는 실제 객체(Knight, Wizard 등)의 메서드 실행
- **핵심**: `job.attack()` 한 줄이 Knight면 칼 공격, Wizard면 파이어볼 실행

```java
Job job = new Knight();      // 다형성: Job 타입이지만 Knight 객체
job.attack();                 // Knight의 attack() 실행
```

### 📌 OCP (Open-Closed Principle)
- **확장에는 열려있고, 수정에는 닫혀있다**
- 새 직업 추가(확장) ✅ → 기존 코드 수정(수정) ✗

## 실무 포인트

### 🎯 1. 의존성 역전 (느슨한 결합)

```java
// ❌ 나쁜 예: 구체적인 클래스에 의존
class Player {
    private Knight knight;
    private Wizard wizard;
    // 직업이 추가될 때마다 필드 추가...
}

// ✅ 좋은 예: 추상화된 Job에만 의존
class Player {
    private Job job;  // 어떤 직업이든 상관없음
}
```

### 🎯 2. 런타임 타입 결정

```java
// 게임 시작 시 직업 선택
Job selectedJob = jobFactory.createJob(playerChoice);  // 사용자 입력에 따라 결정
Player player = new Player(selectedJob);

// 게임 실행 중 공격
player.performAttack();  // if-else 없이도 올바른 공격 실행!
```

### 🎯 3. 팩토리 패턴과의 결합

```java
class JobFactory {
    public static Job createJob(String jobType) {
        return switch(jobType) {
            case "KNIGHT" -> new Knight();
            case "WIZARD" -> new Wizard();
            case "ARCHER" -> new Archer();
            default -> throw new IllegalArgumentException();
        };
    }
}

// if-else를 한곳으로 격리 (변경 포인트 최소화)
Job job = JobFactory.createJob(userInput);
```

### 🎯 4. 인터페이스 사용 (더 나은 방식)

```java
// 추상 클래스보다 더 유연한 인터페이스
interface Attackable {
    void attack();
}

class Knight implements Attackable {
    @Override
    public void attack() {
        System.out.println("칼로 베기!");
    }
}

// 장점: 다중 구현 가능, 더 깔끔한 계약
```

### 🎯 5. 메서드 오버로딩 vs 오버라이딩

```java
// ❌ 오버로딩 (같은 클래스 내에서): 복잡함
void attack(Knight k) { }
void attack(Wizard w) { }
void attack(Archer a) { }

// ✅ 오버라이딩 (다형성): 깔끔함
job.attack();  // 타입에 따라 자동으로 올바른 메서드 호출
```

## 💡 정리하면

| 항목 | 기존 방식 | 상속/다형성 |
|------|---------|-----------|
| 코드 가독성 | if-else 지옥 | 한 줄: `job.attack()` |
| 새 직업 추가 | 기존 코드 수정 필요 | 새 클래스만 추가 |
| 유지보수성 | 낮음 (버그 위험) | 높음 (변경 격리) |
| 확장성 | 제한적 | 무한 확장 가능 |

**결론**: 상속과 다형성은 단순히 OOP 개념이 아니라, **소프트웨어의 변화에 대응하는 실무 전략**입니다.
