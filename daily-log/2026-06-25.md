### 1. 🔧 문제 상황

# JVM 메모리 구조와 OutOfMemoryError 진단

## 모범 답안

**OutOfMemoryError의 원인을 특정하려면 각 메모리 영역의 역할을 명확히 이해해야 합니다.**

### 1️⃣ **세 메모리 영역의 역할**

```
┌─────────────────────────────────────────┐
│         JVM 메모리 구조                  │
├─────────────────────────────────────────┤
│  Method Area (클래스 정보, 상수 풀)      │ → 고정 크기 (대부분)
│  - 클래스 메타데이터, 필드/메서드 정보   │
│  - 상수 풀, 코드, 정적 변수              │
├─────────────────────────────────────────┤
│  Stack (각 스레드마다 독립적)            │ → 스레드당 고정 크기
│  - 로컬 변수, 메서드 호출 정보           │
│  - 참조값(객체 주소)                     │
├─────────────────────────────────────────┤
│  Heap (모든 객체와 배열)                 │ → 동적 크기
│  - 실제 객체 인스턴스                    │
│  - GC의 대상                            │
└─────────────────────────────────────────┘
```

### 2️⃣ **각 메모리가 가득 찼을 때**

| 메모리 영역 | 저장 내용 | 가득 찼을 때 | 원인 |
|-----------|---------|-----------|------|
| **Heap** | 객체 인스턴스 | `OutOfMemoryError: Java heap space` | 메모리 누수, 대용량 데이터 로딩 |
| **Stack** | 메서드 호출, 로컬 변수 | `StackOverflowError` | 무한 재귀, 깊은 호출 스택 |
| **Method Area** | 클래스 메타데이터 | `OutOfMemoryError: Metaspace` | 동적 클래스 생성, ClassLoader 누수 |

### 3️⃣ **이 상황에서 가장 가능성 높은 원인**

**➡️ Heap 메모리 부족 (가능성: 80~90%)**

**이유:**
- 서버가 "계속 실행되다가" 터짐 = 메모리가 시간에 따라 증가
- 이는 **메모리 누수** 또는 **객체 생성 과다**의 전형적 패턴
- Stack이나 Method Area는 보통 시작부터 고정되어 있어 시간에 따라 증가하지 않음

## 핵심 개념 정리

### 📌 **Heap 메모리**
- **특성**: 런타임에 동적으로 할당/해제, 모든 스레드가 공유
- **저장물**: 모든 객체와 배열 (`new`로 생성된 것들)
- **초기 크기**: `-Xms` (예: `-Xms512m`)
- **최대 크기**: `-Xmx` (예: `-Xmx2g`)
- **GC 대상**: Yes (가비지 컬렉션이 자동으로 정리)
- **문제 징후**:
  ```
  Exception in thread "main" java.lang.OutOfMemoryError: Java heap space
  ```

### 📌 **Stack 메모리**
- **특성**: 각 스레드마다 독립적인 스택 생성, 선입선출(LIFO)
- **저장물**: 
  - 메서드 호출 정보
  - 로컬 변수 (int, boolean 등 primitive 타입과 참조값)
  - 메서드 실행 context
- **크기**: 스레드당 `-Xss` (예: `-Xss1m`)
- **메모리 해제**: 메서드 반환 시 자동 해제 (GC 불필요)
- **문제 징후**:
  ```java
  // 무한 재귀
  void recursion() {
      recursion(); // StackOverflowError 발생
  }
  ```

### 📌 **Method Area (Metaspace)**
- **특성**: 모든 스레드가 공유, 클래스별로 한 번만 로드
- **저장물**:
  - 클래스 구조 (필드, 메서드, 생성자 정보)
  - Runtime Constant Pool (문자열, 숫자 상수)
  - 정적 변수(static)
  - 바이트코드
- **크기**: Java 8 이후 Native Memory 사용 (`-XX:MetaspaceSize`, `-XX:MaxMetaspaceSize`)
- **GC 대상**: Yes (클래스 언로드 시 정리)
- **문제 징후**:
  ```
  Exception in thread "main" java.lang.OutOfMemoryError: Metaspace
  ```

### 🛠️ **상황별 진단 및 해결**

```java
// 시나리오 1: Heap 메모리 부족
// 증상: 시간 지남에 따라 메모리 사용량 증가 → 결국 OutOfMemoryError
// 해결:
// - Xmx 크기 증가 (임시 방편)
// - 메모리 프로파일링으로 누수 지점 찾기 (근본 해결)
// - 불필요한 객체 참조 제거

// 시나리오 2: Stack 오버플로우
// 증상: 특정 요청에서 즉시 StackOverflowError 발생
// 해결:
void badRecursion(int n) {
    if (n == 0) return;
    badRecursion(n - 1); // 깊이가 깊으면 스택 오버플로우
}
// → 재귀를 반복문으로 변경하거나 재귀 깊이 제한

// 시나리오 3: Metaspace 부족
// 증상: 동적으로 클래스를 생성하는 경우 (리플렉션, 프록시 대량 생성)
// 해결:
// - MaxMetaspaceSize 증가
// - 불필요한 클래스 언로드 확인
// - ClassLoader 누수 제거
```
