# Spring Bean 생명주기와 스코프 - 모범 답안

## 모범 답안

### 메인 질문: Bean이란 무엇이고 왜 Spring이 관리할까?

**Bean의 정의:**
Bean은 Spring IoC 컨테이너가 생성, 관리, 소멸시키는 객체입니다. 단순한 Java 객체가 아니라 Spring의 관리 대상이 되는 객체를 의미합니다.

**왜 `new` 대신 Spring에게 맡길까? → 4가지 핵심 이유**

1. **의존성 주입(Dependency Injection)**
   ```java
   // ❌ new 키워드 사용 - 강한 결합
   public class UserService {
       private UserRepository repo = new UserRepository(); // 직접 생성
   }
   
   // ✅ Spring Bean - 느슨한 결합
   @Service
   public class UserService {
       @Autowired
       private UserRepository repo; // Spring이 주입
   }
   ```
   객체 간 의존성을 외부에서 주입받으므로 테스트와 유지보수가 쉬워집니다.

2. **생명주기 관리**
   - 생성 → 초기화 → 사용 → 소멸의 과정을 Spring이 자동으로 관리
   - 개발자는 비즈니스 로직에만 집중

3. **싱글톤 패턴 자동 적용**
   ```java
   // Spring이 자동으로 한 번만 생성하고 재사용
   @Bean
   public DataSource dataSource() {
       return new DataSource();
   }
   // 매번 new를 하지 않아도 메모리 효율적
   ```

4. **AOP, 트랜잭션, 보안 등 기능 추가 용이**
   ```java
   @Service
   public class UserService {
       @Transactional  // Spring이 자동으로 트랜잭션 처리
       public void saveUser(User user) { ... }
   }
   ```

### 꼬리 질문: Bean 생명주기를 왜 알아야 할까?

**정확한 답변:**
네, 정확히 그 이유 때문에 생명주기를 알아야 합니다!

**실제 시나리오:**
```java
@Component
public class DatabaseInitializer {
    private DatabaseConnection conn;
    
    // ❌ 생성자에서 초기화하면?
    public DatabaseInitializer() {
        this.conn = new DatabaseConnection();
        this.conn.connect(); // 아직 다른 Bean들이 주입 안 됨 - 위험
    }
    
    // ✅ @PostConstruct - Bean이 완전히 생성된 후 실행
    @PostConstruct
    public void init() {
        this.conn = new DatabaseConnection();
        this.conn.connect(); // 모든 의존성이 주입된 후 실행
        this.conn.runMigrations();
    }
}
```

**Spring Bean 생명주기 5단계:**

```
1️⃣ 인스턴스화(Instantiation)
   ↓
2️⃣ 의존성 주입(Dependency Injection)
   ↓
3️⃣ 초기화(@PostConstruct)
   ↓
4️⃣ 사용(Use)
   ↓
5️⃣ 소멸(@PreDestroy)
```

## 핵심 개념 정리

### 1. Bean 생명주기 (Lifecycle Hooks)

**초기화 방법 3가지:**
```java
// 1️⃣ @PostConstruct 어노테이션 (권장)
@PostConstruct
public void init() {
    System.out.println("Bean 생성 완료 후 초기화");
}

// 2️⃣ InitializingBean 인터페이스 (레거시)
@Component
public class MyBean implements InitializingBean {
    @Override
    public void afterPropertiesSet() throws Exception {
        System.out.println("초기화");
    }
}

// 3️⃣ @Bean(initMethod="...") (XML 설정 시대의 유산)
@Configuration
public class AppConfig {
    @Bean(initMethod = "init")
    public MyBean myBean() {
        return new MyBean();
    }
}
```

**소멸 방법 3가지:**
```java
// 1️⃣ @PreDestroy (권장)
@PreDestroy
public void cleanup() {
    System.out.println("리소스 정리");
}

// 2️⃣ DisposableBean 인터페이스
@Component
public class MyBean implements DisposableBean {
    @Override
    public void destroy() throws Exception {
        System.out.println("소멸");
    }
}

// 3️⃣ @Bean(destroyMethod="...")
@Bean(destroyMethod = "close")
public MyBean myBean() {
    return new MyBean();
}
```

### 2. Bean 스코프 (Scope)

```java
// 1️⃣ Singleton (기본값) - 애플리케이션 전체에서 1개
@Component
public class SingletonBean { }

// 2️⃣ Prototype - 요청할 때마다 새로운 인스턴스
@Component
@Scope("prototype")
public class PrototypeBean { }

// 3️⃣ Request - HTTP 요청당 1개 (Web only)
@Component
@Scope("request")
public class RequestBean { }

// 4️⃣ Session - 세션당 1개 (Web only)
@Component
@Scope("session")
public class SessionBean { }

// 5️⃣ Application - ServletContext 생명주기 동안 1개
@Component
@Scope("application")
public class ApplicationBean { }
```

**스코프별 생명주기:**
- **Singleton**: 컨테이너 시작 ~ 종료
- **Prototype**: 요청 ~ 반환 (소멸은 Spring이 관리 안 함)
- **Request**: HTTP 요청 시작 ~ 종료
- **Session**: 세션 시작 ~ 종료

### 3. Prototype 스코프의 주의점

```java
@Component
@Scope("prototype")
public class PrototypeService {
    @PostConstruct
    public void init() {
        System.out.println("초기화됨: " + System.identityHashCode(this));
    }
    
    @PreDestroy
    public void cleanup() {
        // ⚠️ Prototype은 @PreDestroy가 호출 안 될 수 있음!
        System.out.println("정리됨");
    }
}
```

## 실무 포인트

### 🔴 자주 실수하는 케이스

**Case 1: Singleton 스코프에서 Prototype Bean 주입**
```java
// ❌ 잘못된 패턴
@Component
public class SingletonService {
    @Autowired
    private PrototypeService prototypeService; // 항상 같은 인스턴스
    
    public void execute() {
        // PrototypeService가 매번 같은 객체임
        prototypeService.doSomething();
    }
}

// ✅ 올바른 패턴 1: ObjectProvider 사용
@Component
public class SingletonService {
    @Autowired
    private ObjectProvider<PrototypeService> prototypeProvider;
    
    public void execute() {
        PrototypeService service = prototypeProvider.getObject(); // 매번 새로운 인스턴스
        service.doSomething();
    }
}

// ✅ 올바른 패턴 2: Provider 사용
@Component
public class SingletonService {
    @Autowired
    private Provider<PrototypeService> prototypeProvider;
    
    public void execute() {
        PrototypeService service = prototypeProvider.get();
        service.doSomething();
    }
}
```

**Case 2: @PostConstruct에서 다른 Bean 참조**
```java
// ✅ 좋은 패턴
@Component
public class DatabaseInitializer {
    private final DatabaseConfig config;
    private final Logger logger;
    
    // 생성자 주입 - @PostConstruct 실행 전에 이미 주입됨
    public DatabaseInitializer(DatabaseConfig config, Logger logger) {
        this.config = config;
        this.logger = logger;
    }
    
    @PostConstruct
    public void init() {
        logger.info("DB 초기화 시작");
        config.initialize();
    }
}
```
