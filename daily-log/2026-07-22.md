### 1. 🔧 문제 상황

# 상속과 다형성을 활용한 결제 시스템 개선

## 모범 답안

### 현재 문제점 (Bad Code)
```java
public class OrderProcessor {
    public void processPayment(String paymentMethod, double amount) {
        if (paymentMethod.equals("CREDIT_CARD")) {
            // 신용카드 처리 로직
            validateCardNumber();
            chargeCard(amount);
            recordTransaction();
        } else if (paymentMethod.equals("BANK_TRANSFER")) {
            // 계좌이체 처리 로직
            validateBankAccount();
            transferFund(amount);
            recordTransaction();
        } else if (paymentMethod.equals("MOBILE")) {
            // 휴대폰 결제 처리 로직
            validatePhoneNumber();
            chargeMobile(amount);
            recordTransaction();
        }
        // 새로운 결제 수단 추가할 때마다 여기 수정 필요 ❌
    }
}
```

**문제점:**
- `if-else` 체인이 계속 길어짐 (Open-Closed Principle 위반)
- 새 결제 수단 추가 시 기존 코드 수정 필요
- 각 결제 방식이 강하게 결합됨
- 테스트하기 어려움

### 개선된 솔루션 (Good Code)

**Step 1: 추상 클래스로 결제 인터페이스 정의**
```java
// 모든 결제 수단이 따를 계약(Contract) 정의
public abstract class PaymentProcessor {
    public final void process(double amount) {
        // Template Method Pattern과 함께 사용
        validate();
        pay(amount);
        confirm();
    }
    
    // 각 결제 수단이 반드시 구현해야 할 메서드
    protected abstract void validate();
    protected abstract void pay(double amount);
    protected abstract void confirm();
}
```

**Step 2: 구체적인 결제 수단 구현 (다형성)**
```java
// 신용카드 결제
public class CreditCardProcessor extends PaymentProcessor {
    private String cardNumber;
    
    @Override
    protected void validate() {
        // 카드번호 유효성 검사 (Luhn 알고리즘 등)
        if (!isValidCardNumber(cardNumber)) {
            throw new InvalidPaymentException("Invalid card");
        }
    }
    
    @Override
    protected void pay(double amount) {
        // 신용카드 결제 처리 (PG사 API 호출)
        callPaymentGateway(amount);
    }
    
    @Override
    protected void confirm() {
        // 영수증 발급, 포인트 적립 등
        issueReceipt();
        addPoints();
    }
}

// 계좌이체
public class BankTransferProcessor extends PaymentProcessor {
    private String bankAccount;
    
    @Override
    protected void validate() {
        if (!isValidBankAccount(bankAccount)) {
            throw new InvalidPaymentException("Invalid account");
        }
    }
    
    @Override
    protected void pay(double amount) {
        transferToMerchantAccount(amount);
    }
    
    @Override
    protected void confirm() {
        notifyBankTransaction();
    }
}

// 휴대폰 결제
public class MobilePaymentProcessor extends PaymentProcessor {
    private String phoneNumber;
    
    @Override
    protected void validate() {
        if (!isValidPhoneNumber(phoneNumber)) {
            throw new InvalidPaymentException("Invalid phone");
        }
    }
    
    @Override
    protected void pay(double amount) {
        chargeMobileCarrier(amount);
    }
    
    @Override
    protected void confirm() {
        sendSMS("결제가 완료되었습니다.");
    }
}

// 미래에 추가될 암호화폐 결제 (기존 코드 수정 없음!)
public class CryptoPaymentProcessor extends PaymentProcessor {
    private String walletAddress;
    
    @Override
    protected void validate() {
        if (!isValidWalletAddress(walletAddress)) {
            throw new InvalidPaymentException("Invalid wallet");
        }
    }
    
    @Override
    protected void pay(double amount) {
        submitBlockchainTransaction(amount);
    }
    
    @Override
    protected void confirm() {
        verifyCryptoConfirmation();
    }
}
```

**Step 3: 주문 처리 클래스 (Factory Pattern 적용)**
```java
public class OrderProcessor {
    private PaymentProcessor paymentProcessor;
    
    // 의존성 주입 (Dependency Injection)
    public OrderProcessor(PaymentProcessor paymentProcessor) {
        this.paymentProcessor = paymentProcessor;
    }
    
    // 이제 구현 세부사항을 몰라도 됨!
    public void processOrder(Order order) {
        try {
            paymentProcessor.process(order.getAmount());
            order.markAsCompleted();
        } catch (InvalidPaymentException e) {
            order.markAsFailed(e.getMessage());
        }
    }
}

// Factory로 객체 생성 관리
public class PaymentProcessorFactory {
    public static PaymentProcessor create(PaymentMethod method, PaymentDetails details) {
        return switch(method) {
            case CREDIT_CARD -> new CreditCardProcessor(details.getCardNumber());
            case BANK_TRANSFER -> new BankTransferProcessor(details.getBankAccount());
            case MOBILE -> new MobilePaymentProcessor(details.getPhoneNumber());
            case CRYPTO -> new CryptoPaymentProcessor(details.getWalletAddress());
            default -> throw new IllegalArgumentException("Unknown payment method");
        };
    }
}

// 실제 사용
public class Main {
    public static void main(String[] args) {
        Order order = new Order(50000);
        
        // 각 결제 수단에 따라 동일한 인터페이스로 처리
        PaymentProcessor processor = PaymentProcessorFactory.create(
            PaymentMethod.CREDIT_CARD,
            new CardDetails("1234-5678-9012-3456")
        );
        
        OrderProcessor orderProcessor = new OrderProcessor(processor);
        orderProcessor.processOrder(order);
    }
}
```

## 핵심 개념 정리

### 1️⃣ **상속 (Inheritance)**
- **정의**: 부모 클래스의 속성과 메서드를 자식 클래스가 물려받음
- **역할**: 공통된 인터페이스(계약) 정의
- **예시**: `PaymentProcessor`는 모든 결제 수단의 표준을 정의

### 2️⃣ **다형성 (Polymorphism)**
- **정의**: 같은 메서드 호출이 객체의 타입에 따라 다르게 동작
- **종류**:
  - **메서드 오버라이딩**: 부모 메서드를 자식이 재구현
  - **메서드 오버로딩**: 같은 이름, 다른 파라미터
  
```java
// 다형성의 마법: 타입은 부모, 실제 동작은 자식
PaymentProcessor processor = new CreditCardProcessor(...);
processor.process(50000); // CreditCard의 pay() 호출됨

processor = new CryptoPaymentProcessor(...);
processor.process(50000); // Crypto의 pay() 호출됨
```

### 3️⃣ **SOLID 원칙 준수**

| 원칙 | 설명 | 이점 |
|------|------|------|
| **O** (Open-Closed) | 확장에는 열려있고, 수정에는 닫혀있음 | 새 결제수단 추가 시 기존 코드 수정 불필요 |
| **L** (Liskov Substitution) | 자식은 부모를 대체 가능 | 모든 결제 수단을 동일하게 처리 가능 |
| **D** (Dependency Inversion) | 구체적인 클래스가 아닌 추상화에 의존 | 느슨한 결합 구조 |

### 4️⃣ **if-else 체인 vs 다형성 비교**

```

│  if-else 체인                        │
├─────────────────────────────────────┤

├─────────────────────────────────────┤
│ ✅ 각 클래스는 자신의 책
