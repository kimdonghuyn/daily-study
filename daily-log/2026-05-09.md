## ❓ 질문 (3문제)

### 1. 📖 개념 설명

당신의 팀이 Spring Boot + JPA로 전자상거래 시스템을 운영하고 있습니다. 주문(Order), 상품(Product), 결제(Payment) 엔티티가 있고, 주문 생성 시나리오는 다음과 같습니다:

1. 주문 저장
2. 상품 재고 차감
3. 결제 처리 (외부 결제 게이트웨이 API 호출)
4. 포인트 적립

현재 코드는 OrderService.createOrder()에 @Transactional을 붙여서 전체 프로세스를 한 트랜잭션으로 묶고 있습니다. 그런데 결제 게이트웨이 API가 간헐적으로 30초 이상 응답이 없어서 DB 커넥션을 장시간 점유하고, 이것이 다른 조회 쿼리들을 블로킹하는 장애가 발생했습니다.

이 상황에서 다음을 고민해야 합니다:

**첫 번째 질문**: 결제 처리 로직을 별도 메서드(PaymentService.processPayment())로 분리하고 REQUIRES_NEW 전파 수준을 적용하려고 하는데, 이렇게 하면 "상품 재고 차감은 성공했지만 결제는 실패"하는 케이스를 어떻게 처리할 것인지 생각해봅시다. 즉, 부분 롤백이 가능해지는데, 이게 정말 원하는 동작일까요? 아니면 다른 전파 수준이나 아예 다른 패턴이 나을까요? 그리고 그 판단 기준은 무엇일까요?

**두 번째 질문**: 만약 결제 API 호출을 @Transactional 범위 밖으로 완전히 꺼내고, 결제 완료 후에 데이터베이스에 반영하는 비동기 콜백 방식으로 변경한다면, 결제는 성공했는데 콜백 처리 중 DB 저장에 실패하는 경우를 어떻게 보장할 것인가요? 이 경우 트랜잭션 전파 수준은 어떻게 설정해야 할까요? 그리고 이런 구조가 기존의 "한 트랜잭션으로 모두 묶기"와 비교했을 때 정말로 복잡도 대비 이득이 있을까요?

**세 번째 질문**: 포인트 적립 로직이 실패해도 주문 자체는 성공시켜야 하는 비즈니스 요구사항이 있다고 가정해봅시다. 현재는 포인트 적립이 실패하면 전체 주문도 롤백되고 있습니다. 이를 해결하기 위해 포인트 서비스를 REQUIRES_NEW로 호출하면, 정말로 안전할까요? 만약 포인트 적립 중 예외가 발생하고 롤백되어도, 주문은 살아있으니 나중에 배치로 재시도할 수 있다고 생각할 수 있는데, 실제로는 어떤 부작용이나 고려사항이 있을까요? 특히 동시에 같은 사용자가 여러 주문을 넣을 때는요?

**네 번째 질문**: 팀원이 "모든 트랜잭션을 PROPAGATION_REQUIRES_NEW로 설정하면 각 작업의 독립성이 보장되고 롤백이 세밀해져서 좋지 않을까?"라고 제안했습니다. 이 제안의 장점과 함정을 실무 관점에서 설명해주세요. 특히 데이터 일관성, 성능, 디버깅 측면에서 어떤 문제가 생길 수 있을까요?

**다섯 번째 질문**: 당신이 현재 시스템에서 겪고 있는 "재고 차감은 성공, 결제는 실패" 같은 부분 실패 시나리오를 처리하기 위해 saga 패턴이나 이벤트 기반 아키텍처를 도입해야 할까요? 아니면 더 단순한 방식(예: 보상 트랜잭션)이면 충분할까요? 어떤 기준으로 그 판단을 내려야 할까요?

## 모범 답안

### 첫 번째 질문: REQUIRES_NEW 적용과 부분 롤백 처리

**핵심 답변:**

REQUIRES_NEW를 단순히 적용하는 것만으로는 충분하지 않습니다. "재고 차감 성공, 결제 실패" 시나리오는 **비즈니스적 일관성을 깨뜨리기 때문** 입니다.

```java
// ❌ 위험한 패턴
@Transactional
public void createOrder(OrderRequest req) {
    Order order = orderRepository.save(new Order());  // TX1 commit
    productService.decrementStock(req.productId);     // TX1 commit
    
    try {
        paymentService.processPayment(req);            // TX2 (REQUIRES_NEW)
    } catch (PaymentException e) {
        // 여기서 뭘 할 건데? 재고는 이미 차감됨
        throw e;
    }
}

// ✅ 올바른 패턴 (Saga/Orchestrator)
@Transactional
public void createOrder(OrderRequest req) {
    Order order = orderRepository.save(new Order()); // 상태: PENDING
    productService.decrementStock(req.productId);
    
    try {
        paymentService.processPayment(req);
        order.confirm(); // 상태: CONFIRMED
    } catch (PaymentException e) {
        order.cancel();  // 상태: CANCELLED
        productService.restoreStock(req.productId);  // 보상 로직
        throw e;
    }
}
```

**판단 기준:**

| 시나리오 | 전파 수준 | 이유 |
|---------|---------|------|
| 결제 실패 시 주문까지 롤백 필요 | REQUIRED (기본값) | 원자성 보장 |
| 포인트 실패는 주문 유지, 나중 재시도 | REQUIRES_NEW + 보상 | 비즈니스 요구 |
| 외부 API(장시간) | @Async + Event-driven | 트랜잭션 범위 밖 |

### 두 번째 질문: 비동기 콜백과 이벤트 기반 아키텍처

**핵심 답변:**

외부 API 호출을 트랜잭션 밖으로 꺼내는 것은 좋은 결정이지만, **콜백 처리의 보장**이 핵심입니다.

```java
// ✅ 이벤트 기반 아키텍처 (권장)
@Transactional
public OrderResponse createOrder(OrderRequest req) {
    Order order = orderRepository.save(new Order()); 
    productService.decrementStock(req.productId);
    
    // DB 커밋 전에 이벤트 발행 (Spring 4.1+)
    applicationEventPublisher.publishEvent(
        new OrderCreatedEvent(order.getId(), req.paymentInfo)
    );
    return OrderResponse.from(order);
}

// 별도 리스너 (새 트랜잭션)
@Component
public class PaymentEventListener {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Transactional(propagation = REQUIRES_NEW)
    public void onOrderCreated(OrderCreatedEvent event) {
        try {
            paymentService.processPayment(event);  // 외부 API
            // 결과 저장 (새 TX)
            paymentResultRepository.save(...);
        } catch (Exception e) {
            // 실패했지만 주문은 유지, 나중 배치로 재시도
            paymentFailureRepository.save(...);
            log.error("Payment failed for order: {}", event.getOrderId());
        }
    }
}

// 배치: 실패한 결제 재시도
@Scheduled(fixedDelay = 60000)
@Transactional(propagation = REQUIRES_NEW)
public void retryFailedPayments() {
    paymentFailureRepository.findRetryable()
        .forEach(failure -> {
            try {
                paymentService.processPayment(failure);
                failure.markAsProcessed();
            } catch (Exception e) {
                failure.incrementRetryCount();
            }
        });
}
```

**결제 완료 후 DB 저장 실패 대응:**

```java
// 문제: 결제는 성공했는데 DB 저장 실패
// 해결: Idempotent Key + 멱등성 설계

@Entity
public class Payment {
    @Column(unique = true)
    private String idempotentKey;  // 결제 ID (외부 API에서 반환)
    
    private PaymentStatus status;
}

@Transactional(propagation = REQUIRES_NEW)
public void onPaymentCompleted(PaymentCallback callback) {
    // 멱등성: 같은 idempotentKey로 두 번 호출되어도 안전
    Payment payment = paymentRepository
        .findByIdempotentKey(callback.getPaymentId())
        .orElseGet(() -> Payment.builder()
            .idempotentKey(callback.getPaymentId())
            .status(COMPLETED)
            .build());
    
    paymentRepository.save(payment);
    Order order = orderRepository.findById(callback.getOrderId()).get();
    order.confirmPayment();
}
```

**이득 분석:**

```
기존 (한 트랜잭션)
장점: 단순, 일관성 보장
단점: DB 커넥션 장시간 점유, 외부 API timeout의 영향 큼

비동기 이벤트 기반
장점: DB 커넥션 빠르게 해제, 외부 API 독립적, 재시도 가능
단점: 최종 일관성(Eventual Consistency), 모니터링 필요

결론: 전자상거래 같은 I/O 집약적 시스템에서는 비동기가 필수
```
