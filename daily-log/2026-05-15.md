## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# GC의 필요성과 동작 원리

## 모범 답안

### 1️⃣ "누가" 메모리를 정리하는가?

**자동 메모리 관리 시스템(Garbage Collector, GC)**이 이 역할을 합니다.

- 개발자가 일일이 메모리 해제를 명령할 필요 없이, JVM이나 런타임이 자동으로 불필요한 메모리를 감지하고 정리합니다.
- 마치 카페의 "청소 담당자"처럼 자동으로 사용이 끝난 영수증을 치워주는 것과 같습니다.

### 2️⃣ "무엇을" 쓸모없다고 판단하는가?

**도달 가능성(Reachability) 분석 알고리즘**으로 판단합니다:

```
📍 GC Root에서 도달 가능한 객체 → 살아있는 객체 (Live Object)
❌ GC Root에서 도달 불가능한 객체 → 죽은 객체 (Garbage)
```

**GC Root란?**
- 활성 스레드의 스택 변수
- 정적(static) 변수
- JNI 참조

### 3️⃣ 구체적인 동작 원리 (Mark-Sweep-Compact)

```java
// 실제 상황 예시
public class CafeManager {
    public void processOrder() {
        Receipt receipt = new Receipt();  // ① 메모리 할당
        receipt.print();
        // receipt 변수가 스코프를 벗어남
        // ② GC가 이 객체를 감지하고 정리
    }
}

// 위 메서드 종료 후:
// - receipt 참조는 사라짐
// - GC는 "아, 이 Receipt 객체는 더 이상 누구도 참조하지 않네"라고 판단
// - GC Mark 단계에서 표시 안 함
// - Sweep 단계에서 메모리 회수
```

## 핵심 개념 정리

### 📌 메모리 누수 vs GC

| 항목 | C/C++ (수동 관리) | Java/Python (자동 GC) |
|------|------------------|----------------------|
| 메모리 해제 | 개발자 책임 | GC 책임 |
| 메모리 누수 위험 | 매우 높음 | 참조 순환만 위험 |
| 성능 오버헤드 | 낮음 | GC Pause Time 발생 |

### 🔄 GC 동작 단계

1. **Mark (표시)**
   ```
   GC Root → 도달 가능한 모든 객체에 표시
   ```

2. **Sweep (청소)**
   ```
   표시되지 않은 객체 메모리 회수
   ```

3. **Compact (압축)** - Optional
   ```
   메모리 단편화 해소 (메모리를 연속으로 정렬)
   ```

### 🎯 GC의 필요성

| 필요성 | 설명 |
|--------|------|
| **자동화** | 개발자가 메모리 관리에 집중할 필요 없음 |
| **안정성** | Use-After-Free, Double-Free 버그 방지 |
| **생산성** | 비즈니스 로직에만 집중 가능 |
| **메모리 효율** | 자동으로 단편화된 메모리 정리 |

### 💡 핵심 판단 기준: 도달 가능성(Reachability)

```java
// ✅ 살아있는 객체
User user = new User();  // user 변수가 참조 유지
service.processUser(user);

// ❌ 죽은 객체 (GC 대상)
User tempUser = new User();
tempUser = null;  // 명시적 해제
// 또는 스코프 벗어남
```

## 실무 포인트

### 🚀 1. GC Tuning이 필요한 상황

```java
// ❌ 문제 상황: Full GC 발생으로 서비스 중단
// JVM 메모리가 부족할 때마다 Full GC 발생
// → 수백 ms 이상 모든 요청 정지

// ✅ 해결 방법
// 1) Heap 크기 조정
java -Xms4G -Xmx8G Application

// 2) GC 알고리즘 선택 (G1GC, ZGC 등)
java -XX:+UseG1GC Application
```

### 🚀 2. 메모리 누수 방지 패턴

```java
// ❌ 나쁜 예: 참조 순환
public class OrderListener {
    private Order order;  // 순환 참조 가능
    
    public OrderListener(Order order) {
        this.order = order;
        order.addListener(this);  // 양방향 참조
    }
}

// ✅ 좋은 예: WeakReference 사용
public class OrderListener {
    private WeakReference<Order> order;  // 약한 참조
    
    public OrderListener(Order order) {
        this.order = new WeakReference<>(order);
    }
}
```

### 🚀 3. GC 모니터링

```bash
# JVM GC 로그 확인
java -XX:+PrintGCDetails -XX:+PrintGCTimeStamps Application

# 출력 예시:
# 2024-01-15T10:30:45.123+0900: [GC (Allocation Failure) 
# [PSYoungGen: 256M->32M(256M)] 512M->280M(1024M), 0.0234567 secs]

# ↑ Young Generation에서 256M → 32M으로 회수됨
# ↑ 전체 Heap: 512M → 280M
# ↑ GC 소요 시간: 23.4ms
```

### 🚀 4. 성능 최적화 체크리스트

```java
public class PerformanceChecklist {
    
    // 1. 불필요한 객체 생성 최소화
    // ❌ 나쁜 예
    String result = "";
    for (int i = 0; i < 10000; i++) {
        result += "item" + i;  // 매번 새 String 객체 생성
    }
    
    // ✅ 좋은 예
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < 10000; i++) {
        sb.append("item").append(i);
    }
    String result = sb.toString();
    
    // 2. 콜렉션 초기 크기 지정
    List<User> users = new ArrayList<>(10000);  // 확장 비용 절감
    
    // 3. 대용량 데이터 처리 시 Stream 활용
    users.stream()
        .filter(u -> u.isActive())
        .forEach(this::processUser);  // 지연 평가로 메모리 절감
}
```
