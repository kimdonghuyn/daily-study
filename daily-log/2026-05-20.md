## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 캡슐화와 접근 제어자 - 모범 답안

## 모범 답안

이 문제는 **캡슐화(Encapsulation)**와 **접근 제어자(Access Modifier)**를 활용하여 해결합니다.

### 문제 해결 방식

```java
public class BankAccount {
    // 1. private으로 직접 접근 차단
    private double balance;
    
    // 2. 생성자를 통한 초기화
    public BankAccount(double initialBalance) {
        if (initialBalance < 0) {
            throw new IllegalArgumentException("초기 잔액은 음수일 수 없습니다");
        }
        this.balance = initialBalance;
    }
    
    // 3. 읽기 전용 getter (검증된 조회)
    public double getBalance() {
        return balance;
    }
    
    // 4. 검증 로직이 포함된 setter (비즈니스 규칙 적용)
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("입금액은 0보다 커야 합니다");
        }
        this.balance += amount;
        logTransaction("입금", amount);
    }
    
    public boolean withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("출금액은 0보다 커야 합니다");
        }
        if (amount > balance) {
            throw new InsufficientFundsException("잔액이 부족합니다");
        }
        this.balance -= amount;
        logTransaction("출금", amount);
        return true;
    }
    
    private void logTransaction(String type, double amount) {
        // 거래 기록 (감시 추적)
        System.out.println(String.format("[%s] %s: %.2f원", 
            LocalDateTime.now(), type, amount));
    }
}

// ✅ 올바른 사용
BankAccount account = new BankAccount(10000);
account.deposit(5000);      // OK: 검증을 거침
account.withdraw(3000);     // OK: 검증을 거침
System.out.println(account.getBalance()); // 12000

// ❌ 불가능한 사용 (컴파일 에러)
// account.balance = -1000;  // 컴파일 에러!
// account.balance += 999999; // 컴파일 에러!
```

### 왜 이것이 해결책인가?

| 문제점 | 해결책 | 효과 |
|-------|--------|------|
| 직접 접근 가능 | `private balance` | 외부에서 직접 변경 불가 |
| 검증 없음 | `deposit()`, `withdraw()` 메서드 | 비즈니스 규칙 자동 적용 |
| 추적 불가 | `logTransaction()` | 모든 거래 감시 |
| 의도 모호 | 명시적 메서드명 | 코드의 의도 명확 |

## 핵심 개념 정리

### 1. **캡슐화 (Encapsulation)**
- 관련된 데이터와 메서드를 하나의 단위로 묶고, 내부 구현을 숨기는 것
- 외부에서는 인터페이스(공개 메서드)를 통해서만 접근
- **SOLID의 S(Single Responsibility)와 인터페이스 분리 원칙**의 기초

### 2. **접근 제어자 (Access Modifiers)**

```java
private      // 같은 클래스 내에서만 접근 가능
protected    // 같은 패키지 + 상속 클래스에서 접근 가능
public       // 모든 곳에서 접근 가능
// (default) // 같은 패키지 내에서만 접근 가능
```

### 3. **Getter/Setter의 올바른 사용**

```java
// ❌ 잘못된 예: setter가 아무것도 하지 않음
public void setBalance(double balance) {
    this.balance = balance; // 위험!
}

// ✅ 올바른 예: 검증과 부작용을 함께 처리
public void setBalance(double balance) {
    if (balance < 0) {
        throw new IllegalArgumentException("음수 잔액 불가");
    }
    double change = balance - this.balance;
    this.balance = balance;
    notifyAudit(change); // 감시 시스템에 알림
}

// ✅ 더 나은 예: 행동 중심으로 메서드 설계
// setBalance() 대신 deposit(), withdraw() 사용
```

### 4. **불변성 (Immutability) 고려**

```java
// 민감한 정보는 수정 불가하게
public final class BankAccount {
    private final String accountNumber; // 계좌번호 변경 불가
    private double balance;              // 잔액만 메서드로 변경
}
```

### 5. **Validation의 위치**

```
계층별 검증 전략
┌─────────────────────────────────┐
│ 1. 입력 검증 (Controller)       │  ← 형식, 범위
├─────────────────────────────────┤
│ 2. 비즈니스 검증 (Service)      │  ← 규칙, 권한
├─────────────────────────────────┤
│ 3. 제약 검증 (Domain/Model)     │  ← 불변식, 상태
└─────────────────────────────────┘
```

### 2. **동시성 문제 고려**

```java
// 멀티스레드 환경에서는 synchronized 추가
public synchronized void deposit(double amount) {
    if (amount <= 0) throw new IllegalArgumentException();
    this.balance += amount;
}

// 또는 더 현대적으로는 Lock 사용
private final Lock lock = new ReentrantLock();

public void deposit(double amount) {
    lock.lock();
    try {
        if (amount <= 0) throw new IllegalArgumentException();
        this.balance += amount;
    } finally {
        lock.unlock();
    }
}
```

### 3. **감시와 로깅의 중요성**

```java
public void withdraw(double amount) {
    double beforeBalance = this.balance;
    
    if (amount <= 0) {
        logSuspiciousActivity("Invalid withdrawal: " + amount);
        throw new IllegalArgumentException();
    }
    
    if (amount > balance) {
        logSuspiciousActivity("Insufficient funds attempt: " + amount);
        throw new InsufficientFundsException();
    }
    
    this.balance -= amount;
    
    // 모든 거래 기록 (감시, 감사, 규제 대응)
    auditLog.record(new TransactionRecord(
        TransactionType.WITHDRAWAL,
        amount,
        beforeBalance,
        this.balance,
        LocalDateTime.now()
    ));
}
```

### 4. **금융 시스템의 특수성**

```java
// 정밀도 문제: double 대신 BigDecimal 사용
public class BankAccount {
    private BigDecimal balance; // ✅ 금융에서는 필수
    
    public void deposit(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException();
        }
        this.balance = this.balance.add(amount);
    }
}
```
