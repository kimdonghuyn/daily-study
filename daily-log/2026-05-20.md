### 1. 📖 개념 설명

# 캡슐화와 접근 제어자

## 모범 답안

캡슐화(Encapsulation)는 **데이터와 메서드를 하나의 단위로 묶고, 접근 제어자를 통해 외부 접근을 제한하는 객체 지향의 핵심 원칙**입니다.

### 구체적인 제어 방식

**1. 접근 제어자를 이용한 계층적 접근 관리**

```java
public class BankAccount {
    // private: 클래스 내부에서만 접근 가능 (가장 제한적)
    private double balance;
    private String accountNumber;
    
    // protected: 같은 패키지 + 상속 클래스에서 접근
    protected String accountType;
    
    // package-private(default): 같은 패키지 내에서만 접근
    void internalAudit() { }
    
    // public: 모든 곳에서 접근 가능 (가장 개방적)
    public double getBalance() {
        return balance;
    }
}
```

**2. Getter/Setter를 통한 검증 로직 추가**

```java
public class BankAccount {
    private double balance;
    
    // 읽기만 허용하고 검증 로직을 거침
    public double getBalance() {
        return balance;
    }
    
    // 쓰기에 비즈니스 로직 추가
    public void withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("출금액은 0보다 커야 합니다");
        }
        if (amount > balance) {
            throw new IllegalArgumentException("잔액이 부족합니다");
        }
        balance -= amount;
        logTransaction("WITHDRAW", amount);
    }
    
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("입금액은 0보다 커야 합니다");
        }
        balance += amount;
        logTransaction("DEPOSIT", amount);
    }
    
    private void logTransaction(String type, double amount) {
        // 외부에서 호출 불가능한 내부 로직
        System.out.println(String.format("[%s] %s: %.2f", 
            LocalDateTime.now(), type, amount));
    }
}
```

**3. 불변성(Immutability) 구현**

```java
public class Account {
    private final String accountNumber;  // final로 변경 불가
    private final String holderName;
    private double balance;
    
    public Account(String accountNumber, String holderName, double initialBalance) {
        this.accountNumber = accountNumber;
        this.holderName = holderName;
        this.balance = initialBalance;
    }
    
    // 읽기만 가능
    public String getAccountNumber() {
        return accountNumber;
    }
    
    public String getHolderName() {
        return holderName;
    }
    
    // 변경은 새 객체 반환 (복사 방지)
    public Account transfer(double amount) {
        if (amount > balance) {
            throw new IllegalArgumentException("잔액 부족");
        }
        this.balance -= amount;
        return this;
    }
}
```

## 핵심 개념 정리

### 📌 접근 제어자의 4단계

| 제어자 | 같은 클래스 | 같은 패키지 | 상속 클래스 | 외부 |
|--------|-----------|-----------|----------|-----|
| **private** | ✅ | ❌ | ❌ | ❌ |
| **package-private** | ✅ | ✅ | ❌ | ❌ |
| **protected** | ✅ | ✅ | ✅ | ❌ |
| **public** | ✅ | ✅ | ✅ | ✅ |

### 🔒 캡슐화의 주요 이점

- **정보 은닉**: 내부 구현을 숨겨 외부의 잘못된 사용 방지
- **검증 로직**: 데이터 변경 시 무결성 검증 가능
- **유지보수성**: 내부 구현 변경해도 공개 인터페이스 유지
- **단일 책임**: 각 클래스가 명확한 역할 수행

### 💡 설계 원칙 (SOLID 중 일부)

- **최소 권한 원칙(Principle of Least Privilege)**: 필요한 최소한의 접근만 허용
- **인터페이스 분리**: 꼭 필요한 메서드만 공개

### ✅ 백엔드 개발에서의 실제 적용

**1. REST API 응답 객체 설계**

```java
@RestController
@RequestMapping("/api/accounts")
public class AccountController {
    
    @GetMapping("/{id}")
    public AccountResponseDto getAccount(@PathVariable Long id) {
        Account account = accountService.findById(id);
        // DTO를 통해 필요한 정보만 노출
        return AccountResponseDto.from(account);
    }
}

// DTO: 공개할 정보만 선택적으로 노출
public class AccountResponseDto {
    private String accountNumber;
    private String holderName;
    private double balance;
    
    // 민감 정보는 DTO에 포함 안 함
    // 예: pinNumber, securityCode 등
    
    public static AccountResponseDto from(Account account) {
        return new AccountResponseDto(
            account.getAccountNumber(),
            account.getHolderName(),
            account.getBalance()
        );
    }
}
```

**2. 데이터베이스 접근 계층 보호**

```java
@Service
public class AccountService {
    private final AccountRepository repository;
    
    // public: 비즈니스 로직을 통한 안전한 접근만 제공
    public void transferMoney(Long fromId, Long toId, double amount) {
        Account fromAccount = repository.findById(fromId)
            .orElseThrow(() -> new AccountNotFoundException());
        Account toAccount = repository.findById(toId)
            .orElseThrow(() -> new AccountNotFoundException());
        
        // 트랜잭션, 검증, 감사로그 등 모든 제어
        fromAccount.withdraw(amount);
        toAccount.deposit(amount);
        repository.saveAll(List.of(fromAccount, toAccount));
    }
    
    // private: 내부 헬퍼 메서드는 외부 접근 불가
    private void validateAccountOwnership(Account account, User user) {
        if (!account.getOwnerId().equals(user.getId())) {
            throw new UnauthorizedException("계정 소유자가 아닙니다");
        }
    }
}
```

**3. 설정값 보호**

```java
@Configuration
public class DatabaseConfig {
    
    // private: 민감한 설정값
    private final String dbPassword;
    private final String apiKey;
    
    // 생성자 주입으로 final 보장
    public DatabaseConfig(
        @Value("${db.password}") String dbPassword,
        @Value("${api.key}") String apiKey
    ) {
        this.dbPassword = dbPassword;
        this.apiKey = apiKey;
    }
    
    // public: 필요한 기능만 메서드로 제공
    public DataSource getDataSource() {
        return createDataSource(dbPassword);
    }
    
    // 직접 접근 불가능
    // dbPassword를 직접 노출하지 않음
}
```

### ⚠️ 주의사항

- **과도한 캡슐화**: getter/setter 남용은 캡슐화를 약화시킴
- **성능 고려**: 접근 제어는 컴파일 타임에 결정되므로 성능 영향 미미
- **테스트 시 주의**: private 메서드는 public 인터페이스를 통해 간접 테스트
