### 1. 📖 개념 설명

# SOLID — SRP와 OCP 모범 답안

## 모범 답안

### SRP(Single Responsibility Principle)가 필요한 이유

당신의 카페 예시는 **책임 분산의 필요성**을 완벽하게 보여줍니다.

한 사람이 모든 업무를 하면:
- **변경의 영향 범위가 넓다**: 주문 시스템을 바꾸면 → 커피 제조 로직도 함께 수정되어야 함
- **테스트가 어렵다**: 계산 로직만 테스트하려면 주문받기, 커피 만들기 까지 모두 검증해야 함
- **확장성이 떨어진다**: 신입을 교육할 때 모든 분야를 가르쳐야 함

**SRP 적용**: 역할을 분리합니다.
```
카운터 직원 → 주문 받기 (Order Service)
바리스타 → 커피 만들기 (CoffeeService)
계산대 직원 → 결제 처리 (PaymentService)
```

---

### OCP(Open/Closed Principle)와의 연결

SRP로 역할을 나누면 **자연스럽게 OCP를 만족**하게 됩니다.

```
상황: "새로운 결제 수단 추가 (카카오페이, 애플페이)"

❌ SRP 미적용 (한 사람이 모든 업무)
→ 그 사람의 모든 업무 프로세스 수정 필요
→ 기존 로직 변경(Closed 위반)

✅ SRP 적용 (PaymentService 분리)
→ PaymentService만 확장
→ 다른 서비스는 수정 없음 (Closed to modification)
→ 새 결제 방식 추가 (Open to extension)
```

---

## 핵심 개념 정리

### SRP (Single Responsibility Principle)
- **정의**: 클래스는 **하나의 책임만** 가져야 하며, 그 책임을 **완전히 캡슐화**해야 함
- **변경의 이유가 하나여야 함**: "왜 이 클래스를 수정하는가?"라는 질문에 하나의 답만 나와야 함
- **응집도 ↑, 결합도 ↓**: 관련된 것끼리 묶이고, 불필요한 의존성 제거

### OCP (Open/Closed Principle)
- **정의**: 소프트웨어는 **확장에는 열려있고, 수정에는 닫혀있어야 함**
- **SRP의 선행 조건**: 책임이 분리되어야 확장 포인트를 명확하게 설계 가능
- **추상화 활용**: 인터페이스를 통해 새 구현을 추가하되, 기존 코드는 건드리지 않음

### 둘의 관계
```
SRP (책임 분리)
    ↓
변경 영향 범위 최소화
    ↓
확장 포인트 명확화
    ↓
OCP (확장/수정 분리) 달성
```

## 실무 포인트

### 1️⃣ 코드 예시로 보는 SRP 위반 → 적용

#### ❌ SRP 위반
```python
class CafeWorker:
    def take_order(self, customer):
        # 주문 받기
        pass
    
    def make_coffee(self, order):
        # 커피 만들기
        pass
    
    def process_payment(self, amount):
        # 결제 처리
        pass

# 문제: 결제 시스템 변경 시 CafeWorker 수정 필요
```

#### ✅ SRP 적용
```python
class OrderService:
    def take_order(self, customer) -> Order:
        return Order(customer)

class CoffeeService:
    def make_coffee(self, order) -> Coffee:
        return Coffee(order.type)

class PaymentService:
    def process_payment(self, amount) -> Payment:
        return Payment(amount)

class Cafe:
    def __init__(self):
        self.order_svc = OrderService()
        self.coffee_svc = CoffeeService()
        self.payment_svc = PaymentService()
    
    def serve_customer(self, customer):
        order = self.order_svc.take_order(customer)
        coffee = self.coffee_svc.make_coffee(order)
        self.payment_svc.process_payment(order.price)
```

### 2️⃣ OCP 적용: 새로운 결제 수단 추가

#### ❌ OCP 위반 (수정이 필요함)
```python
class PaymentService:
    def process_payment(self, amount, method):
        if method == "card":
            self.charge_card(amount)
        elif method == "cash":
            self.charge_cash(amount)
        elif method == "kakao":  # ← 매번 수정 필요!
            self.charge_kakao(amount)
```

#### ✅ OCP 적용 (확장만 함)
```python
from abc import ABC, abstractmethod

class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount: int) -> bool:
        pass

class CardPayment(PaymentMethod):
    def pay(self, amount: int) -> bool:
        print(f"카드로 {amount}원 결제")
        return True

class CashPayment(PaymentMethod):
    def pay(self, amount: int) -> bool:
        print(f"현금 {amount}원 결제")
        return True

class KakaoPayment(PaymentMethod):  # ← 기존 코드 수정 없음!
    def pay(self, amount: int) -> bool:
        print(f"카카오페이로 {amount}원 결제")
        return True

class PaymentService:
    def process_payment(self, amount: int, method: PaymentMethod) -> bool:
        return method.pay(amount)  # ← 변경 없음!

# 사용
payment_svc = PaymentService()
payment_svc.process_payment(10000, CardPayment())
payment_svc.process_payment(10000, KakaoPayment())  # 새 결제 수단 추가 완료
```

### 3️⃣ 실제 시스템 설계 체크리스트

| 항목 | 질문 | 확인 |
|------|------|------|
| **SRP** | "이 클래스를 수정할 이유가 몇 개인가?" | 1개여야 함 |
| **SRP** | "이 메서드들이 정말 관련이 있는가?" | 높은 응집도 |
| **OCP** | "새 요구사항이 기존 코드 수정을 강제하는가?" | 인터페이스로 해결 |
| **OCP** | "확장 포인트가 명확한가?" | 추상화 레이어 확인 |

### 4️⃣ 흔한 실수와 해결책

| 실수 | 원인 | 해결책 |
|------|------|--------|
| 클래스가 너무 커짐 | 책임이 섞여있음 | SRP로 분리 |
| 작은 변경이 전체 영향 | 강한 결합도 | 인터페이스 도입 |
| 확장할 때마다 기존 코드 수정 | OCP 미적용 | 추상화와 상속 활용 |
| 테스트가 어려움 | 의존성 많음 | 의존성 주입(DI) 사용 |

## 💡 시니어 관점 조언

**"SRP와 OCP는 순차적 관계입니다"**

1. **SRP 먼저**: 책임을 명확히 분리하지 않으면, OCP를 적용할 확장 포인트 자체를 찾을 수 없습니다.

2. **과도한 분리 주의**: SRP를 과도하게 적용하면 오히려 관리할 클래스가 늘어납니다. **변경 빈도**를 기준으로 판단하세요.

3. **문제가 생기면 리팩토링**: 초기부터 완벽한 설계를 기대하지 마세요. 변경 요청이 들어올 때 SRP/OCP 위반을 감지하고 개선하는 것이 실무 방식입니다.
