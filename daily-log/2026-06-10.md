## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# Spring AOP를 통한 횡단 관심사 분리

## 모범 답안

이 문제는 **Spring AOP(Aspect Oriented Programming)**로 해결합니다.

**문제의 본질**: 로깅은 비즈니스 로직과 무관하지만, 모든 메서드에 반복적으로 적용해야 하는 **횡단 관심사(Cross-Cutting Concern)**입니다.

**해결 방법**:
1. **로깅 기능을 별도의 Aspect로 분리** → 비즈니스 코드는 순수하게 유지
2. **@Pointcut으로 대상 메서드 자동 선택** → 패턴 매칭으로 일괄 적용
3. **@Before, @After 등으로 실행 시점 제어** → 메서드 실행 전후에 자동 삽입
4. **로그 형식 변경 시 한 곳만 수정** → 모든 메서드에 자동 반영

## 핵심 개념 정리

### 🎯 AOP의 5가지 핵심 용어

| 용어 | 설명 | 예시 |
|------|------|------|
| **Aspect** | 횡단 관심사를 모듈화한 클래스 | `@Aspect` 붙은 LoggingAspect |
| **JoinPoint** | AOP가 적용될 수 있는 지점 | 메서드 실행 시점 |
| **Pointcut** | JoinPoint를 선별하는 표현식 | `execution(* order.*(..))` |
| **Advice** | JoinPoint에서 실행할 로직 | @Before, @After, @Around 등 |
| **Weaving** | Aspect를 코드에 적용하는 과정 | 스프링이 런타임에 프록시로 적용 |

### 📍 Advice의 종류 (실행 시점별)

```
@Before       : 메서드 실행 '전' (예: 입력값 검증)
@After        : 메서드 실행 '후' (무조건 실행)
@AfterReturning : 정상 반환 후 (예: 결과 로깅)
@AfterThrowing : 예외 발생 후 (예: 에러 로깅)
@Around       : 메서드 전/후 + 실행 제어 (가장 강력함)
```

### 🔍 Pointcut 표현식 문법

```
execution(
  [접근제어자] 
  반환타입 
  [패키지명.]클래스명.메서드명(매개변수)
  [throws 예외]
)

예시:
execution(public void com.shop.order.OrderService.*(..))
  → OrderService의 모든 public void 메서드

execution(* com.shop..*Service.process*(..))
  → com.shop 하위의 모든 *Service 클래스의 process*로 시작하는 메서드
```

## 실무 포인트

### ✅ 구현 예시 (실제 적용 코드)

```java
// 1️⃣ Aspect 클래스 정의
@Aspect
@Component
@Slf4j
public class AuditLoggingAspect {
    
    // 2️⃣ Pointcut 정의 - 모든 비즈니스 메서드를 자동 감지
    @Pointcut("execution(* com.shop.order..*(..)) || " +
              "execution(* com.shop.payment..*(..)) || " +
              "execution(* com.shop.refund..*(..))")
    public void auditMethods() {}
    
    // 3️⃣ Around Advice - 가장 유연한 방식
    @Around("auditMethods()")
    public Object logAudit(ProceedingJoinPoint pjp) throws Throwable {
        String methodName = pjp.getSignature().getName();
        String className = pjp.getTarget().getClass().getSimpleName();
        String userId = getCurrentUserId(); // 현재 사용자
        long startTime = System.currentTimeMillis();
        
        try {
            // 4️⃣ 실제 메서드 실행
            Object result = pjp.proceed();
            
            // 5️⃣ 성공 로그 기록
            long duration = System.currentTimeMillis() - startTime;
            log.info("✅ [AUDIT] User: {}, Class: {}, Method: {}, " +
                    "Duration: {}ms, Status: SUCCESS",
                    userId, className, methodName, duration);
            
            return result;
            
        } catch (Exception e) {
            // 6️⃣ 실패 로그 기록
            log.error("❌ [AUDIT] User: {}, Class: {}, Method: {}, " +
                     "Error: {}",
                     userId, className, methodName, e.getMessage(), e);
            throw e;
        }
    }
    
    private String getCurrentUserId() {
        return SecurityContextHolder.getContext()
                .getAuthentication()
                .getName();
    }
}

// 2️⃣ 비즈니스 로직 - 순수하게 유지됨! (로깅 코드 없음)
@Service
public class OrderService {
    public void processOrder(Order order) {
        // 실제 주문 처리 로직만 집중
        order.setStatus("PROCESSING");
        // ... 데이터베이스 저장 등
        // 위 Aspect가 자동으로 로깅을 담당함
    }
    
    public void refund(String orderId) {
        // 환불 로직만 집중
        // 마찬가지로 자동 로깅 적용됨
    }
}
```

### 🚀 실무 적용 팁

#### 1) **@Pointcut 재사용으로 유지보수성 향상**
```java
@Pointcut("execution(* com.shop..*Service.*(..))")
public void allBusinessMethods() {}

@Pointcut("allBusinessMethods() && !execution(* *.get*(..))")
public void writeMethods() {} // 조회 제외 (쓰기만)

@Before("writeMethods()")
public void validateBeforeWrite() { ... }
```

#### 2) **어노테이션 기반 Pointcut (더 명시적)**
```java
// 커스텀 어노테이션 정의
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Auditable {
    String value() default "USER_ACTION";
}

// Aspect에서 어노테이션으로 선별
@Around("@annotation(auditable)")
public Object logAnnotated(ProceedingJoinPoint pjp, 
                          Auditable auditable) throws Throwable {
    // auditable.value() 로 로그 레벨 조정 가능
    return pjp.proceed();
}

// 실제 사용
@Service
public class OrderService {
    @Auditable("ORDER_PROCESSING")
    public void processOrder(Order order) { ... }
}
```

#### 3) **성능 고려사항 - @Around는 주의**
```java
// ❌ 나쁜 예: 모든 메서드에 @Around (프록시 오버헤드)
@Around("execution(* com.shop..*(..))")
public Object heavyAspect(ProceedingJoinPoint pjp) { ... }

// ✅ 좋은 예: 필요한 범위만 선정
@Around("execution(public * com.shop..*Service.*(..)) && " +
        "!execution(* *.get*(..))")
public Object smartAspect(ProceedingJoinPoint pjp) { ... }
```

#### 4) **로그 형식 일괄 변경 (이것이 AOP의 진정한 가치)**
```java
// 기존: 모든 메서드 로직에서 로그 형식을 일일이 수정
// AOP: 한 곳만 수정하면 모든 메서드에 반영됨

// Before (AOP 없을 때)
log.info("User: {}, Action: {}", userId, action); // 방식 1
log.debug("{}|{}|{}", userId, action, time);      // 방식 2 (불일관)
// ... 50개 메서드에 흩어져 있음

// After (AOP 적용)
// Aspect 1곳만 수정하면 완료!
```

```java
@Around("auditMethods()")
public Object logDetails(ProceedingJoinPoint pjp) throws Throwable {
    MethodSignature sig = (MethodSignature) pjp.getSignature();
    String[] paramNames = sig.getParameterNames();
