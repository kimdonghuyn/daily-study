## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 상속과 다형성 - 게임 캐릭터 시스템

## 모범 답안

### 문제점 분석
현재 코드는 **Open-Closed Principle 위반**입니다.
- 새 직업 추가 시마다 기존 코드를 수정해야 함 (닫혀있지 않음)
- 각 직업의 로직이 흩어져 있어 유지보수 어려움
- 조건문 지옥(if-else hell)으로 복잡도 증가

### 해결책: 상속과 다형성 적용

```python
# ✅ 상속과 다형성을 활용한 개선된 설계

from abc import ABC, abstractmethod

# 1️⃣ 부모 클래스(추상 클래스): 공통 인터페이스 정의
class Character(ABC):
    def __init__(self, name, hp):
        self.name = name
        self.hp = hp
    
    @abstractmethod
    def attack(self):
        """각 직업별로 구현하도록 강제"""
        pass
    
    def take_damage(self, damage):
        """공통 기능"""
        self.hp -= damage
        print(f"{self.name}이 {damage}의 피해를 입었습니다!")

# 2️⃣ 자식 클래스들: 각각 attack() 다형성 구현
class Warrior(Character):
    def attack(self):
        return f"{self.name}이 검으로 강력하게 베었습니다! (공격력: 25)"

class Wizard(Character):
    def attack(self):
        return f"{self.name}이 마법을 시전했습니다! (공격력: 30)"

class Archer(Character):
    def attack(self):
        return f"{self.name}이 화살을 날렸습니다! (공격력: 20)"

# 3️⃣ 새로운 직업 추가 (기존 코드 수정 없음!)
class Paladin(Character):
    def attack(self):
        return f"{self.name}이 신성한 검격을 날렸습니다! (공격력: 28)"

# ✅ 사용: 조건문 없이 깔끔함!
def execute_battle(characters):
    """기존 코드 수정 없이 새로운 캐릭터도 자동으로 처리"""
    for char in characters:
        print(char.attack())  # 다형성: 같은 메서드가 다르게 동작

# 실행
party = [
    Warrior("아서"),
    Wizard("머린"),
    Archer("로빈"),
    Paladin("랜슬롯")  # ⭐ 새 직업 추가, 기존 코드 수정 없음!
]

execute_battle(party)
```

**출력:**
```
아서이 검으로 강력하게 베었습니다! (공격력: 25)
머린이 마법을 시전했습니다! (공격력: 30)
로빈이 화살을 날렸습니다! (공격력: 20)
랜슬롯이 신성한 검격을 날렸습니다! (공격력: 28)
```

## 핵심 개념 정리

### 📌 상속 (Inheritance)
```
Character (부모/기본 클래스)
    ↓
    ├─ Warrior (자식)
    ├─ Wizard (자식)
    ├─ Archer (자식)
    └─ Paladin (자식)
```
- **목적**: 공통된 속성/메서드를 한 곳에 정의
- **이점**: 중복 코드 제거, 유지보수 용이

### 🔀 다형성 (Polymorphism)
```python
# 같은 인터페이스, 다른 구현
characters = [warrior, wizard, archer, paladin]

for char in characters:
    char.attack()  # 각각 다르게 동작!
```
- **런타임에** 객체의 실제 타입에 따라 메서드 결정
- **조건문 없이** 타입별 행동 처리 가능

### 🎯 추상 클래스 (Abstract Class)
```python
from abc import ABC, abstractmethod

class Character(ABC):
    @abstractmethod
    def attack(self):
        pass  # "반드시 구현해야 함"을 강제
```
- 자식 클래스가 반드시 구현해야 할 메서드 정의
- 인터페이스 계약 역할

## 실무 포인트

### ✅ 이전 방식 (❌ 나쁜 예)
```python
# if-else 지옥
def perform_attack(character_type):
    if character_type == "warrior":
        print("검으로 베었습니다!")
    elif character_type == "wizard":
        print("마법을 시전했습니다!")
    elif character_type == "archer":
        print("화살을 날렸습니다!")
    elif character_type == "paladin":  # ← 새로 추가할 때마다 수정 필요!
        print("신성한 검격을 날렸습니다!")
    # 매번 이 함수를 건드려야 함 (버그 위험 ⚠️)
```

### ✅ 개선된 방식 (⭐ 좋은 예)
```python
def perform_attack(character):
    print(character.attack())  # 한 줄! 새 직업 추가해도 변경 없음

# SOLID 원칙 준수:
# - Open-Closed Principle: 확장에 열려있고, 수정에 닫혀있음
# - Liskov Substitution: 어떤 Character든 대체 가능
```

### 🔧 실제 적용 시나리오

| 상황 | 해결 |
|------|------|
| **새 직업 추가** | Character 상속 클래스만 만들면 됨 |
| **공통 기능 추가** | Character 부모 클래스만 수정 |
| **직업별 AI 구현** | 각 클래스에서 `think()` 메서드 오버라이딩 |
| **아이템 시스템** | `apply_buff(character)` → 모든 직업 자동 지원 |

### 📊 복잡도 비교

```
❌ 조건문 방식:
  새 직업 추가마다 O(n) 시간에 코드 수정 필요

✅ 다형성 방식:
  새 클래스 추가만으로 자동 처리 (O(1))
  - 확장성: ⭐⭐⭐⭐⭐
  - 유지보수: ⭐⭐⭐⭐⭐
```
