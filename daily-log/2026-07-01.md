## 📚 모범 답안 & 개념 정리

### 1. 🎤 면접 질문

# 캡슐화와 접근 제어자 면접 완벽 대답

## 모범 답안

캡슐화와 접근 제어자가 필요한 가장 근본적인 이유는 **"데이터 무결성 보호"**와 **"변경 영향도 최소화"**입니다.

### 핵심 답변 구조

**1단계: 왜 제한이 필요한가?**
```
모든 데이터를 열어두면 → 객체 외부에서 임의로 수정 가능
→ 데이터 일관성 깨짐 → 예측 불가능한 버그 발생
```

**2단계: 은행 계좌 예시로 설명**
```java
// ❌ 나쁜 예: 캡슐화 없음
class BankAccount {
    public double balance = 1000;  // 누구나 접근 가능
}

// 손님이 임의로 조작 가능
account.balance = 999999999;  // 💣 문제 발생!
account.balance = -1000000;   // 💣 음수 잔액?

// ✅ 좋은 예: 캡슐화 적용
class BankAccount {
    private double balance = 1000;  // 외부 직접 접근 불가
    
    public void withdraw(double amount) {
        if (amount > 0 && amount <= balance) {
            balance -= amount;  // 유효성 검증 후 수정
        } else {
            throw new IllegalArgumentException("출금 불가");
        }
    }
    
    public double getBalance() {
        return balance;  // 조회만 허용
    }
}
```

**3단계: 캡슐화의 4가지 이점**

| 이점 | 설명 | 예시 |
|------|------|------|
| **데이터 보호** | 잘못된 값으로부터 보호 | 음수 잔액 방지 |
| **검증 로직 적용** | 변경 전 무조건 검사 | 출금 시 잔액 확인 |
| **내부 구현 변경** | 외부 영향 없이 리팩토링 | DB 구조 변경해도 API 동일 |
| **권한 관리** | 누가 뭘 할 수 있는지 명확화 | 관리자만 초기화 가능 |

## 핵심 개념 정리

### 📌 접근 제어자의 4가지 레벨

```java
// 1️⃣ public - 가장 열려있음
public class PublicExample {
    public String publicData = "누구나 접근";
}

// 2️⃣ protected - 같은 패키지 + 상속받은 클래스
class ProtectedExample {
    protected String protectedData = "가족(상속)과 이웃(패키지)만";
}

// 3️⃣ package-private (default) - 같은 패키지만
class DefaultExample {
    String defaultData = "같은 패키지의 이웃들만";
}

// 4️⃣ private - 가장 제한적
class PrivateExample {
    private String privateData = "자신(클래스)만";
}
```

### 📌 캡슐화의 3가지 핵심 원칙

1. **정보 은닉 (Information Hiding)**
   - 내부 구현을 감춤
   - 인터페이스만 노출

2. **검증된 접근**
   - Getter/Setter를 통한 통제된 접근
   - 데이터 변경 시 항상 검증

3. **단일 책임 원칙**
   - 각 클래스가 자신의 데이터만 책임
   - 다른 객체의 내부를 수정하지 않음

### 📌 실무 패턴: Getter/Setter

```java
public class User {
    private String email;
    private int age;
    
    // ✅ 검증이 포함된 Setter
    public void setEmail(String email) {
        if (email == null || !email.contains("@")) {
            throw new IllegalArgumentException("유효한 이메일 형식 필요");
        }
        this.email = email;
    }
    
    // ✅ 방어적 복사 (외부 수정으로부터 보호)
    public String getEmail() {
        return this.email;
    }
    
    public void setAge(int age) {
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("유효한 나이 범위: 0-150");
        }
        this.age = age;
    }
}
```

## 실무 포인트

### 🛠️ 실제 사례 1: Spring의 Entity 클래스

```java
// JPA Entity 예시
@Entity
public class Account {
    @Id
    private Long id;
    
    @Column(nullable = false)
    private BigDecimal balance;
    
    // ✅ 비즈니스 로직을 메서드에 캡슐화
    public void deposit(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new InvalidAmountException("양수만 입금 가능");
        }
        this.balance = this.balance.add(amount);
    }
    
    public void withdraw(BigDecimal amount) {
        if (amount.compareTo(balance) > 0) {
            throw new InsufficientBalanceException("잔액 부족");
        }
        this.balance = this.balance.subtract(amount);
    }
    
    // ✅ 조회만 허용
    public BigDecimal getBalance() {
        return balance;
    }
}
```

### 🛠️ 실제 사례 4: Lombok의 @Getter/@Setter

```java
// Lombok으로 간편하게 캡슐화
@Getter
@Setter(AccessLevel.PACKAGE)  // package-private로 설정
public class Product {
    private Long id;
    private String name;
    
    @Setter(AccessLevel.PRIVATE)  // private setter
    private BigDecimal price;
    
    public void updatePrice(BigDecimal newPrice) {
        // 검증 로직
        if (newPrice.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException();
        }
        this.price = newPrice;
    }
}
```
