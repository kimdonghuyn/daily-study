## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 게임 캐릭터 관리 시스템 - 상속과 다형성 활용

## 모범 답안

이 문제는 **상속(Inheritance)과 다형성(Polymorphism)**으로 해결합니다.

### 1단계: 기본 구조 설계
```java
// 추상 기본 클래스 정의
public abstract class Character {
    protected String name;
    protected int hp;
    protected int mana;
    
    // 모든 캐릭터가 구현해야 할 메서드
    public abstract void attack();
    public abstract void useSkill();
    public abstract void move();
    
    // 공통 메서드
    public void takeDamage(int damage) {
        this.hp -= damage;
    }
}
```

### 2단계: 직업별 구체적인 클래스 구현
```java
// 전사
public class Warrior extends Character {
    private int armor;
    
    @Override
    public void attack() {
        System.out.println(name + "이(가) 칼로 베기 공격!");
        // 전사 공격 로직
    }
    
    @Override
    public void useSkill() {
        System.out.println(name + "이(가) 방어 태세 발동!");
        this.armor += 20;
    }
    
    @Override
    public void move() {
        System.out.println(name + "이(가) 무거운 갑옷을 입고 천천히 이동!");
    }
}

// 마법사

    private int spellPower;

        this.mana -= 20;
    }
    
    @Override
    public void useSkill() {
        System.out.println(name + "이(가) 순간이동!");
    }
    
    @Override
    public void move() {
        System.out.println(name + "이(가) 빠르게 이동!");
    }

// 궁수

```

### 3단계: 조건문 없는 깔끔한 사용
```java
public class GameManager {
    private List<Character> characters = new ArrayList<>();
    
    // 캐릭터 추가 (타입 상관없음)
    public void addCharacter(Character character) {
        characters.add(character);
    }
    
    // 모든 캐릭터에게 명령 (조건문 전혀 필요 없음!)
    public void commandAll(String action) {
        for (Character character : characters) {
            switch(action) {
                case "attack":
                    character.attack();      // 각 클래스의 구현이 자동으로 호출됨
                    break;
                case "skill":
                    character.useSkill();
                    break;
                case "move":
                    character.move();
                    break;
            }
        }
    }
    
    // 실행 예시
    public static void main(String[] args) {
        GameManager game = new GameManager();
        
        game.addCharacter(new Warrior("아서"));
        game.addCharacter(new Mage("메를린"));
        game.addCharacter(new Archer("로빈"));
        
        game.commandAll("attack");  // 모두 다른 방식으로 공격
        game.commandAll("skill");   // 모두 다른 스킬 사용
        game.commandAll("move");    // 모두 다른 방식으로 이동
    }
}
```

## 핵심 개념 정리

### 📌 상속 (Inheritance)
- **정의**: 기존 클래스의 속성과 메서드를 새로운 클래스가 물려받는 것
- **목적**: 공통된 특성을 한 곳에서 관리
- **키워드**: `extends`, `abstract`
- **이점**:
  - 코드 중복 제거
  - 일관된 인터페이스 제공
  - 유지보수성 향상

### 📌 다형성 (Polymorphism)
- **정의**: 같은 메서드 이름이 다양한 방식으로 동작하는 것
- **종류**:
  - **컴파일 타임 다형성** (메서드 오버로딩)
  - **런타임 다형성** (메서드 오버라이딩) ← 이 문제에서 사용
- **원리**: 부모 타입 참조변수가 자식 객체를 가리킬 수 있음
  ```java
  Character c1 = new Warrior();  // OK!
  Character c2 = new Mage();     // OK!
  Character c3 = new Archer();   // OK!
  ```

### 📌 추상 클래스 (Abstract Class)
- 인스턴스화 불가능
- 하위 클래스가 반드시 구현해야 할 메서드 정의 (계약)
- 공통 로직은 구현, 직업별 로직은 추상메서드로 강제

### 📌 SOLID 원칙 적용
| 원칙 | 적용 |
|------|-----|
| **O** - Open/Closed | 새 직업 추가는 쉽고(Open), 기존 코드 수정은 없음(Closed) |
| **L** - Liskov Substitution | 어느 Character 자식이든 대체 가능 |
| **D** - Dependency Inversion | Character 추상클래스에 의존 |

## 실무 포인트

### ✅ 장점
1. **확장성**: 새 직업 추가 = 새 클래스 하나만 생성
   ```java
   // 신규 직업 추가 (기존 코드 수정 0)
   public class Paladin extends Character {
       @Override
       public void attack() { /* ... */ }
       // 구현...
   }
   ```

2. **조건문 제거**: 유지보수 악의 근원인 `if-else` 체인 해결
3. **타입 안정성**: 컴파일 단계에서 실수 발견
4. **테스트 용이**: 각 클래스를 독립적으로 테스트 가능

### ⚠️ 주의사항

**1. 과도한 추상화 피하기**
```java
// ❌ 나쁜 예: 불필요하게 복잡
public abstract class LivingThing { }
public abstract class GameObject extends LivingThing { }
public abstract class Character extends GameObject { }
```

**2. 공통 메서드는 추상화하지 않기**
```java
// ✅ 좋은 예
public abstract class Character {
    // 공통 로직: 구현
    public void takeDamage(int damage) {
        this.hp = Math.max(0, hp - damage);
    }
    
    // 직업별 로직: 추상메서드
    public abstract void attack();
}
```

**3. 인터페이스 vs 추상 클래스 구분**
```java
// 추상 클래스: "이것은 Character다" (is-a)
public abstract class Character { }

// 인터페이스: "이것은 이러한 능력을 가진다" (can-do)
public interface Attackable {
    void attack();
}

// 다중 구현 가능 (조합의 유연성)
public class Warrior extends Character implements Attackable, Defensible { }
```
