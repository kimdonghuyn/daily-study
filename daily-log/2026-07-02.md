# 캡슐화와 접근 제어자 - 모범 답안

## 모범 답안

### 1️⃣ 직접 접근의 문제점

**만약 잔액에 누구나 직접 접근할 수 있다면:**

```java
// ❌ 문제가 있는 코드
public class BankAccount {
    public double balance = 10000; // 누구나 접근 가능
}

// 어디서나 이렇게 할 수 있음
account.balance = -50000;  // 음수 잔액?
account.balance = 999999999; // 무한정 증가?
```

**발생하는 문제들:**
- 🔴 **데이터 무결성 위반**: 논리적으로 불가능한 상태 (음수 잔액)
- 🔴 **보안 위협**: 권한 없는 사람이 임의로 수정
- 🔴 **감사 추적 불가**: 언제, 누가, 어떻게 수정했는지 알 수 없음
- 🔴 **비즈니스 로직 우회**: 수수료 계산, 한도 확인 등이 무시됨

### 2️⃣ 접근 제어자가 필요한 이유

**접근 제어자는 "계약서" 역할:**
- 객체의 상태를 보호하는 보안 경계선 설정
- 의도하지 않은 사용 방식 방지
- 나중에 내부 구현을 변경해도 외부에 영향 없음

### 3️⃣ 올바른 설계 방식

```java
public class BankAccount {
    private double balance;  // 🔒 직접 접근 차단
    private String accountNumber;
    private List<Transaction> transactionHistory = new ArrayList<>();
    
    // ✅ 공개 인터페이스: 입금
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("입금액은 0보다 커야 합니다");
        }
        balance += amount;
        recordTransaction("입금", amount);
    }
    
    // ✅ 공개 인터페이스: 출금
    public void withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("출금액은 0보다 커야 합니다");
        }
        if (amount > balance) {
            throw new InsufficientFundsException("잔액 부족");
        }
        balance -= amount;
        recordTransaction("출금", amount);
    }
    
    // ✅ 공개 인터페이스: 잔액 조회 (읽기만 허용)
    public double getBalance() {
        return balance;
    }
    
    // 🔒 private: 내부 로직 (외부에서 직접 호출 불가)
    private void recordTransaction(String type, double amount) {
        transactionHistory.add(new Transaction(type, amount, LocalDateTime.now()));
        // 거래 기록, 이상 탐지, 알림 등의 비즈니스 로직
    }
}

// ✅ 올바른 사용
BankAccount account = new BankAccount();
account.deposit(50000);      // OK
account.withdraw(10000);     // OK
double balance = account.getBalance(); // OK (읽기만 가능)

// ❌ 불가능한 사용
account.balance = -1000;     // 컴파일 에러!
```

## 핵심 개념 정리

### 📌 캡슐화 (Encapsulation)
- **정의**: 관련된 데이터와 메서드를 하나로 묶고, 내부 구현을 숨기는 개념
- **목표**: 객체의 상태 보호 + 변경의 영향 범위 최소화
- **원칙**: "구현은 숨기고, 인터페이스만 노출"

### 📌 접근 제어자 4가지

| 제어자 | 범위 | 사용 사례 |
|--------|------|---------|
| `public` | 모든 곳 | 공개 API, 외부에서 호출해야 하는 메서드 |
| `protected` | 같은 패키지 + 상속클래스 | 서브클래스가 재정의해야 하는 메서드 |
| `(default)` | 같은 패키지만 | 패키지 내부에서만 사용되는 클래스 |
| `private` | 같은 클래스만 | 내부 로직, helper 메서드, 필드 |

### 📌 Getter/Setter의 역할

```java
// ❌ 나쁜 예: setter로 모든 것을 허용
private int age;
public void setAge(int age) {
    this.age = age; // 150, -5 등이 들어올 수 있음
}

// ✅ 좋은 예: 비즈니스 규칙 적용
private int age;
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("유효하지 않은 나이");
    }
    this.age = age;
}

// ✅ 더 좋은 예: setter가 필요없으면 안 만들기
private final int age;
public BankAccount(int age) {
    this.age = age;
}
public int getAge() {
    return age;
}
```

### 📌 불변성 (Immutability)
- 한번 생성되면 변경할 수 없는 객체
- `private final`로 안전성 강화

```java
public class Transaction {
    private final String type;
    private final double amount;
    private final LocalDateTime timestamp;
    
    public Transaction(String type, double amount) {
        this.type = type;
        this.amount = amount;
        this.timestamp = LocalDateTime.now();
    }
    // getter만 제공, setter는 없음
}
```

## 실무 포인트

### 🏢 실제 적용 사례

**1. 금융 시스템 (결제, 환전)**
```java
public class Wallet {
    private double balance;
    private String userId;
    
    public synchronized void transfer(Wallet recipient, double amount) {
        // 동시성 제어
        if (this.balance < amount) {
            throw new InsufficientFundsException();
        }
        this.balance -= amount;
        recipient.balance += amount;
        
        // 감사 로그
        auditLog.record("TRANSFER", userId, amount);
    }
}
```

**2. 전자상거래 (상품, 재고)**
```java
public class Product {
    private int stock;
    private BigDecimal price;
    
    public void decreaseStock(int quantity) {
        if (stock < quantity) {
            throw new OutOfStockException();
        }
        stock -= quantity;
    }
    
    // 가격은 매니저만 변경 가능
    public void updatePrice(BigDecimal newPrice, User updater) {
        if (!updater.hasRole("PRICE_MANAGER")) {
            throw new UnauthorizedException();
        }
        this.price = newPrice;
    }
}
```

### ⚠️ 주의사항

**1. Over-encapsulation 피하기**

```java
// ❌ 과도함: 필요없는 getter/setter
private String firstName;
public String getFirstName() { return firstName; }
public void setFirstName(String firstName) { this.firstName = firstName; }

// ✅ 적절함: 필요한 것만

    this.email = newEmail;
}
```

**2. Getter 접근 제어도 고려**
```java
public class Employee {
    private double salary; // 민감한 정보
    
    public double getSalary() { // 누구나 호출 가능?
        return salary;
    }
    
    // ✅ 더 나은 방식
    public double getSalaryForCurrentUser(User viewer) {
        if (viewer.getId().equals(this.id) || viewer.isManager()) {
            return salary;
        }
        throw new UnauthorizedException();
    }
}
```

### 💡 설계 원칙 (SOLID)

- **Single Responsibility**: 각 클래스는 하나의 책임만
- **Open/Closed**: 확장에는 열려있고, 수정에는 닫혀있음
- **Liskov Substitution**: 자식 클래스는 부모 클래스 규약 준수
- **Interface Segregation**: 클라이언트별 인터페이스 분리
