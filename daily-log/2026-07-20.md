### 1. 🔧 문제 상황

# 상속과 다형성 - 모범 답안

## 모범 답안

**"공통점을 찾아 추상화하고, 구체적인 구현은 각 몬스터에게 위임하는 설계"**

현재 문제의 핵심은 **변경에 취약한 구조(Open-Closed Principle 위반)**입니다. 해결 방법은:

### 1단계: 공통점 추출 (추상화)
```java
// 모든 몬스터의 공통 특성과 행동을 정의
abstract class Monster {
    protected String name;
    protected int hp;
    protected int damage;
    
    // 공통 기능
    public void takeDamage(int damage) {
        this.hp -= damage;
    }
    
    // 각 몬스터가 다르게 구현할 부분
    abstract void attack(Monster target);
    abstract void useSkill();
    abstract int calculateDamage();
}
```

### 2단계: 다형성 구현 (구체적인 구현)
```java
class Slime extends Monster {
    @Override
    void attack(Monster target) {
        int damage = calculateDamage();
        target.takeDamage(damage);
    }
    
    @Override
    void useSkill() {
        // 슬라임만의 스킬
    }
    
    @Override
    int calculateDamage() {
        return this.damage + 5; // 슬라임 특성
    }
}

class Orc extends Monster {
    @Override
    void attack(Monster target) {
        int damage = calculateDamage();
        target.takeDamage(damage);
    }
    
    @Override
    void useSkill() {
        // 오크만의 스킬
    }
    
    @Override
    int calculateDamage() {
        return (int)(this.damage * 1.2); // 오크 특성
    }
}

class Dragon extends Monster {
    @Override
    void attack(Monster target) {
        int damage = calculateDamage();
        target.takeDamage(damage);
    }
    
    @Override
    void useSkill() {
        // 드래곤만의 스킬
    }
    
    @Override
    int calculateDamage() {
        return (int)(this.damage * 1.5); // 드래곤 특성
    }
}
```

### 3단계: 일관된 인터페이스로 관리
```java
class BattleSystem {
    private List<Monster> monsters = new ArrayList<>();
    
    // 모든 몬스터를 동일하게 처리
    public void simulateBattle(Monster attacker, Monster defender) {
        attacker.attack(defender);
        
        if(defender.getHp() <= 0) {
            monsters.remove(defender);
        }
    }
    
    // 새로운 몬스터 타입 추가 시 코드 수정 불필요
    public void addMonster(Monster monster) {
        monsters.add(monster);
    }
}
```

**핵심: 이제 새 몬스터(예: 좀비)를 추가할 때 기존 코드를 건드리지 않음**
```java
class Zombie extends Monster {
    // 새로운 몬스터만 구현하면 됨
}
```

## 핵심 개념 정리

### **상속 (Inheritance)**
- **목적**: 공통 특성과 행동을 부모 클래스에 정의하여 코드 중복 제거
- **관계**: `is-a` 관계 (슬라임은 몬스터이다)
- **이점**: 
  - 공통 기능 일괄 관리
  - 코드 재사용
  - 새로운 타입 추가 용이

### **다형성 (Polymorphism)**
- **목적**: 같은 메서드명으로 다양한 동작 구현
- **실제 동작**: 런타임에 객체의 실제 타입에 따라 메서드 결정
- **형태**:
  - **메서드 오버라이딩**: 부모의 메서드를 자식이 재정의
  - **메서드 오버로딩**: 매개변수 다르게 같은 이름 메서드 생성

### **추상화 (Abstraction)**
- **목적**: 반드시 구현해야 할 메서드를 강제하고 구체적 구현은 자유롭게
- **추상 클래스**: 공통점 + 강제할 메서드 정의
- **인터페이스**: 순수 계약(contract) 정의

### **SOLID 원칙 중 관련 원칙**
| 원칙 | 설명 |
|------|------|
| **Open-Closed** | 확장에는 열려있고(새 몬스터 추가) 수정에는 닫혀있음(기존 코드 수정 X) |
| **Liskov Substitution** | 모든 Monster 자식은 부모 자리에 치환 가능 |
| **Interface Segregation** | 필요한 메서드만 구현하는 작은 인터페이스 선호 |

## 실무 포인트

### ✅ **설계 시작 체크리스트**

1. **공통점 먼저 찾기**
   ```
   Q: 슬라임, 오크, 드래곤의 공통점은?
   A: 모두 hp가 있고, 공격할 수 있고, 스킬이 있다
   ```

2. **다양성 다음에 찾기**
   ```
   Q: 어디가 다른가?
   A: 공격 방식, 스킬, 데미지 계산이 다르다
   ```

3. **추상화 레벨 결정**
   ```java
   // ❌ 너무 구체적 (각 몬스터 클래스 필요)
   class SlimeAttack { }
   class OrcAttack { }
   
   // ✅ 적절한 추상화
   abstract class Monster {
       abstract void attack();
   }
   ```

### 🔄 **변경 시나리오별 영향도**

```
현재 코드 (나쁜 구조):
- 새 몬스터 추가 → 기존 코드 4곳 수정 필요
- 공격 로직 변경 → 3개 클래스 모두 수정 필요

개선된 코드 (상속/다형성):
- 새 몬스터 추가 → Monster 상속 1줄, 메서드 구현만 추가
- 공격 로직 변경 → Monster 부모 클래스만 수정
```

### 💡 **성장 패턴 예시**

```java
// Phase 1: 초기 상태 (상속/다형성 적용)
Monster[] monsters = {new Slime(), new Orc(), new Dragon()};
for(Monster m : monsters) m.attack(target);

// Phase 2: 복잡도 증가 (디자인 패턴 추가)
// → 팩토리 패턴으로 몬스터 생성 관리
// → 옵저버 패턴으로 이벤트 처리
// → 상태 패턴으로 몬스터 상태 변경

// Phase 3: 스케일 증가 (시스템 아키텍처)
// → 영속성 계층 (DB 저장)
// → 직렬화 (네트워크 전송)
```
