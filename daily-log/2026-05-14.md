### 1. 📖 개념 설명

# GC의 필요성과 동작 원리

## 모범 답안

### 메모리 정리가 없을 때의 문제점

**메모리 누수(Memory Leak)** 문제가 발생합니다:

```
프로그램 실행 시간 → 메모리 점유량 계속 증가 → Out of Memory 에러 → 프로그램 강제 종료
```

예를 들어, Java에서 GC 없이 object를 계속 생성하면:
- 1시간 운영 시 메모리 100MB 사용 가능
- 2시간 운영 시 메모리 200MB 필요
- 결국 시스템 메모리 부족 → 서버 크래시

### GC가 필요한 근본적 이유

**1. 개발 생산성 향상**
```c
// C/C++ - 수동 메모리 관리 (버그 위험)
int* ptr = malloc(sizeof(int));
*ptr = 10;
free(ptr);  // 개발자가 일일이 해제해야 함
ptr = NULL; // 실수 시 double-free, use-after-free 위험
```

**2. 메모리 안전성 보장**
- 수동 해제 시 발생하는 버그들:
  - **Use-After-Free**: 해제된 메모리 접근
  - **Double-Free**: 같은 메모리 두 번 해제
  - **Memory Leak**: 해제 안 한 메모리 누적

**3. 멀티스레드 환경의 복잡성 제거**
```java
// Java - 자동 메모리 관리
List<String> list = new ArrayList<>();
for(int i = 0; i < 1000000; i++) {
    list.add("data" + i);
}
// GC가 자동으로 필요없는 객체들을 정리해줌
```

## 핵심 개념 정리

### 📌 GC의 기본 원리

**Mark-Sweep-Compact 알고리즘**
```
1️⃣ Mark(표시) - Root에서 도달 가능한 객체 마크
2️⃣ Sweep(청소) - 마크 안 된 객체 메모리 해제  
3️⃣ Compact(압축) - 조각난 메모리 정렬
```

**Root 객체 정의**
- 스택의 지역변수
- 정적 변수(static)
- JNI 참조
- 현재 실행 중인 메서드의 참조들

### 📌 GC의 동작 방식

**Generational Hypothesis(세대별 가설)**
```
Young Generation          Old Generation
┌─────────┐              ┌──────────┐
│ Eden    │─────┐        │          │
│ S0, S1  │     └───────→│ Tenured  │
└─────────┘     대부분    └──────────┘
(짧은 생명)     여기서    (긴 생명)
             죽음
```

**GC 종류별 특성**

| GC 종류 | 특징 | 적합한 경우 |
|--------|------|-----------|
| Serial GC | Single-threaded, 간단함 | 소규모 애플리케이션 |
| Parallel GC | Multi-threaded | CPU 중심 배치 처리 |
| CMS GC | Low pause time | 낮은 지연시간 필요 |
| G1 GC | Region 기반 | 대용량 힙(4GB+) |
| ZGC | Ultra low-latency | 밀리초 단위 정지 불가 |

### 📌 메모리 구조

```
┌─────────────────────┐
│   Stack (스레드별)   │ ← 빠른 접근, 자동 해제
├─────────────────────┤
│                     │
│   Heap              │ ← GC 관리 대상
│  (공유 메모리)       │
│                     │
└─────────────────────┘
```

### 📌 GC 성능 지표

- **GC Pause Time**: GC 실행 중 프로그램 정지 시간
- **Throughput**: GC 오버헤드 제외 프로그램 실행 시간
- **GC Frequency**: GC 발생 빈도
- **Heap Usage**: 메모리 사용 효율성

## 실무 포인트

### ⚡ 성능 최적화 전략

**1. GC 튜닝 - 힙 크기 설정**
```bash
# Xms: 초기 힙 크기, Xmx: 최대 힙 크기 (동일 권장)
java -Xms2G -Xmx2G -XX:+UseG1GC application.jar
```

**2. GC 로그 분석**
```bash
java -XX:+PrintGCDetails -XX:+PrintGCDateStamps \
     -Xloggc:gc.log application.jar
```

**3. 메모리 누수 감지**
```java
// 의도하지 않은 객체 참조 예시
public class LeakExample {
    private static List<byte[]> list = new ArrayList<>(); // static!
    
    public void addData(int size) {
        list.add(new byte[size]); // 메모리가 계속 쌓임
        // 절대 자동으로 정리되지 않음!
    }
}
```

### 🔍 실제 현장 이슈

**Case 1: 예상치 못한 Full GC**
```
증상: 갑자기 응답 시간 급증 (수 초)
원인: Old Generation 메모리 부족
해결: 힙 크기 증가, 혹은 메모리 누수 코드 수정
```

**Case 2: GC 빈도가 너무 높음**
```
증상: CPU 사용률 높음 (50% 이상이 GC)
원인: 힙 크기 부족 or 불필요한 임시 객체 대량 생성
해결: 
  - 힙 크기 증가
  - 객체 재사용 패턴 (Object Pool)
  - StringBuilder 사용 (String 대신)
```

**Case 3: GC 오버헤드 모니터링**
```java
// Spring Boot에서 메트릭 수집
@Configuration
public class GCMetrics {
    // Micrometer로 GC 메트릭 자동 수집
    // Prometheus + Grafana로 시각화
}
```

### 💡 개발자가 GC를 고려할 때

**피해야 할 패턴:**
```java
// ❌ 루프에서 대량 객체 생성
for(int i = 0; i < 1000000; i++) {
    String result = new String("data" + i); // Young Gen 바뀜
}

// ✅ 객체 재사용
StringBuilder sb = new StringBuilder();
for(int i = 0; i < 1000000; i++) {
    sb.append("data").append(i);
}
```

**모니터링 습관:**
```bash
# JVM 현재 상태 확인
jps
jstat -gc <pid> 1000

# 힙 덤프 분석
jmap -dump:live,format=b,file=heap.bin <pid>
jhat heap.bin
```

## 요약

GC는 **개발 생산성**과 **메모리 안전성**을 위한 필수 시스템이지만, 성능 목표를 달성하려면 **GC 동작을 이해하고 적절히 튜닝**해야 합니다. 특히 대규모 시스템에서는 GC Pause Time 최소화가 중요한 성능 지표입니다.
