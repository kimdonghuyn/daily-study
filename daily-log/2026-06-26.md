## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# JVM 메모리 구조와 OutOfMemoryError 분석

## 모범 답안

**현상의 원인: 메모리 누수(Memory Leak)**

프로그램이 재시작하면 다시 잘 도는 이유는, **Heap 메모리가 초기화되기 때문**입니다. 며칠 동안 잘 돌다가 에러가 나는 것은 다음 메커니즘입니다:

1. **점진적 메모리 누수**: 사용이 끝난 객체들이 여전히 참조되어 있어 GC(Garbage Collector)의 대상이 되지 않음
2. **Heap 고갈**: 생성된 객체는 계속 쌓이고, Heap 크기 한계에 도달
3. **에러 발생**: Full GC를 시도해도 메모리를 확보할 수 없어 OutOfMemoryError 발생
4. **재시작 후 정상**: Heap이 비워지며 다시 초기화되므로 같은 현상이 반복

### Heap의 역할
- **모든 객체와 배열의 인스턴스가 할당되는 메모리 영역**
- **Garbage Collector가 자동으로 관리** (하지만 완벽하지 않음)
- **런타임에 동적으로 할당되고 해제되어야 함**

## 핵심 개념 정리

### 📊 JVM 메모리 구조

| 영역 | 특징 | 생명주기 | 주의사항 |
|------|------|---------|---------|
| **Stack** | 메서드 호출, 로컬 변수, 참조 저장 | 메서드 실행 중에만 유지 | 스레드마다 독립적 |
| **Heap** | 객체 인스턴스 저장 | 참조가 있는 동안 유지 | GC의 대상, 공유 영역 |
| **Method Area** | 클래스 구조, 메서드, 상수 | JVM 시작~종료 | 메타데이터 저장 |

### 🔴 Heap 메모리 세대별 구조

```
Young Generation (대부분의 객체가 여기서 죽음)
├── Eden Space (새로운 객체 할당)
├── Survivor Space 0 (Minor GC에서 살아남은 객체)
└── Survivor Space 1

Old Generation (장수한 객체)
└── 여러 번 Minor GC를 견딘 객체들

Permanent/Metaspace (클래스 메타데이터)
```

### 🚨 메모리 누수 패턴

```java
// ❌ 문제 코드: Static 컬렉션에 계속 추가
public class BadExample {
    static List<Data> cache = new ArrayList<>();  // 절대 비워지지 않음!
    
    public void addToCache(Data data) {
        cache.add(data);  // 며칠 지나면 OutOfMemoryError
    }
}

// ❌ 문제 코드: Listener 등록 후 해제 안 함
public class EventSource {
    List<EventListener> listeners = new ArrayList<>();
    
    public void registerListener(EventListener listener) {
        listeners.add(listener);  // 제거 안 하면 누수!
    }
}

// ❌ 문제 코드: 자원 미해제
public void processFile() {
    FileInputStream fis = new FileInputStream("large_file.txt");
    // 예외 발생하면 close() 호출 안 됨
}

// ✅ 해결: try-with-resources
public void processFileCorrect() {
    try (FileInputStream fis = new FileInputStream("large_file.txt")) {
        // 자동으로 close() 호출
    }
}
```

### 🔍 메모리 생명주기

```
1. New (Allocation)
   ↓
2. Live (참조 유지)
   ↓
3. Unreachable (참조 끊김)
   ↓
4. GC 대상
   ↓
5. Garbage (메모리 해제)
```

### ✅ 메모리 누수 예방 체크리스트

#### 1️⃣ **Collections 안전하게 관리**
```java
// ❌ 위험
static Map<String, Object> cache = new HashMap<>();

// ✅ 안전 (용량 제한)
static Map<String, Object> cache = new LinkedHashMap<String, Object>() {
    protected boolean removeEldestEntry(Map.Entry eldest) {
        return size() > MAX_ENTRIES;
    }
};

// ✅ 더 나은 방법 (라이브러리 사용)
Cache<String, Object> cache = CacheBuilder.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build();
```

#### 2️⃣ **Listener/Observer 패턴에서 등록 해제**
```java
// ✅ 반드시 언젠가 unregister 되어야 함
EventBus.getInstance().register(this);

@PreDestroy  // 객체 소멸 전 호출
public void cleanup() {
    EventBus.getInstance().unregister(this);
}
```

#### 3️⃣ **명시적 자원 해제**
```java
// ✅ Try-with-resources (Java 7+)
try (Connection conn = dataSource.getConnection()) {
    // ...
} // 자동으로 close() 호출

// ✅ Try-finally (필요시)
Connection conn = null;
try {
    conn = dataSource.getConnection();
    // ...
} finally {
    if (conn != null) {
        conn.close();
    }
}
```

### 📈 **메모리 모니터링**

```java
// JVM 메모리 상태 확인
MemoryMXBean memoryBean = ManagementFactory.getMemoryMXBean();
MemoryUsage heapUsage = memoryBean.getHeapMemoryUsage();

long used = heapUsage.getUsed();      // 사용 중인 메모리
long max = heapUsage.getMax();        // 최대 메모리
double usagePercent = (double) used / max * 100;

logger.warn("Heap Usage: {}/{} ({}%)", 
    formatBytes(used), 
    formatBytes(max), 
    String.format("%.2f", usagePercent));

// 주의: 85% 이상이면 위험 신호!
```

### 🔧 **JVM 옵션으로 모니터링**

```bash
# 메모리 크기 설정
java -Xms1024m -Xmx2048m MyApp

# GC 로그 출력
java -Xms1024m -Xmx2048m \
     -XX:+PrintGCDetails \
     -XX:+PrintGCTimeStamps \
     -Xloggc:gc.log \
     MyApp

# Heap dump 생성 (OutOfMemoryError 시)
java -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/var/logs/heapdump.hprof \
     MyApp
```
