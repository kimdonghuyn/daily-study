## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# JVM 메모리 구조와 성능 문제 해석

## 모범 답안

Java 프로그램이 실행될 때, JVM이 할당받은 메모리는 **용도별로 나뉘어 관리**됩니다.

### 간단한 비유
학교 식당을 생각해보세요:
- **Stack**: 각 학생이 먹을 식판(현재 필요한 것만)
- **Heap**: 공용 냉장고(여러 학생이 공유하는 음식)
- **Method Area**: 식당의 레시피/조리법(프로그램 구조)

**문제 상황 분석:**

100명이 동시에 사용할 때 느려지는 이유:

```
1. Stack 메모리 부족
   └─ 각 사용자(스레드)마다 독립적인 Stack이 필요
   └─ 100명 = 100개의 Stack 필요 → 제한된 메모리 초과

2. Heap 메모리 부족
   └─ 객체(데이터)가 계속 쌓임
   └─ Garbage Collection(정리)이 자주 발생 → 느려짐
   └─ 결국 메모리 부족으로 프로그램 중단

3. 메모리 누수(Memory Leak)
   └─ 불필요한 객체가 계속 메모리 차지
   └─ 사용자 많을수록 누적 가속화
```

## 핵심 개념 정리

### 📊 JVM 메모리 구조

#### **1. Stack (스택)**
```
특징:
├─ 각 스레드마다 독립적으로 할당
├─ LIFO(Last In First Out) 구조
├─ 매우 빠른 메모리 접근
├─ 메모리 크기: 보통 1MB ~ 8MB (고정)
└─ 메서드 호출, 지역변수, 참조값 저장

용도:
├─ 메서드 실행 정보
├─ 지역 변수
├─ 메서드 호출 순서 추적
└─ 참조 타입 변수의 참조값(주소)

위험성:
└─ StackOverflowError 발생 가능
   (무한 재귀 호출 등)
```

**Stack 문제 시나리오:**
```java
// 위험한 코드: 무한 재귀
public void recursiveMethod() {
    recursiveMethod();  // 계속 Stack 쌓임 → StackOverflowError
}

// 100명 동시 접속 시
// 100명 × 4MB(스택 크기) = 400MB만 스택으로 사용
```

#### **2. Heap (힙)**
```
특징:
├─ 모든 스레드가 공유
├─ 동적 메모리 할당
├─ Garbage Collection으로 자동 정리
├─ 메모리 크기: 매우 크지만 제한됨 (보통 최대 1GB~)
└─ 객체(instance) 저장

용도:
├─ 객체 인스턴스
├─ 배열
└─ 문자열 상수 풀(String constant pool)

위험성:
└─ OutOfMemoryError: Java heap space 발생
   (메모리 부족 시)
```

**Heap 문제 시나리오:**
```java
// 문제 코드: 계속 객체 생성
List<UserData> userCache = new ArrayList<>();

for (int i = 0; i < 100000; i++) {
    UserData user = new UserData();  // Heap에 계속 쌓임
    userCache.add(user);
}

// 100명이 동시에 접속 → Heap 메모리 부족
// GC(쓰레기 수집)가 자주 실행 → 느려짐
```

#### **3. Method Area (메서드 영역)**
```
특징:
├─ 모든 스레드가 공유
├─ JVM 시작부터 종료까지 유지
├─ 런타임 상수 풀 저장
├─ 클래스 구조, 메서드 데이터 저장
└─ 고정 크기 (보통 64MB~512MB)

용도:
├─ 클래스 정보
├─ 메서드 코드
├─ 상수(static 변수)
└─ 필드 정보

위험성:
└─ PermGen/Metaspace 부족
   (동적으로 클래스 생성 시)
```

### 📈 메모리 할당 크기 예시
```
JVM -Xms512m -Xmx2g 실행 시:
├─ Heap: 512MB ~ 2GB
├─ Stack (스레드당): 1MB (기본값)
├─ Method Area: 128MB (가변)
└─ 기타: ~100MB

→ 최대 스레드 수: 대략 (2GB - 메서드영역) / 1MB ≈ 1900개
```

### 🚨 **100명 동시 접속 문제 진단**

#### **1단계: 증상 확인**


**확인할 것:**
```
1. GC 빈도가 매우 높은가? → Heap 부족
2. Full GC 후에도 메모리가 줄어들지 않는가? → 메모리 누수
3. User Threads가 계속 증가하는가? → Stack 부족
```

#### **2단계: 원인별 대처**

**문제 1: Stack 메모리 부족**
```java
// ❌ 문제 코드: 깊은 재귀
public int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 큰 숫자면 StackOverflow
}

// ✅ 해결: 반복문 사용
public int factorial(int n) {
    int result = 1;
    for (int i = 2; i <= n; i++) {
        result *= i;
    }
    return result;
}
```

**JVM 옵션으로 Stack 크기 증가:**
```bash
# 각 스레드 스택 크기 2MB로 설정
java -Xss2m MyApplication
```

```java
// ✅ 해결 2: Object Pool 패턴
class UserDataPool {
    private Queue<UserData> pool = new LinkedList<>();
    
    public UserData acquire() {
        return pool.isEmpty() ? new UserData() : pool.poll();
    }
    
    public void release(UserData data) {
        data.reset();  // 초기화
        pool.offer(data);  // 재사용
    }
}
```
