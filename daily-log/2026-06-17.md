## 🤖 AI 피드백

✅ 잘한 점 (구체적으로)
- 트랜잭션의 핵심인 '원자성(Atomicity)' 개념을 정확히 이해하고 있음
- 출금과 입금 사이의 실패 시나리오를 구체적으로 제시함
- 롤백(Rollback) 개념을 명확히 이해하고 있음

⚠️ 보완할 점 (구체적으로)
- ACID 중 원자성만 언급했으나, 트랜잭션이 보장하는 다른 특성들도 설명 필요
  (일관성/Consistency: 데이터 무결성, 격리성/Isolation: 동시성 제어, 지속성/Durability)
- @Transactional 없을 때의 구체적 문제를 더 다각도로 설명 부족
  예: 출금은 성공, 입금 중 DB 연결 끊김 → A 계좌 -100만원, B 계좌 변화 없음 (돈 소실)
  예: 동시에 여러 사용자가 같은 계좌에 접근할 때의 Dirty Read/Lost Update 문제
- Spring의 @Transactional의 동작 메커니즘 설명 부족
  (프록시 객체, AOP 기반 구현, default Isolation Level인 READ_COMMITTED 등)
- 예외 발생 시 자동 롤백되는 조건을 언급하지 않음
  (checked exception은 기본적으로 롤백 안 됨, rollbackFor 설정 필요)

💡 핵심 포인트 (놓친 부분)
- 트랜잭션의 4가지 특성(ACID) 전체를 언급하면 더 완벽한 답변
- @Transactional이 없을 때 동시성 문제(Race Condition)도 함께 설명할 수 있으면 시니어 레벨
- 실무에서 자주 만나는 함정: 같은 클래스 내 @Transactional 메서드 호출 시 프록시가 작동하지 않는 문제

### 1. 🎤 면접 질문

# @Transactional 기본 동작 - 면접 답변 가이드

## 모범 답안

### 메인 질문에 대한 답변

네, 실무에서 자주 사용하고 있습니다. @Transactional은 **데이터베이스 트랜잭션의 ACID 특성을 보장하기 위한 어노테이션**입니다.

은행 송금 예시를 들면, A 계좌에서 출금하는 로직과 B 계좌에 입금하는 로직은 **반드시 함께 성공하거나 함께 실패해야 합니다**. 

```java
@Transactional
public void transferMoney(Long fromAccountId, Long toAccountId, BigDecimal amount) {
    Account fromAccount = accountRepository.findById(fromAccountId);
    Account toAccount = accountRepository.findById(toAccountId);
    
    fromAccount.withdraw(amount);  // UPDATE 실행
    toAccount.deposit(amount);      // UPDATE 실행
    
    accountRepository.save(fromAccount);
    accountRepository.save(toAccount);
}
```

@Transactional이 보장하는 것은:
- 둘 다 성공 → commit
- 중간에 예외 발생 → rollback (모든 변경사항 취소)
- 데이터 일관성 보장

### 꼬리 질문에 대한 답변

@Transactional **없이** 실행하면 다음과 같은 구체적인 문제들이 발생합니다:

```java
// @Transactional 없는 상황
public void transferMoneyWithoutTransaction(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepository.findById(fromId);
    Account to = accountRepository.findById(toId);
    
    from.withdraw(amount);
    accountRepository.save(from);  // ✅ DB에 반영됨
    
    // ❌ 여기서 NullPointerException 발생!
    to.deposit(amount);
    accountRepository.save(to);    // 실행 안 됨
}
```

**발생 가능한 문제들:**

| 시나리오 | 결과 | 손해 |
|---------|------|------|
| **Dirty Write** | A 계좌: -100만 원 저장 후 예외 발생 | A에서만 차감, B는 입금 안 됨 (100만 원 소실) |
| **Inconsistent Read** | 트랜잭션 중 다른 쿼리가 부분 데이터 읽음 | 임시 상태의 데이터 조회 |
| **Lost Update** | 동시성 문제 발생 | 두 출금이 동시에 처리되어 중복 차감 |
| **Phantom Read** | 조회 후 INSERT되는 데이터까지 조회됨 | 예기치 않은 데이터 포함 |

## 핵심 개념 정리

### 1️⃣ **ACID 특성**
- **A (Atomicity)**: All or Nothing - 모두 성공하거나 모두 실패
- **C (Consistency)**: 데이터 무결성 보장 (제약조건 준수)
- **I (Isolation)**: 트랜잭션 간 격리 (다른 tx의 영향 안 받음)
- **D (Durability)**: 커밋된 데이터는 영구 보존

### 2️⃣ **@Transactional의 동작 원리**
```java
// Spring AOP를 이용한 프록시 패턴
@Transactional
public void someMethod() {
    // 1. 트랜잭션 시작 (BEGIN)
    // 2. 메서드 실행
    // 3. 예외 없음 → COMMIT
    // 4. 예외 발생 → ROLLBACK
}
```

### 3️⃣ **주요 속성들**
```java
@Transactional(
    propagation = Propagation.REQUIRED,  // 기존 tx 참여 또는 새 tx 생성
    isolation = Isolation.DEFAULT,       // DB 기본값 (MySQL: READ_COMMITTED)
    readOnly = false,                    // 읽기 전용 여부
    timeout = -1,                        // 타임아웃 (초 단위, -1 = 무제한)
    rollbackFor = Exception.class         // 롤백 대상 예외
)
public void transferMoney(...) { }
```

### 4️⃣ **Isolation Level 비교**
| Level | Dirty Read | Non-Repeatable Read | Phantom Read |
|-------|-----------|-------------------|--------------|
| **READ_UNCOMMITTED** | ✅ | ✅ | ✅ |
| **READ_COMMITTED** | ❌ | ✅ | ✅ |
| **REPEATABLE_READ** | ❌ | ❌ | ✅ |
| **SERIALIZABLE** | ❌ | ❌ | ❌ |

### 5️⃣ **Propagation 전파 옵션**
```java
// REQUIRED: 기존 tx가 있으면 참여, 없으면 새로 생성 (기본값)
@Transactional(propagation = Propagation.REQUIRED)

// REQUIRES_NEW: 항상 새로운 tx 생성 (기존 tx 일시 중지)
@Transactional(propagation = Propagation.REQUIRES_NEW)

// NESTED: 기존 tx 내 중첩 tx 생성 (savepoint 활용)
@Transactional(propagation = Propagation.NESTED)

// NOT_SUPPORTED: tx 없이 실행
@Transactional(propagation = Propagation.NOT_SUPPORTED)
```

## 실무 포인트

### ✅ **올바른 사용법**

```java
// 1. 서비스 계층에 적용 (권장)
@Service
public class AccountService {
    @Transactional
    public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
        // 비즈니스 로직
    }
}

// 2. 읽기 전용 쿼리 최적화
@Transactional(readOnly = true)
public List<Account> getAllAccounts() {
    return accountRepository.findAll();
}

// 3. 특정 예외만 롤백
@Transactional(rollbackFor = {BusinessException.class})
public void complexOperation() {
    // RuntimeException은 자동 롤백
    // checked exception은 롤백 안 됨 (명시 필요)
}
```

### ❌ **주의사항**

```java
// ❌ 잘못된 예: 프라이빗 메서드 (프록시 바이패스)
@Transactional
private void privateMethod() { }

// ❌ 잘못된 예: self-invocation (tx 적용 안 됨)
public void publicMethod() {
    this.privateTransactionalMethod();  // tx 미적용
}

// ❌ 잘못된 예: checked exception 롤백 안 됨
@Transactional  // IOException은 롤백 안 됨
public void readFile() throws IOException { }

// ✅ 올바른 예:
@Transactional(rollbackFor = IOException.class)
public void readFile() throws IOException { }

// ❌ 잘못된 예: 컨트롤러에 적용 (안티패턴)
@RestController
@Transactional  // 하지 말 것!
public class AccountController { }
```
