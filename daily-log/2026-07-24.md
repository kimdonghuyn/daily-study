## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 상속과 다형성 - 음식 배달 앱 설계

## 모범 답안

**절대 아닙니다.** 복사-붙여넣기는 유지보수의 악순환입니다. 이 상황은 상속과 다형성의 **가장 전형적인 사용 사례**입니다.

**해결 방안:**

1. **공통 부분을 추상화한 기반 클래스 설계**
```java
public abstract class Food {
    protected String name;
    protected int price;
    
    // 공통 흐름은 Template Method 패턴으로
    public final void processOrder(Order order) {
        confirmOrder(order);      // 주문 확인
        processPayment(order);    // 결제 처리
        startCooking(order);      // 조리 시작
        deliverFood(order);       // 배달
    }
    
    // 각 음식별로 구현하도록 강제
    protected abstract void startCooking(Order order);
    
    // 공통 로직
    protected void confirmOrder(Order order) {
        // 모든 음식 공통
    }
    
    protected void processPayment(Order order) {
        // 모든 음식 공통
    }
    
    protected void deliverFood(Order order) {
        // 모든 음식 공통
    }
}
```

2. **각 음식 종류별 구현**
```java
public class Pizza extends Food {
    public Pizza() {
        this.name = "피자";
        this.price = 15000;
    }
    
    @Override
    protected void startCooking(Order order) {
        System.out.println("오븐에서 180도로 12분 조리");
        // 피자만의 조리 로직
    }
}

public class Chicken extends Food {
    public Chicken() {
        this.name = "치킨";
        this.price = 18000;
    }
    
    @Override
    protected void startCooking(Order order) {
        System.out.println("튀김기에서 180도로 18분 조리");
        // 치킨만의 조리 로직
    }
}

public class ChinesFood extends Food {
    @Override
    protected void startCooking(Order order) {
        System.out.println("웍에서 강화력으로 5분 조리");
    }
}
```

3. **다형성을 활용한 통일된 처리**
```java
public class OrderService {
    // 음식 종류를 알 필요가 없음 - 다형성!
    public void handleOrder(Food food, Order order) {
        food.processOrder(order);  // 같은 메서드 호출
        // Pizza든 Chicken이든 자신의 startCooking() 실행됨
    }
}

// 사용
Food pizza = new Pizza();
Food chicken = new Chicken();
Food chinese = new ChinesFood();

orderService.handleOrder(pizza, order1);    // 피자 조리 로직 실행
orderService.handleOrder(chicken, order2);  // 치킨 조리 로직 실행
```

**비용-편익:**
- **추가 전에:** 코드 3줄 + 13줄 + 13줄 = 29줄 × 13가지 음식 = **~377줄**
- **상속 후:** 기반 클래스 25줄 + 각 음식 5줄 × 13가지 = **~90줄**
- **버그 수정:** 공통 로직 수정 시 1곳만 변경 (vs 13곳 모두 수정)

## 핵심 개념 정리

### 📌 **상속 (Inheritance)**
- **정의:** 기반 클래스(부모)의 속성과 메서드를 자식 클래스가 물려받음
- **목적:** 코드 중복 제거 및 계층적 구조 표현
- **이 경우:** `Food` 클래스의 공통 흐름을 모든 음식이 상속

### 📌 **다형성 (Polymorphism)**
- **정의:** 같은 인터페이스로 다양한 객체를 처리하는 능력
- **메커니즘:** 실행 시점에 실제 객체 타입의 메서드가 호출됨
- **이 경우:** `processOrder()`는 동일하지만, `startCooking()`은 각 음식마다 다르게 동작

### 📌 **메서드 오버라이딩 (Overriding)**
- **정의:** 부모 클래스의 메서드를 자식에서 재정의
- **조건:** 서명(시그니처)은 같되, 구현만 다름
- **이 경우:** `startCooking()`을 각 음식이 다르게 구현

### 📌 **Abstract(추상화)**
- **추상 클래스:** 인스턴스 생성 불가, 설계도 역할
- **추상 메서드:** 구현 없이 선언만 하고 자식이 반드시 구현하도록 강제
```java
public abstract void startCooking(Order order);  // 각 음식이 반드시 구현해야 함
```

### 📌 **Template Method 패턴**
- **개념:** 알고리즘의 골격은 기반 클래스에, 구체적 단계는 자식이 구현
- **이 경우:** `processOrder()`는 흐름을 정하고, `startCooking()`은 자식이 정의

### ✅ **실무 체크리스트**

| 상황 | 상속 적용 | 효과 |
|------|---------|------|
| **1. 음식 추가** | 새 클래스만 작성 | +5줄 vs 복사+수정 +15줄 |
| **2. 공통 로직 변경** | 기반 클래스만 수정 | 1번의 수정 vs 13번의 수정 |
| **3. 배열/리스트 처리** | `List<Food>` 사용 가능 | 조건문 제거 (if-else 지옥 탈출) |
| **4. 테스트** | 각 클래스만 테스트 | 공통 부분은 1번만 검증 |
