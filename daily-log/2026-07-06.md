## 🤖 AI 피드백

✅ 잘한 점 (구체적으로)
- 상속과 다형성의 핵심 개념을 정확히 이해하고 있음 (공통 관리, 오버라이딩)
- 부모 클래스에 공통 기능을 모으고, 자식 클래스에서 특화된 동작을 구현한다는 기본 원리를 올바르게 설명함
- 문제 상황에 대한 해결책으로서 상속의 필요성을 명확히 인식하고 있음

⚠️ 보완할 점 (구체적으로)
- '다형성'의 개념이 명확하게 드러나지 않음. 오버라이딩은 다형성의 한 형태일 뿐, 다형성의 진정한 힘은 "부모 타입으로 자식 객체를 다루면서도 각자의 동작을 수행한다"는 점임
  예: Character[] characters = {new Player(), new Monster(), new Boss()}; 이런 식으로 다양한 타입을 하나의 컬렉션으로 관리하고, for(Character c : characters) { c.attack(); } 처럼 통일된 인터페이스로 다양한 동작을 수행하는 것
- 실제 코드 구조를 제시하지 않아 추상적 수준에만 머물러 있음. 구체적인 클래스 설계(Character 부모 클래스 정의, 메서드 선언 등)를 보여줬으면 더 설득력 있었을 것
- "50개 이상의 캐릭터 추가"라는 구체적인 문제 상황에서 얼마나 유지보수가 쉬워지는지 정량적으로 설명하지 않음
- 추상 클래스(Abstract Class) vs 인터페이스(Interface) 같은 선택지에 대한 언급이 없음

💡 핵심 포인트 (놓친 부분)
- "다형성"의 진정한 가치: 부모 참조변수로 모든 캐릭터를 관리할 수 있어, 새 캐릭터 추가 시 기존 코드(게임 루프, 렌더링 등)를 수정할 필요가 없다는 Open/Closed Principle
- 템플릿 메서드 패턴 같은 구체적인 디자인 패턴의 언급이 있으면 업계 표준적 답변이 됨
- 실제 구현 시 주의점: 공통 로직은 부모에 구현, 캐릭터별 차이는 오버라이드 또는 추상 메서드 강제

## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 상속과 다형성 - 게임 캐릭터 설계

## 모범 답안

### 문제 분석
현재 상황은 **코드 중복(Code Duplication)**과 **유지보수성 악화**라는 전형적인 OOP 문제입니다. 50개 이상의 캐릭터 클래스가 동일한 로직을 반복하면:
- 버그 수정 시 모든 파일 수정 필요 → 실수 위험 증대
- 새 기능 추가 시 50곳 모두 수정 → 생산성 저하
- 로직 일관성 유지 어려움 → 예측 불가능한 동작

### 해결 방안: 상속과 다형성 활용

```java
// 1️⃣ 부모 클래스: 공통 기능 정의
public abstract class Character {
    protected int hp;
    protected int attackPower;
    
    // 공통 구현 (Template Method Pattern)
    public final void takeDamage(int damage) {
        hp -= damage;
        if (hp <= 0) {
            die();
        }
    }
    
    // 추상 메서드 (각 캐릭터별 구현)
    abstract void move();
    abstract void attack();
    abstract void die();
}

// 2️⃣ 자식 클래스: 고유 기능만 구현
public class Player extends Character {
    @Override
    void move() {
        System.out.println("플레이어가 움직입니다");
    }
    
    @Override
    void attack() {
        System.out.println("플레이어가 공격합니다: " + attackPower + " 데미지");
    }
    
    @Override
    void die() {
        System.out.println("플레이어가 죽었습니다");
    }
}

public class Monster extends Character {
    @Override
    void move() {
        System.out.println("몬스터가 배회합니다");
    }
    
    @Override
    void attack() {
        System.out.println("몬스터가 물어뜯습니다: " + attackPower + " 데미지");
    }
    
    @Override
    void die() {
        System.out.println("몬스터가 죽었습니다");
    }
}

public class Boss extends Character {
    private int phase = 1;
    
    @Override
    void move() {
        System.out.println("보스가 날아다닙니다");
    }
    
    @Override
    void attack() {
        System.out.println("보스가 필살기를 사용합니다: " + (attackPower * phase) + " 데미지");
    }
    
    @Override
    void die() {
        System.out.println("보스가 격렬하게 폭발합니다");
    }
}

// 3️⃣ 다형성으로 모든 캐릭터를 동일하게 처리
public class GameEngine {
    private List<Character> characters = new ArrayList<>();
    
    public void update() {
        for (Character character : characters) {
            character.move();      // 각 캐릭터의 move() 실행
            character.attack();    // 각 캐릭터의 attack() 실행
        }
    }
    
    // 새로운 캐릭터 타입이 추가되어도 이 코드는 변경 안 됨!
    public void addCharacter(Character character) {
        characters.add(character);
    }
}

// 4️⃣ 사용 예시
Character player = new Player();
Character monster = new Monster();
Character boss = new Boss();

GameEngine engine = new GameEngine();
engine.addCharacter(player);
engine.addCharacter(monster);
engine.addCharacter(boss);

engine.update(); // 모든 캐릭터의 move(), attack() 자동 실행
```

## 핵심 개념 정리

### 🏗️ 상속(Inheritance)
- **정의**: 부모 클래스의 속성과 메서드를 자식 클래스가 물려받음
- **장점**:
  - 코드 중복 제거
  - 공통 기능을 한 곳에서 관리
  - 유지보수성 향상
- **예시**: `Monster extends Character`
  - Monster는 Character의 모든 공통 기능을 자동으로 상속

### 🎭 다형성(Polymorphism)
- **정의**: 같은 메서드명이라도 객체 타입에 따라 다른 동작 수행
- **메커니즘**:
  ```java
  Character ch = new Boss();  // 실제 타입: Boss, 참조 타입: Character
  ch.attack();                // Boss의 attack() 실행
  ```
- **이점**:
  - 객체의 실제 타입을 모르고도 호출 가능
  - 새 타입 추가 시 기존 코드 수정 불필요 (OCP: Open-Closed Principle)

### 🎯 메서드 오버라이딩(Overriding)
- **정의**: 부모 클래스의 메서드를 자식 클래스에서 재정의
- **규칙**:
  - 메서드명, 파라미터 타입/개수, 반환 타입이 동일해야 함
  - 접근 권한은 같거나 더 넓어야 함 (`public` > `protected` > `default` > `private`)

### 📋 추상 클래스(Abstract Class)
- **용도**: 자식 클래스들이 반드시 구현해야 할 메서드 강제
- **특징**:
  ```java
  abstract class Character {
      abstract void attack();  // 구현 없음, 반드시 오버라이드 필요
  }
  ```
- **vs 인터페이스**: 공통 상태(hp, attackPower)가 있으면 추상 클래스 사용

## 실무 포인트

### ✅ 적용 사례
```java
// ❌ 나쁜 예: 중복 코드
public class PlayerManager {
    void updatePlayer(Player p) { p.move(); p.attack(); }
}

public class MonsterManager {
    void updateMonster(Monster m) { m.move(); m.attack(); }
}

// ✅ 좋은 예: 다형성으로 통일
public class GameManager {
    void updateAllCharacters(List<Character> characters) {
        for (Character c : characters) {
            c.move();
            c.attack();
        }
    }
}
```

### 🔧 버그 수정 사례
```java
// takeDamage() 로직 수정 필요 → 1곳만 수정!
public abstract class Character {
    public final void takeDamage(int damage) {
        // 방어력 계산 로직 추가
        int actualDamage = Math.max(damage - defense, 1);
        hp -= actualDamage;
        if (hp <= 0) die();
    }
}
// → 모든 50개 캐릭터에 자동 적용!
```

### 📈 확장성
```java
// 새 캐릭터 추가 시
public class NPC extends Character {
    @Override
    void move() { /* NPC 이동 로직 */ }
    
    @Override
    void attack() { /* NPC는 공격 안 함 */ }
    
    @Override
    void die() { /* NPC는 죽지 않음 */ }
}

// GameManager 코드는 수정 안 함! (SOLID 원칙)
```

### ⚠️ 주의사항

| 항목 | 주의점 |
|------|--------|
| **과도한 상속** | 깊이 3단계 이상 → 복잡도 증가, 인터페이스 고려 |
| **깨지기 쉬운 부모(Fragile Base Class)** | 부모 수정 시 모든 자식 영향 → 신중히 설계 |
| **리스코프 치환 원칙(LSP)** | 자식은 부모 역할을 완벽히 수행해야 함 |
| **Interface vs Abstract Class** | 다중 구현 필요 → Interface 사용 |
