# SOLID 원칙 - LSP, ISP, DIP 모범 답안

## 모범 답안

### 문제 분석
현재 설계의 핵심 문제는 **신용카드를 기준으로 추상화**했다는 것입니다. 이는 LSP(리스코프 치환 원칙) 위반이며, 각 결제 수단의 고유한 특성을 무시한 설계입니다.

### 올바른 설계 접근

**1단계: 공통 인터페이스 식별 (ISP 적용)**
```python
# ❌ 잘못된 설계: 신용카드 기준
class PaymentProcessor(ABC):
    @abstractmethod
    def charge(self, amount: int) -> bool:
        """신용카드 방식의 결제"""
        pass
    
    @abstractmethod
    def refund(self, amount: int) -> bool:
        """신용카드 방식의 환불"""
        pass

# ✅ 올바른 설계: 공통 책임만 추상화
class Payment(ABC):
    """모든 결제 수단의 최소 공통 책임"""
    @abstractmethod
    def process(self, amount: int) -> PaymentResult:
        pass
```

**2단계: 결제 수단별 인터페이스 분리 (ISP)**
```python
from abc import ABC, abstractmethod
from typing import Protocol
from dataclasses import dataclass
from enum import Enum

@dataclass
class PaymentResult:
    success: bool
    transaction_id: str
    message: str

# 기본 인터페이스 (모든 결제 수단이 구현)
class Payment(ABC):
    @abstractmethod
    def process(self, amount: int) -> PaymentResult:
        """결제 처리"""
        pass

# 선택적 인터페이스들 (필요한 수단만 구현)
class Refundable(ABC):
    @abstractmethod
    def refund(self, transaction_id: str, amount: int) -> PaymentResult:
        pass

class PreAuthRequired(ABC):
    @abstractmethod
    def pre_authenticate(self, user_info: dict) -> bool:
        pass

class AutoCancelable(ABC):
    @abstractmethod
    def schedule_auto_cancel(self, transaction_id: str, hours: int) -> bool:
        pass

# 신용카드: 기본 + 환불만 가능
class CreditCardPayment(Payment, Refundable):
    def process(self, amount: int) -> PaymentResult:
        # 신용카드 고유 로직
        transaction_id = self._charge_card(amount)
        return PaymentResult(True, transaction_id, "승인됨")
    
    def refund(self, transaction_id: str, amount: int) -> PaymentResult:
        return self._process_refund(transaction_id, amount)
    
    def _charge_card(self, amount: int) -> str:
        # 실제 결제 로직
        pass

# 계좌이체: 기본 + 사전인증 필수
class BankTransfer(Payment, PreAuthRequired):
    def process(self, amount: int) -> PaymentResult:
        # 인증 확인
        if not self.is_authenticated:
            return PaymentResult(False, "", "사전 인증 필요")
        
        transaction_id = self._transfer(amount)
        return PaymentResult(True, transaction_id, "송금됨")
    
    def pre_authenticate(self, user_info: dict) -> bool:
        # 심사 로직
        self.is_authenticated = self._verify_user(user_info)
        return self.is_authenticated

# 휴대폰결제: 기본 + 자동취소 기능
class MobilePayment(Payment, AutoCancelable):
    def process(self, amount: int) -> PaymentResult:
        transaction_id = self._charge_mobile(amount)
        
        # 자동 취소 스케줄링
        self.schedule_auto_cancel(transaction_id, hours=24)
        
        return PaymentResult(True, transaction_id, "결제됨 (24시간 후 취소)")
    
    def schedule_auto_cancel(self, transaction_id: str, hours: int) -> bool:
        # 스케줄러에 등록
        return self._register_cancel_job(transaction_id, hours)
```

**3단계: 의존성 역전 (DIP) - 클라이언트 코드**
```python
class PaymentService:
    """의존성을 추상화에만 의존"""
    
    def __init__(self, payment_processor: Payment):
        self.payment = payment_processor
    
    def execute_payment(self, amount: int) -> PaymentResult:
        # 결제 수단이 무엇인지 알 필요 없음
        result = self.payment.process(amount)
        return result
    
    def execute_refund(self, transaction_id: str, amount: int) -> PaymentResult:
        # Refundable 인터페이스 체크
        if isinstance(self.payment, Refundable):
            return self.payment.refund(transaction_id, amount)
        return PaymentResult(False, "", "환불 불가능한 결제 수단")
    
    def verify_payment_method(self, payment: Payment) -> bool:
        # PreAuthRequired 체크
        if isinstance(payment, PreAuthRequired):
            return payment.pre_authenticate({"user_id": "123"})
        return True

# 사용 예
credit_card = CreditCardPayment()
service = PaymentService(credit_card)
result = service.execute_payment(10000)

bank = BankTransfer()
service = PaymentService(bank)
if service.verify_payment_method(bank):
    result = service.execute_payment(10000)
```

**4단계: 팩토리 패턴으로 생성 관리 (DIP 강화)**
```python
class PaymentFactory:
    """구체적인 클래스 생성은 한곳에서만"""
    
    @staticmethod
    def create_payment(method: str) -> Payment:
        if method == "credit_card":
            return CreditCardPayment()
        elif method == "bank_transfer":
            return BankTransfer()
        elif method == "mobile":
            return MobilePayment()
        else:
            raise ValueError(f"Unknown payment method: {method}")

# 클라이언트는 생성 방식을 모름
payment = PaymentFactory.create_payment("credit_card")
service = PaymentService(payment)
```

## 핵심 개념 정리

### 📍 LSP (Liskov Substitution Principle)
- **정의**: 부모 타입의 객체를 자식 타입으로 대체할 수 있어야 함
- **이 문제에서의 위반**: 신용카드를 기준으로 설계 → 다른 결제 수단이 신용카드처럼 동작해야 함
- **해결방법**: 공통의 최소 책임(`Payment`)을 기준으로 추상화

### 📍 ISP (Interface Segregation Principle)
- **정의**: 클라이언트는 자신이 필요하지 않은 인터페이스에 의존하면 안 됨
- **이 문제에서의 위반**: 모든 결제 수단이 `refund()`, `pre_auth()`, `auto_cancel()` 모두 구현하도록 강요
- **해결방법**: 
  - 기본 인터페이스: `Payment` (모두 필수)
  - 선택적 인터페이스: `Refundable`, `PreAuthRequired`, `AutoCancelable`
  - 필요한 것만 구현

### 📍 DIP (Dependency Inversion Principle)
- **정의**: 고수준 모듈은 저수준 모듈에 의존하면 안 됨. 둘 다 추상화에만 의존해야 함
- **이 문제에서의 위반**: `PaymentService`가 구체적인 결제 수단 클래스에 직접 의존
- **해결방법**:
  - 구체적 클래스가 아닌 추상 인터페이스(`Payment`)에만 의존
  - 클라이언트는 `PaymentFactory`를 통해 간접적으로만 의존
