## 📚 모범 답안 & 개념 정리

### 1. 🎤 면접 질문

# IoC와 DI — 모범 답안 및 개념 정리

## 모범 답안

### 비유 분석

**직접 구매하는 방식 (DI 없음)**
- 피자 가게 사장이 직접 밀가루, 토마토소스, 치즈를 매번 사러 다님
- 코드로는: `class PizzaShop { private Cheese cheese = new Cheese(); }`
- 문제: 번거롭고, 공급업체가 바뀌면 직접 모든 것을 수정해야 함

**공급업체와 계약하는 방식 (DI)**
- 피자 가게는 "필요한 것을 주문하면 배달해달라"고만 계약
- 공급업체 관리는 외부 중개인(Container)이 담당
- 코드로는: `class PizzaShop { constructor(Cheese cheese) { this.cheese = cheese; } }`
- 이점: 피자 가게는 피자만 만드는 데 집중, 공급 관리는 별도

---

### IoC와 DI의 핵심

**IoC (Inversion of Control)**
- 제어권이 뒤바뀐다는 의미
- 일반적: 내 코드가 필요한 객체를 **직접** 생성하고 관리 (주도권이 개발자)
- IoC 적용: 프레임워크/컨테이너가 객체 생성과 주입을 담당 (주도권이 프레임워크)

**DI (Dependency Injection)**
- IoC를 구현하는 구체적인 패턴
- 필요한 의존성을 **외부에서 주입**받는 방식
- 종류: Constructor Injection, Setter Injection, Interface Injection

---

### 핵심 이유 3가지

1. **낮은 결합도 (Low Coupling)**
   - PizzaShop이 Cheese 구현체에 의존하지 않음
   - Cheese 인터페이스에만 의존 (추상화)

2. **높은 확장성 (High Cohesion)**
   - 치즈 공급업체 변경 = 설정만 변경, 코드 수정 없음

3. **테스트 용이성**
   - Mock 치즈를 주입해서 독립적으로 테스트 가능

---

### 꼬리 질문 답변: 치즈 공급업체 변경 시

**DI 없는 방식 (직접 구매)**
```java
class PizzaShop {
    private Cheese cheese = new KoreanCheese(); // 직접 생성
    
    public void makePizza() {
        cheese.slice();
    }
}

// 이탈리안 치즈로 바꾸려면? → 모든 PizzaShop 클래스를 수정!
// 100곳의 피자 가게가 있으면 100곳 모두 수정 필요
```

**DI 방식 (공급업체 계약)**
```java
// PizzaShop.java (코드 수정 없음!)
class PizzaShop {
    private Cheese cheese; // 인터페이스에만 의존
    
    public PizzaShop(Cheese cheese) {
        this.cheese = cheese; // 주입받음
    }
}

// Spring 설정에서만 변경
@Configuration
public class AppConfig {
    @Bean
    public Cheese cheese() {
        return new ItalianCheese(); // 변경 완료!
    }
}
```

**결과**
- DI 없음: 피자 가게 코드 전체 수정 필요 → **높은 위험도**
- DI: 설정 파일만 수정 → **0% 코드 변경**

## 핵심 개념 정리

### 🎯 객체지향 설계 원칙
- **DIP (Dependency Inversion Principle)**
  - 구체적 구현이 아닌 추상화에 의존
  - 고수준 모듈이 저수준 모듈에 의존하지 않음

- **SRP (Single Responsibility Principle)**
  - PizzaShop: 피자 만드는 책임만
  - Container: 객체 생성 책임만

### 🔄 DI의 3가지 패턴
| 패턴 | 예시 | 장점 | 단점 |
|------|------|------|------|
| **Constructor Injection** | `new PizzaShop(cheese)` | 필수 의존성 명확, 불변 | 파라미터 많으면 복잡 |
| **Setter Injection** | `shop.setCheese(cheese)` | 선택적 의존성 가능 | 순서 의존, 보안 약함 |
| **Interface Injection** | `cheese` implements `CheeseProvider` | 세밀한 제어 | 복잡도 높음 |

### 📌 IoC 컨테이너가 하는 일
```
객체 생성 → 의존성 분석 → 주입 순서 결정 → 주입 실행
```

### ⚠️ 반대 개념: 강한 결합 (Tight Coupling)
```java
// 안 좋은 예
class PizzaShop {
    private KoreanCheese cheese = new KoreanCheese();
}
// 문제: KoreanCheese 변경 → PizzaShop 수정 필요
```

## 실무 포인트

### ✅ DI를 써야 하는 경우
1. **변경 가능성이 있는 의존성**
   - 데이터베이스 변경 (MySQL → PostgreSQL)
   - 결제 API 변경 (PG사 변경)
   - 메시지 큐 변경 (RabbitMQ → Kafka)

2. **테스트가 필요한 부분**
   ```java
   @Test
   void testPizzaMaking() {
       Cheese mockCheese = mock(Cheese.class);
       PizzaShop shop = new PizzaShop(mockCheese); // DI로 Mock 주입
       shop.makePizza();
       verify(mockCheese).slice(); // 검증
   }
   ```

3. **마이크로서비스 환경**
   - 서비스마다 다른 설정 필요
   - 환경별(Dev/Prod) 다른 구현체 필요

### ❌ DI를 과도하게 쓰면 안 되는 경우
```java
// 과도한 예
class Logger {
    private DateProvider dateProvider; // 불필요한 DI
    private StringUtil stringUtil;      // 불필요한 DI
}

// Logger는 stateless 유틸이므로 DI 불필요
// 유틸/헬퍼 클래스는 직접 사용해도 됨
```

### 🏢 Spring Framework에서의 실제 사용
```java
@Service
public class OrderService {
    private final PaymentGateway paymentGateway; // DI
    
    // Constructor Injection (Spring 권장)
    public OrderService(PaymentGateway paymentGateway) {
        this.paymentGateway = paymentGateway;
    }
}

// application.yml에서 구현체 결정
payment:
  provider: stripe  # 또는 pg, nicepay 등 설정만 변경
```

### 🎬 면접에서 추가로 언급하면 좋은 포인트
1. **자신의 경험 공유**
   - "저희 팀에서도 결제 모듈을 DI로 분리해서, PG사 변경 시 설정만 수정하고 비즈니스 로직은 건드리지 않았습니다"

2. **트레이드오프 인식**
   - "과도한 DI는 코드 복잡도를 높이므로, 변경 가능성이 실제 있을 때만 적용합니다"

3. **실무 관점**
   - "DI는 단순히 테스트만을 위한 게 아니라, 유지보수성과 확장성 측면에서 큰 이점입니다"
