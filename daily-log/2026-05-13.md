# JVM 메모리 구조와 분리 설계의 필요성

## 모범 답안

**JVM이 메모리를 분리하는 이유는 "접근 패턴과 생명주기가 다르기 때문"입니다.**

### 1️⃣ 왜 분리되어 있는가?

```
┌─────────────────────────────────┐
│        JVM 메모리 구조          │
├─────────────────────────────────┤
│  Heap (동적 할당, GC 대상)      │
│  - 객체 인스턴스 저장           │
│  - 스레드 간 공유               │
│  - 생명주기: 불명확             │
├─────────────────────────────────┤
│  Stack (스레드마다 독립)        │
│  - 메서드 호출 스택             │
│  - 지역변수, 참조값             │
│  - 생명주기: 메서드 종료 시 자동 해제 │
├─────────────────────────────────┤
│  Method Area (클래스 로딩)      │
│  - 클래스 정보, 메서드 코드     │
│  - 상수 풀, 필드 정보           │
│  - 생명주기: 클래스 언로드 시까지│
└─────────────────────────────────┘
```

    // 혼합 시 복잡도 폭증

### 3️⃣ 실제 OutOfMemoryError 분석

```java
public class OOMAnalysis {
    
    // Case 1: HeapOutOfMemory (가장 일반적)
    // 증상: 객체가 계속 생성되고 GC로도 해제 안 됨
    public void heapOOM() {
        List<byte[]> list = new ArrayList<>();
        while(true) {
            list.add(new byte[1024 * 1024]); // 1MB씩 계속 추가
            // → 메모리가 찬다 → GC 실행 → 참조가 남아있어서 해제 못함
        }
    }
    
    // Case 2: StackOverflowError (재귀 호출)
    // 증상: Stack 공간이 부족 (보통 JVM 설정으로 크기 고정)
    public void stackOOM() {
        stackOOM(); // 무한 재귀 → 스택 프레임 계속 쌓임
        // Stack은 동적 할당 불가 → 즉시 에러 발생
    }
    
    // Case 3: PermGen/Metaspace OOM (클래스 언로드 못함)
    // 증상: 너무 많은 클래스 로딩 또는 언로드 실패
    public void methodAreaOOM() {
        // 동적 클래스 생성 (프록시, CGLIB 등)
        for(int i = 0; i < 1000000; i++) {
            Class<?> clazz = new ClassLoader() {
                // 계속 새로운 클래스 로드
            }.loadClass("Dynamic" + i);
        }
    }
}
```

## 핵심 개념 정리

### 📌 **Heap (힙)**
- **용도**: 동적으로 생성되는 모든 객체 저장
- **특징**:
  - 스레드 간 공유
  - GC의 대상
  - 크기 동적 할당 가능
  - 접근 속도 상대적으로 느림
- **크기 조정**: `-Xms`, `-Xmx` (초기/최대 크기)
- **OOM 원인**: 메모리 누수, 과도한 객체 생성

### 📌 **Stack (스택)**
- **용도**: 메서드 호출, 지역변수, 참조값(주소) 저장
- **특징**:
  - 스레드마다 독립적
  - LIFO(후입선출) 방식
  - 자동 메모리 해제
  - 고정 크기 (동적 할당 불가)
  - 접근 속도 빠름
- **크기 조정**: `-Xss` (스레드별 스택 크기)
- **OOM 원인**: 재귀 호출 깊이, 많은 지역변수

### 📌 **Method Area (메서드 영역)**
- **용도**: 클래스 구조, 메서드 코드, 상수, 필드 정보 저장
- **특징**:
  - 모든 스레드 공유
  - 클래스 로딩 시 할당
  - 클래스 언로드 시 해제
  - Java 8+에서는 Metaspace (Native Memory)
- **OOM 원인**: 동적 클래스 생성, 클래스로더 메모리 누수

### 📌 **분리의 이점**

| 측면 | 효과 |
|------|------|
| **성능** | Stack은 빠른 접근, Heap은 GC로 최적화 |
| **메모리 안정성** | Stack 오버플로우가 Heap에 영향 없음 |
| **동시성** | 스레드별 Stack으로 동기화 비용 감소 |
| **생명주기 관리** | 각 영역에 맞는 해제 전략 적용 |
