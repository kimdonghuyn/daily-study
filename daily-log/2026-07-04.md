## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 상속과 다형성 - 모범 답안

## 모범 답안

현재 상황은 **Open/Closed Principle 위반**입니다. 새로운 기능(캐릭터)이 추가될 때마다 기존 코드를 수정해야 하기 때문입니다.

**해결책: 상속과 다형성을 활용한 구조 설계**

```java
// 1단계: 추상 기본 클래스 정의
abstract class Character {
    protected String name;
    
    // 구현을 강제하는 추상 메서드
    abstract void attack();
    
    // 공통 로직
    void takeDamage(int damage) {
        System.out.println(name + "이 " + damage + "의 피해를 입었습니다");
    }
}

// 2단계: 각 캐릭터 타입별 구현
class Knight extends Character {
    Knight(String name) { this.name = name; }
    
    @Override
    void attack() {
        System.out.println(name + "이 칼로 베었습니다!");
    }
}

class Wizard extends Character {
    Wizard(String name) { this.name = name; }
    
    @Override
    void attack() {
        System.out.println(name + "이 마법을 시전했습니다!");
    }
}

class Archer extends Character {
    Archer(String name) { this.name = name; }
    
    @Override
    void attack() {
        System.out.println(name + "이 화살을 쏘았습니다!");
    }
}

// 3단계: 타입 구분 없이 사용
class BattleSystem {
    private List<Character> characters = new ArrayList<>();
    
    void addCharacter(Character character) {
        characters.add(character);
    }
    
    // 핵심: if문이 없음!
    void executeAllAttacks() {
        for (Character character : characters) {
            character.attack();  // 다형성: 실제 타입에 따라 다른 동작
        }
    }
}

// 사용 예
public class Game {
    public static void main(String[] args) {
        BattleSystem battle = new BattleSystem();
        
        battle.addCharacter(new Knight("아서"));
        battle.addCharacter(new Wizard("간달프"));
        battle.addCharacter(new Archer("레골라스"));
        battle.addCharacter(new Knight("랜슬롯")); // 새로 추가되어도 기존 코드 수정 X
        
        battle.executeAllAttacks();
        // 출력:
        // 아서이 칼로 베었습니다!
        // 간달프이 마법을 시전했습니다!
        // 레골라스이 화살을 쏘았습니다!
        // 랜슬롯이 칼로 베었습니다!
    }
}
```

**Before (나쁜 예)**
```java
void attack(String characterType) {
    if (characterType.equals("knight")) {

    } else if (characterType.equals("archer")) {

    }
    // 캐릭터 200개 = if문 200개... 악몽!
```

## 핵심 개념 정리

### 📌 상속 (Inheritance)
- **목적**: 공통적인 속성과 메서드를 부모 클래스에서 관리
- **역할**: `Character` 같은 기본 클래스에서 공통 기능(`takeDamage`) 정의
- **추상 클래스**: 구현을 강제하되 세부 내용은 자식에게 맡김

### 📌 다형성 (Polymorphism)
- **핵심**: 같은 메서드 이름인데 다른 동작
- **메커니즘**: `Character` 참조로 실제 타입(`Knight`, `Wizard` 등)에 맞는 `attack()` 호출
- **이점**: 새 타입 추가 시 기존 코드(`executeAllAttacks()`) 변경 불필요

### 📌 메서드 오버라이딩 (Method Overriding)
- 부모 추상 메서드를 자식 클래스에서 구체적으로 구현
- `@Override` 애노테이션으로 명시적 표시

### 📌 SOLID 원칙과의 관계
- **Open/Closed Principle**: 확장에 열려있고 수정에 닫혀있음
- **Liskov Substitution Principle**: `Character` 타입이면 어떤 캐릭터든 사용 가능

## 실무 포인트

### 🎯 즉시 적용 가능한 패턴

**1) 스트래티지 패턴과 결합**
```java
interface AttackStrategy {
    void execute();
}

class Character {
    private AttackStrategy strategy;
    
    void attack() {
        strategy.execute();  // 런타임에 동적 변경 가능
    }
    
    void setStrategy(AttackStrategy newStrategy) {
        this.strategy = newStrategy;  // 같은 캐릭터도 다른 공격 가능!
    }
}
```

**2) 팩토리 패턴으로 생성 간소화**
```java
class CharacterFactory {
    static Character create(String type, String name) {
        return switch(type) {
            case "knight" -> new Knight(name);
            case "wizard" -> new Wizard(name);
            case "archer" -> new Archer(name);
            default -> throw new IllegalArgumentException();
        };
        // if문을 한곳에만 모아서 관리
    }
}
```

### 🔴 흔한 실수

| 문제 | 해결책 |
|------|--------|
| 모든 캐릭터를 `Object`로 관리 | `Character` 같은 공통 기본 타입 정의 |
| 부모 클래스에 모든 구현을 다 넣기 | 추상 메서드로 자식에게 구현 위임 |
| instanceof로 타입 체크 후 캐스팅 | 다형성 사용 (타입 체크 필요 없음) |

### 📊 성능/유지보수 비교

```
200개 캐릭터 추가 시 비용:

❌ if문 방식:
   - 기존 코드 수정: 1회
   - 테스트 회귀: 심각
   - 버그 가능성: 매우 높음
   
✅ 상속/다형성:
   - 기존 코드 수정: 0회
   - 테스트 회귀: 없음
   - 새 클래스만 작성: 자동으로 작동
```

### 🏢 대규모 프로젝트에서

Spring Framework를 사용하면:

```java
@Component
class Knight extends Character { ... }

@Component  
class Wizard extends Character { ... }

// 자동으로 모든 Character 구현체를 주입
@Autowired
List<Character> allCharacters;  // 200개 클래스도 자동 수집!
```

**결론**: 처음부터 추상화를 통해 확장성을 확보하면, 나중에 100배 많은 코드를 추가해도 기존 비즈니스 로직은 건드릴 필요가 없습니다. 이것이 프로페셔널 개발자의 역할입니다.
