## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# 상속과 다형성을 통한 게임 캐릭터 시스템 설계

## 모범 답안

### 문제의 본질
현재 코드는 **타입 확인 후 분기 처리(Type Checking + Switch/If)**를 반복하고 있습니다. 이는 새로운 캐릭터가 추가될 때마다 기존 코드를 수정해야 하는 **Open/Closed Principle 위반** 상태입니다.

### 해결책: 상속과 다형성

```java
// 1. 부모 클래스(추상화): 공통 인터페이스 정의
public abstract class Character {
    protected String name;
    protected int hp;
    protected int attackPower;
    
    // 자식 클래스마다 다르게 구현되어야 하는 메서드
    public abstract void attack(Enemy enemy);
    
    // 공통 기능
    public void takeDamage(int damage) {
        this.hp -= damage;
    }
}

// 2. 자식 클래스들: 각자의 공격 방식 구현
public class Swordsman extends Character {
    @Override
    public void attack(Enemy enemy) {
        System.out.println(name + "이(가) 검으로 베기를 시전합니다!");
        int damage = attackPower + 10; // 검사 보너스
        enemy.takeDamage(damage);
    }
}

public class Mage extends Character {
    private int mana;
    
    @Override
    public void attack(Enemy enemy) {
        if (mana >= 20) {
            System.out.println(name + "이(가) 파이어볼을 시전합니다!");
            int damage = attackPower * 2; // 마법사는 높은 공격력
            enemy.takeDamage(damage);
            mana -= 20;
        }
    }
}

public class Archer extends Character {
    @Override
    public void attack(Enemy enemy) {
        System.out.println(name + "이(가) 화살을 날립니다!");
        int damage = attackPower;
        enemy.takeDamage(damage);
    }
}

// 3. 게임 로직: 타입 확인 없이 다형성 활용
public class GameBattle {
    public void executeRound(List<Character> characters, Enemy enemy) {
        for (Character character : characters) {
            // 어떤 캐릭터든 상관없이 attack() 호출
            // 각 캐릭터의 실제 타입에 따라 적절한 attack() 실행됨
            character.attack(enemy);
        }
    }
}

// 4. 사용 예시
public class Main {
    public static void main(String[] args) {
        List<Character> party = new ArrayList<>();
        party.add(new Swordsman("아서", 100, 15));
        party.add(new Mage("멀린", 80, 20));
        party.add(new Archer("로빈", 90, 12));
        
        Enemy goblin = new Enemy("고블린", 50);
        
        GameBattle battle = new GameBattle();
        battle.executeRound(party, goblin);
        
        // 출력:
        // 아서이(가) 검으로 베기를 시전합니다!
        // 멀린이(가) 파이어볼을 시전합니다!
        // 로빈이(가) 화살을 날립니다!
    }
}
```

### 개선 효과

| 항목 | 문제 상황 | 개선 후 |
|------|---------|--------|
| **코드 반복** | 매번 타입 확인 필요 | 한 번의 `attack()` 호출 |
| **새 캐릭터 추가** | 기존 코드 수정 필요 | 새 클래스만 추가 (기존 코드 무변경) |
| **유지보수** | 모든 분기 로직 관리 | 각 클래스가 자신의 로직만 관리 |
| **확장성** | 낮음 (스파게티 코드) | 높음 (SOLID 원칙 준수) |

## 핵심 개념 정리

### 🏗️ **상속 (Inheritance)**
- **정의**: 부모 클래스의 속성과 메서드를 자식 클래스가 물려받는 것
- **목적**: 코드 재사용 및 계층적 관계 표현
- **이 예시에서**: `Character`의 `hp`, `name`, `takeDamage()` 모든 캐릭터가 공유

### 🎭 **다형성 (Polymorphism)**
- **정의**: 같은 메서드 호출이 객체의 실제 타입에 따라 다르게 동작
- **핵심**: 컴파일 타임에 타입을 몰라도 런타임에 올바른 메서드가 실행됨
  ```java
  Character character = new Mage(...); // 부모 타입으로 선언
  character.attack(enemy); // Mage의 attack() 실행됨
  ```
- **장점**: 
  - Type casting이나 instanceof 체크 불필요
  - 새로운 타입 추가에 유연함

### 📋 **메서드 오버라이딩 (Overriding)**
- **정의**: 자식 클래스가 부모 클래스의 메서드를 재구현
- **`@Override` 어노테이션**: 컴파일 타임에 올바른 오버라이딩인지 검증

### 🔐 **추상 클래스 vs 인터페이스**

```java
// 추상 클래스: "이것은 캐릭터다" (IS-A 관계)
public abstract class Character {
    abstract void attack(Enemy enemy); // 반드시 구현해야 함
}

// 인터페이스: "공격 가능한 것이다" (CAN-DO 관계)
public interface Attackable {
    void attack(Enemy enemy);
}

// 선택 기준
// - 추상 클래스: 강한 계층 구조, 상태(hp, name)를 공유
// - 인터페이스: 느슨한 계약, 다중 구현 필요 시
```

## 실무 포인트

### ✅ **백엔드 시니어로서 주의할 점**

#### 1. **SOLID 원칙 준수**
```java
// ❌ 나쁜 예: 타입 확인 후 처리
public void executeAttack(Character character, Enemy enemy) {
    if (character instanceof Swordsman) {
        // 검사 로직
    } else if (character instanceof Mage) {
        // 마법사 로직
    }
    // 새로운 캐릭터 추가 시 이 함수를 수정해야 함 (Open/Closed Principle 위반)
}

// ✅ 좋은 예: 다형성 활용
public void executeAttack(Character character, Enemy enemy) {
    character.attack(enemy); // 구체적인 타입 몰라도 됨
    // 새로운 캐릭터 추가 시 이 함수는 수정 불필요
}
```

#### 2. **인터페이스 분리**
```java
// 자식 클래스별로 필요한 기능이 다를 수 있음
public interface Caster {
    void castSpell(String spellName);
}

public class Mage extends Character implements Caster {
    @Override
    public void castSpell(String spellName) {
        // 마법사만 주문 시전 가능
    }
}

// 클라이언트는 필요한 인터페이스만 의존
public void executeSpell(Caster caster) {
    caster.castSpell("파이어볼");
}
```
