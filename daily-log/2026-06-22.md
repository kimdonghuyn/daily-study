## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# JVM 메모리 구조 - 모범 답안

## 모범 답안

### 1단계: 왜 메모리를 나눴는가?

**핵심 이유는 "데이터의 생명주기와 접근 패턴이 다르기 때문"입니다.**

```
🏠 메모리를 방으로 나눈 이유
├─ Stack: 빠르게 생성/소멸되는 임시 데이터 (함수 호출 정보)
├─ Heap: 오래 살아있는 공유 데이터 (객체)
└─ Method Area: 변하지 않는 청사진 (클래스 정보)
```

**비유로 이해하기:**
- **Stack**: 일회용 접시 (쓰고 즉시 버림, 빠르고 효율적)
- **Heap**: 공용 칠판 (여러 사람이 공유, 관리 복잡하지만 유연함)
- **Method Area**: 건축 설계도 (바뀌지 않음, 모두가 참고)

---

### 2단계: 각 영역이 나뉜 구체적 이유

#### **Stack 독립 운영**
```java
public void userLogin(String username) {  // <- 이 함수 호출마다
    String tempToken = generateToken();   // <- 이 변수들이 생성되었다가
    int sessionId = 12345;               // <- 함수 끝나면 즉시 버려짐
    userService.authenticate(tempToken); // <- 메모리 낭비 없음
}
// 함수 끝 → Stack에서 자동 청소 ✨
```

**왜 Stack을 따로 만들었나?**
1. **자동 메모리 관리**: 함수 종료 = 자동 정리 (GC 필요 없음)
2. **극도의 속도**: 메모리 할당이 단순 포인터 이동 (O(1) 시간)
3. **예측 가능성**: 크기가 고정되어 있어 보안/안정성 ↑

#### **Heap 공동 운영**
```java
public class UserService {
    User user = new User("john");  // <- Heap에 생성
    user.setAge(25);               // <- 다른 곳에서도 접근 가능
    // user 변수는 Stack에서 사라져도
    // 객체 자체는 Heap에서 계속 존재
}

// 누군가 아직 참조하고 있으면 → 살아있음
// 아무도 참조 안 하면 → GC가 나중에 청소
```

**왜 Heap을 따로 만들었나?**
1. **유연한 크기**: 동적으로 할당/해제 가능
2. **공유 가능**: 여러 함수/쓰레드가 동일 객체 접근
3. **수명 관리**: 개발자 의도대로 객체 생명주기 조절

#### **Method Area (Class 정보)**
```java
public class Order {  // <- 클래스 정의는 Method Area에
    private String orderId;  // <- 필드 정보
    public void process() {} // <- 메서드 정보
}

// 이 정보는:
// - 프로그램 시작부터 종료까지 변하지 않음
// - 모든 Order 인스턴스가 공유함
// - Heap의 객체들이 참조함
```

**왜 따로 만들었나?**
1. **중복 제거**: 같은 클래스 100개 생성 → 정보는 1번만 저장
2. **참조 효율**: 모든 인스턴스가 가리킬 수 있음
3. **불변성**: 런타임 중 변경 안 됨

---

### 3단계: OutOfMemoryError가 발생하는 과정

```
사용자 1명: Stack(작음) + Heap(작음) → OK ✓

사용자 100명:
for (int i = 0; i < 100; i++) {
    User user = new User();        // <- 100개 객체가 Heap에!
    // user.logout() 호출 안 함    // <- 정리 안 됨
    // → Heap 메모리 계속 증가
}
// Heap 한계 초과 → OutOfMemoryError ❌

⚠️ Stack은 문제 없음 (함수 호출 끝나면 자동 정리)
⚠️ Heap이 문제 (객체가 쌓여만 감)
```

## 핵심 개념 정리

### 📌 **Stack (스택)**

| 항목 | 설명 |
|------|------|
| **저장 데이터** | 함수 호출, 지역 변수, 메서드 참조, 원시 타입 값 |
| **크기** | 고정 (기본 1MB ~ 1GB) |
| **할당 방식** | LIFO (Last In First Out) - 자동 |
| **관리** | JVM이 자동 (함수 리턴 시 초기화) |
| **속도** | 매우 빠름 ⚡ |
| **수명** | 매우 짧음 (함수 실행 중만) |
| **쓰레드** | 쓰레드별 독립 할당 |

**Stack에 저장되는 것:**
```java
public void example() {
    int age = 25;              // ← Stack: 원시 타입
    String name = "John";      // ← Stack: 참조(주소만)
    User user = new User();    // ← Stack: 참조(주소만)
}
// 함수 종료 → age, name, user 참조 모두 삭제
```

---

### 📌 **Heap (힙)**

| 항목 | 설명 |
|------|------|
| **저장 데이터** | 객체, 배열 (실제 데이터) |
| **크기** | 동적 (보통 256MB ~ 수GB) |
| **할당 방식** | 동적 할당 |
| **관리** | GC (Garbage Collection)가 자동 정리 |
| **속도** | 상대적으로 느림 |
| **수명** | 길음 (참조가 있는 동안) |
| **쓰레드** | 모든 쓰레드가 공유 |

**Heap에 저장되는 것:**
```java
User user = new User("John", 25);
// Stack:  user → [주소 0x1A2B] ─┐
// Heap:                      └→ [User 객체의 실제 데이터]
```

---

### 📌 **Method Area (메서드 영역)**

| 항목 | 설명 |
|------|------|
| **저장 데이터** | 클래스 구조, 메서드 코드, 상수, static 변수 |
| **크기** | 동적 (기본 64MB) |
| **관리** | 클래스 언로드 시만 해제 |
| **수명** | 프로그램 실행 중 유지 |
| **쓰레드** | 모든 쓰레드가 공유 |

**Method Area에 저장되는 것:**
```java
public class User {              // ← Method Area: 클래스 정보
    private static int count;    // ← Method Area: static 변수
    
    public User(String name) {}  // ← Method Area: 메서드 정보
}
```
