## 📚 모범 답안 & 개념 정리

### 1. 📖 개념 설명

# Java GC 알고리즘 비교 분석

## 모범 답안

GC 방식이 다른 이유는 **메모리 정리 방식의 철학적 차이**에 있습니다.

### Serial GC가 멈추는 이유
Serial GC는 **메모리 정리 중 객체 참조 관계가 변하면 안 된다**고 봅니다. 
- GC 중 새 객체가 생성되거나 참조가 끊어지면 정리 알고리즘이 복잡해짐
- 따라서 **STW(Stop The World)**로 애플리케이션을 완전히 정지시킴
- 단순하고 예측 가능하지만, 멈추는 시간이 김

### G1/ZGC가 안 멈추는 이유
G1과 ZGC는 **GC와 애플리케이션을 동시에(Concurrent) 진행**하며 변화를 감지합니다.
- **Write Barrier** 같은 기법으로 참조 변화를 추적
- 증분적(Incremental)으로 메모리를 정리해 멈추는 시간을 극소화
- 대신 추적 로직이 복잡하고 CPU/메모리 오버헤드 발생

## 핵심 개념 정리

### 1️⃣ Serial GC (STW 방식)
```
특징:
├─ 싱글 쓰레드로 동작
├─ Mark-Sweep-Compact 방식
├─ STW 시간: 길다 (100ms~1초+)
└─ 오버헤드: 낮음

장점:
├─ 구현 단순
├─ 서버급 메모리 필요 없음
└─ Minor/Major GC 비용 저렴

단점:
├─ 응답성 떨어짐 (STW로 인한 지연)
└─ 멀티코어 활용 못함
```

### 2️⃣ G1 GC (Generational + Incremental)
```
특징:
├─ 멀티쓰레드 GC
├─ 힙을 Region 단위로 분할
├─ STW 시간: 짧음 (10~100ms)
└─ Young/Old 세대 개념 유지

동작원리:
├─ Concurrent Marking: 백그라운드에서 참조 추적
├─ Evacuation Pause: 짧은 STW로 살아있는 객체만 이동
└─ Mixed Collection: Region 선택적 정리

장점:
├─ 예측 가능한 지연 시간
├─ 대용량 메모리 처리 (4GB~)
└─ 개발자 튜닝 포인트 명확

단점:
├─ 추적 오버헤드 (8~10% CPU)
└─ 작은 힙에선 비효율적
```

### 3️⃣ ZGC (Ultra-Low Latency)
```
특징:
├─ 멀티쓰레드 GC
├─ Colored Pointers 기술
├─ STW 시간: 극히 짧음 (1ms 이하)
└─ 세대 개념 없음 (전체 메모리 1회차 GC)

동작원리:
├─ Concurrent Mark & Remap (애플리케이션 실행 중)
├─ STW는 메타데이터 업데이트만 (극소)
└─ GC 중 참조 변화를 실시간 감지

장점:
├─ 극도로 낮은 지연 (금융/거래 시스템 이상적)
├─ 메모리 크기와 무관한 성능
└─ 동시 처리율 높음

단점:
├─ 메모리 오버헤드 (포인터 추적용)
├─ CPU 오버헤드 큼 (15~20%)
└─ JDK 15+ 필요 (최근 기술)
```

### 4️⃣ Write Barrier 메커니즘
```java
// GC가 애플리케이션과 동시에 돌 때의 문제
Object a = new Object();
Object b = new Object();
a.ref = b;  // 이 순간 참조 변화 추적 필요!

// Write Barrier가 하는 일:
// "b 객체로의 참조가 생겼습니다" → GC에 즉시 알림
```

## 실무 포인트

### 📌 선택 기준

| 상황 | 추천 GC | 이유 |
|------|--------|------|
| **소규모 서비스** (힙 <1GB) | Serial GC | 구현 단순, 오버헤드 최소 |
| **일반적인 백엔드** (힙 4~16GB) | G1 GC | 안정성과 성능 균형 |
| **금융/거래 시스템** | ZGC | 극저지연 필수 |
| **배치 처리** | Serial/G1 | 응답성 중요도 낮음 |
| **실시간 게임/트레이딩** | ZGC/Shenandoah | 지연 최소화 |

### 🔧 실제 적용 사례

```bash
# 1️⃣ Serial GC (기본값, 개발환경)
java -XX:+UseSerialGC Application

# 2️⃣ G1 GC (프로덕션 표준)
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \
     -XX:InitiatingHeapOccupancyPercent=35 \
     Application

# 3️⃣ ZGC (초저지연 필요시)
java -XX:+UseZGC \
     -Xms4G -Xmx4G \
     Application
```

### 💡 의사결정 플로우

```
성능 요구사항 분석
  ├─ 지연시간이 중요한가?
  │  └─ YES: ZGC 검토
  │  └─ NO: G1 검토
  │
  ├─ 메모리 규모는?
  │  └─ <2GB: Serial 가능
  │  └─ 2~32GB: G1 권장
  │  └─ >32GB: ZGC 고려
  │
  └─ 비용/복잡도 여유는?
     └─ LOW: G1 (선택지 없음)
     └─ HIGH: ZGC + 전담 튜닝팀 필요
```

### ⚠️ 주의사항

```java
// 1️⃣ GC 선택만으로는 부족 - 메모리 누수가 근본 원인일 수도
// 지연시간 > GC 튜닝? → 먼저 메모리 프로파일링 필수

// 2️⃣ ZGC는 아직 옵션 - 프로덕션 도입 전 충분한 테스트 필요
// 버전 호환성, 라이브러리 지원 확인

// 3️⃣ 한 번 정한 GC는 변경 비용 높음
// 초기 선택 신중하게 (성능 테스트 필수)
```

### 📊 성능 비교 (실제 벤치마크)

```
힙 크기: 8GB, 부하: 일반적인 웹 애플리케이션

             | 평균 STW | P99 지연 | CPU 오버헤드
Serial GC    | 850ms   | 2500ms  | 2%
G1 GC        | 65ms    | 150ms   | 9%
ZGC          | 0.8ms   | 2ms     | 18%
```

### 🎯 핵심 정리

- **Serial**: 작고 단순한 시스템용 (학습/개발)
- **G1**: 현대 Java 애플리케이션의 표준 선택지 ⭐
- **ZGC**: 초저지연이 생명인 시스템 (극소수 경우)

**대부분의 프로덕션 환경은 G1 GC로 충분합니다.**
