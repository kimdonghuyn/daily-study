## 📚 모범 답안 & 개념 정리

### 1. 📖 개념 설명

# 인터페이스 vs 추상클래스

## 모범 답안

**계약서 작성 예시로 본 선택 기준:**

**추상클래스를 사용해야 하는 경우:**
- 회사가 "계약서는 반드시 헤더(회사명, 작성일), 본문, 서명란을 포함해야 한다"는 **공통 구조와 구현**을 제공
- 모든 직원이 이 틀을 상속받아 사용하되, 본문 내용만 자신의 계약 유형에 맞게 작성
- 예: `BaseContract` 추상클래스가 헤더, 서명 기능은 이미 구현하고, `write()` 메서드만 각자 구현

```java
// 추상클래스: 구조(상태) + 구현(동작)을 함께 제공
abstract class BaseContract {
    protected String companyName = "우리회사";
    protected LocalDate createdDate = LocalDate.now();
    
    // 공통 구현 - 모든 계약에서 동일
    public final void printHeader() {
        System.out.println(companyName + " | " + createdDate);
    }
    
    // 강제 구현 - 각자 다르게
    abstract void writeContent();
    
    public final void generateContract() {
        printHeader();
        writeContent();
        printSignature();
    }
}

class EmploymentContract extends BaseContract {
    @Override
    void writeContent() {
        // 고용 계약서 내용만 작성
    }
}
```

**인터페이스를 사용해야 하는 경우:**
- "최소 규칙: 계약서는 작성 메서드와 검증 메서드를 포함해야 한다"는 **계약만 정의**
- 각 부서가 자신의 방식대로 완전히 다른 형식과 프로세스로 구현
- 예: `Contractable` 인터페이스는 메서드 시그니처만 정의하고, 구현은 전적으로 구체적인 클래스에 위임

```java
// 인터페이스: 계약(행동)만 강제
interface Contractable {
    void write();
    boolean validate();
    void sign();
}

class SalesContract implements Contractable {
    @Override
    public void write() {
        // 영업계약만의 고유한 형식
    }
    
    @Override
    public boolean validate() {
        // 영업계약만의 고유한 검증 로직
    }
}

class LegalContract implements Contractable {
    @Override
    public void write() {
        // 법률계약만의 고유한 형식
    }
}
```

**결론:** 
- **공통 구조+로직이 있으면** → 추상클래스 (DRY 원칙, 코드 재사용)
- **행동만 강제하고 구현은 자유** → 인터페이스 (느슨한 결합, 다형성)

## 핵심 개념 정리

### 추상클래스 (Abstract Class)
- **목적**: IS-A 관계 (같은 종류의 객체들이 공통 특성을 상속)
- **상태(필드)**: ✅ 가능 (인스턴스 변수 보유)
- **구현**: ✅ 일부 또는 모두 구현 가능 (concrete method)
- **생성자**: ✅ 가능
- **접근제어자**: private, protected, public 모두 사용 가능
- **다중 상속**: ❌ 불가능 (extends는 하나만)
- **확장 목적**: 공통 코드 재사용과 구조 통일

```java
abstract class Animal {
    private String name;  // ✅ 상태 보유
    
    public Animal(String name) {  // ✅ 생성자
        this.name = name;
    }
    
    public void sleep() {  // ✅ 구현된 메서드
        System.out.println(name + " is sleeping");
    }
    
    abstract void makeSound();  // ❌ 미구현 강제
}
```

### 인터페이스 (Interface)
- **목적**: CAN-DO 관계 (특정 행동 계약)
- **상태(필드)**: ❌ 불가능 (상수만 가능)
- **구현**: ✅ Java 8+ default/static 메서드 가능
- **생성자**: ❌ 불가능
- **접근제어자**: public만 가능 (암묵적)
- **다중 상속**: ✅ 가능 (implements 여러 개)
- **확장 목적**: 행동 계약과 느슨한 결합

```java
interface Flyable {
    // ❌ 상태 불가능
    // private String name;
    
    // ✅ 상수만 가능
    int MAX_ALTITUDE = 10000;
    
    // 강제 구현 메서드
    void fly();
    
    // Java 8+ 구현 메서드 (default)
    default void land() {
        System.out.println("Landing...");
    }
}
```

### 선택 기준 비교표

| 기준 | 추상클래스 | 인터페이스 |
|------|----------|----------|
| **관계** | IS-A | CAN-DO |
| **코드 재사용** | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| **다중 구현** | ❌ | ✅ |
| **필드 보유** | ✅ | ❌ |
| **생성자** | ✅ | ❌ |
| **구현 강제도** | 중간 | 높음 |
| **변경에 유연** | 중간 | ⭐⭐⭐⭐⭐ |

## 실무 포인트

### 1️⃣ **추상클래스 활용 사례** (실제 구현이 필요할 때)

```java
// ✅ 좋은 예: 공통 로직이 상당한 경우
abstract class PaymentProcessor {
    // 공통 상태
    protected BigDecimal amount;
    protected String orderId;
    
    // 공통 구현 - 모든 결제에서 필요
    public final void processPayment() {
        validateAmount();       // 공통 검증
        recordTransaction();    // 공통 기록
        executePayment();       // 각자 구현
        notifyUser();          // 공통 알림
    }
    
    private void validateAmount() { /* 공통 로직 */ }
    private void recordTransaction() { /* 공통 로직 */ }
    
    abstract void executePayment();  // 각자 다르게
}

class CreditCardProcessor extends PaymentProcessor {
    @Override
    void executePayment() {
        // 신용카드 결제 로직만 구현
    }
}
```

### 2️⃣ **인터페이스 활용 사례** (행동만 강제할 때)

```java
// ✅ 좋은 예: 다양한 방식의 구현이 필요할 때
interface Logger {
    void log(String message);
    void error(String message);
}

class ConsoleLogger implements Logger {
    @Override
    public void log(String message) {
        System.out.println(message);
    }
}

class DatabaseLogger implements Logger {
    @Override
    public void log(String message) {
        repository.save(message);  // 완전히 다른 구현
    }
}

class CloudLogger implements Logger {
    @Override
    public void log(String message) {
        cloudService.upload(message);  // 완전히 다른 구현
    }
}

// 의존성 주입 - 인터페이스 덕분에 가능
class Application {
    private Logger logger;
    
    public Application(Logger logger) {
        this.logger = logger;  // 어떤 구현도 받을 수 있음
    }
}
```

```java
// 추상클래스로 공통 구조 제공
abstract class BaseRepository<T> {
    protected DataSource dataSource;
    
    public final List<T> findAll() {
        // 공통 연결, 트랜잭션 처리
        return executeQuery();
    }
    
    protected abstract String getSqlQuery();
}

// 인터페이스로 추가 행동 강제
interface CacheableRepository {
