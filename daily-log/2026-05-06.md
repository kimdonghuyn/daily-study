# @Transactional 전파 수준과 롤백 기술 면접 완벽 가이드

## 메인 질문 - 모범 답안

### REQUIRES_NEW의 의도적 사용 케이스

**REQUIRES_NEW는 부모 트랜잭션의 성공/실패와 무관하게 독립적으로 커밋되어야 하는 작업에 사용됩니다.**

```java
// 📌 실제 사례: 비동기 감사 로그 기록
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(OrderRequest request) {
        // 부모 트랜잭션: 주문 생성
        Order order = orderRepository.save(request.toEntity());
        
        // 자식 트랜잭션: 감시 로그 (부모 실패해도 기록되어야 함)
        recordAuditLog(order);
        
        // 주문 후속 작업 중 예외 발생
        if (shouldFail()) {
            throw new OrderProcessException("주문 처리 실패");
        }
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void recordAuditLog(Order order) {
        // 부모가 롤백되어도 이 트랜잭션은 독립적으로 커밋
        auditLogRepository.save(AuditLog.of(
            order.getId(), 
            "ORDER_CREATED",
            LocalDateTime.now()
        ));
    }
}
```

### 의도적 사용 이유

| 시나리오 | 이유 | 예시 |
|---------|------|------|
| **감사/로깅** | 실패해도 기록 유지 필요 | 주문 생성 실패해도 시도 로그 남기기 |
| **외부 API 호출** | 이미 외부에 영향 줬으면 취소 불가 | 결제 게이트웨이 결제 후 DB 실패 |
| **메일/알림** | 비즈니스 로직 실패 후에도 발송 | 주문 실패 후 사용자 알림 메일 |
| **통계 집계** | 원본 트랜잭션과 독립적 집계 | 실시간 분석 데이터 갱신 |

### 데이터 일관성 위험

```java
// ⚠️ 위험한 패턴
@Transactional
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepository.findById(fromId).lock();
    Account to = accountRepository.findById(toId).lock();
    
    from.debit(amount);  // 출금 (부모 트랜잭션)
    accountRepository.save(from);
    
    // ❌ REQUIRES_NEW: to의 입금은 독립적으로 커밋됨
    creditToAccount(toId, amount);
    
    // 이 지점에서 예외 발생 → from은 롤백, to는 커밋
    // 결과: 돈이 소실됨!
    if (shouldFail()) {
        throw new TransferException();
    }
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
private void creditToAccount(Long toId, BigDecimal amount) {
    Account to = accountRepository.findById(toId).lock();
    to.credit(amount);
    accountRepository.save(to);
    // 독립적으로 즉시 커밋
}
```

**결론: REQUIRES_NEW는 강력하지만 "도미노 효과"를 고려해야 함**

## 핵심 개념 정리

### 1️⃣ 전파 수준의 원리

```
┌─────────────────────────────────────────┐
│  부모 트랜잭션 (Parent TX)               │
├─────────────────────────────────────────┤
│ REQUIRED      → 부모 TX 재사용           │
│ REQUIRES_NEW  → 새 TX 시작 (부모 일시중지) │
│ NESTED        → 세이브포인트 생성        │
│ SUPPORTS      → TX 있으면 참여, 없으면 X │
│ NOT_SUPPORTED → TX 없이 실행             │
└─────────────────────────────────────────┘
```

### 2️⃣ REQUIRES_NEW의 동작 메커니즘

```java
// 내부 동작 흐름
1. 부모 트랜잭션 시작
   ├─ SavePoint 생성 (TX 일시중지)
   ├─ 새 트랜잭션 시작 (자식)
   │  └─ 자식 TX 작업 수행 및 커밋
   └─ 부모 TX 재개 (새로운 SavePoint)

2. 부모에서 예외 발생
   └─ 부모만 롤백 (자식은 이미 커밋됨)
```
