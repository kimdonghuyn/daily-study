# 프록시 패턴과 AOP 내부 동작 원리

## 모범 답안

### 프록시 패턴이 필요한 이유

**좋은 질문입니다. 핵심은 "관심사의 분리(Separation of Concerns)"입니다.**

편의점 예시를 그대로 들면:
- **핵심 관심사**: 물건을 판다
- **횡단 관심사**: 결제, 영수증, 포장, 고객 정보 기록

만약 점원(계산원)이 없다면, 물건을 파는 사람이 직접 모든 걸 처리해야 합니다. 그러면:
1. 판매 로직이 복잡해짐
2. 결제 규칙이 바뀌면 모든 판매 코드를 수정해야 함
3. 같은 추가 작업들을 여러 곳에서 중복으로 코딩

**프록시 패턴을 쓰는 이유:**
```java
// 프록시 없을 때 - 비즈니스 로직에 혼재됨
public class Product {
    public void sell(User user) {
        // 보안 체크
        if (!user.isAuthenticated()) throw new Exception();
        
        // 로깅
        log.info("판매 시작");
        
        // 캐싱 체크
        if (cache.contains(user)) return cache.get(user);
        
        // 실제 판매 로직
        processPayment(user);
        updateInventory();
        
        // 또 다른 횡단 관심사들...
        notifyWarehouse();
        recordAnalytics();
    }
}

// 프록시로 분리했을 때 - 핵심 로직 명확함
public class ProductProxy implements ProductService {
    private Product realProduct;
    
    @Override
    public void sell(User user) {
        // 횡단 관심사는 여기서 처리
        authenticate(user);
        logAccess(user);
        
        // 핵심 로직만 깔끔하게
        realProduct.sell(user);
        
        // 후처리도 여기서
        notifyWarehouse();
    }
}

// 실제 Product는 순수 비즈니스 로직만
public class RealProduct implements ProductService {
    @Override
    public void sell(User user) {
        processPayment(user);
        updateInventory();
    }
}
```

**왜 이게 좋은가:**
- ✅ 핵심 로직과 부가 기능이 분리됨
- ✅ 변경에 강함 (보안 정책 변경 → ProductProxy만 수정)
- ✅ 재사용성 (같은 프록시를 다른 상품에도 적용 가능)
- ✅ 테스트 용이 (핵심 로직만 테스트 가능)

### 꼬리 질문: AOP가 나온 이유

**정확한 지적입니다. 바로 그 이유 때문입니다.**

프록시를 매번 만들면:

```java
// 상품 판매 프록시
class ProductProxy { /* 프록시 로직 */ }

// 사용자 조회 프록시
class UserServiceProxy { /* 같은 로직... */ }

// 주문 관리 프록시
class OrderServiceProxy { /* 또 같은 로직... */ }

// 100개의 클래스면? 100개의 프록시를 수동으로??
```

**스프링 AOP의 등장:**

```java
// 1번만 정의하면 여러 곳에 자동으로 적용됨
@Aspect
@Component
public class SecurityAspect {
    @Before("execution(* com.example.service.*.*(..))")
    public void checkSecurity(JoinPoint jp) {
        // 모든 service 메서드 호출 전에 자동으로 실행
        if (!SecurityContext.isAuthenticated()) {
            throw new UnauthorizedException();
        }
    }
}

@Aspect
@Component
public class LoggingAspect {
    @Around("execution(* com.example.service.*.*(..))")
    public Object logExecution(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return pjp.proceed(); // 실제 메서드 실행
        } finally {
            log.info("메서드: {}, 실행시간: {}ms", 
                pjp.getSignature(), System.currentTimeMillis() - start);
        }
    }
}

// 서비스 클래스는 깔끔함
@Service
public class ProductService {
    public void sell(User user) {
        // 위의 Aspect들이 자동으로 적용됨
        processPayment(user);
        updateInventory();
    }
}
```

**AOP의 진짜 가치:**
- 🎯 **선언적 프로그래밍**: 어디에 적용할지 패턴으로 정의 (pointcut)
- 🔄 **자동 프록시 생성**: Spring이 런타임에 동적으로 프록시 생성
- 📦 **재사용성**: 한 번 정의한 관점을 100개 클래스에 동시 적용
- 🔗 **체이닝**: 여러 Aspect가 자동으로 순서대로 적용

## 핵심 개념 정리

### 프록시 패턴
- **정의**: 실제 객체의 접근을 제어하는 대리자 객체를 사용하는 패턴
- **역할**: 접근 제어, 로깅, 캐싱, 검증 등 추가 작업 담당
- **구조**: `Client → Proxy → RealObject`

### 프록시 패턴 vs 데코레이터 패턴
| 구분 | 프록시 | 데코레이터 |
|------|--------|-----------|
| 목적 | 접근 제어, 보안 | 기능 추가 |
| 생성 시점 | 사전에 정의 | 런타임 조합 |
| 책임 | 대리자 역할 | 기능 확장 |

### AOP (Aspect-Oriented Programming)
- **핵심 개념**:
  - `Aspect`: 횡단 관심사 모듈 (로깅, 보안, 캐싱)
  - `Pointcut`: Aspect를 적용할 대상 지정 (메서드 패턴)
  - `JoinPoint`: 실제 적용되는 지점
  - `Advice`: 실행할 부가 기능 (Before, After, Around 등)

- **AOP 동작 방식**:
  ```
  1. 빈 생성 시점
  2. BeanPostProcessor가 빈 감지
  3. Pointcut 매칭으로 프록시 대상 판단
  4. 동적으로 프록시 객체 생성 (CGLIB 또는 JDK Dynamic Proxy)
  5. 원본 객체 대신 프록시 주입
  6. 메서드 호출 시 Advice 실행 후 원본 메서드 호출
  ```

### Spring AOP의 프록시 생성 방식

```java
// 1. JDK Dynamic Proxy (인터페이스 기반)
public interface ProductService {
    void sell(User user);
}

// 2. CGLIB (상속 기반)
public class ProductService {  // 인터페이스 없어도 프록시 생성 가능
    public void sell(User user) { }
}

// Spring이 자동으로 선택:
// - 인터페이스 구현 → JDK Dynamic Proxy
// - 일반 클래스 → CGLIB
```

### Advice 타입과 실행 순서

```java
@Aspect
public class MultipleAdviceExample {
    
    @Before("pointcut()")  // 1번 실행
    public void before() { }
    
    @Around("pointcut()")  // 2-1번 실행 (전)
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        // 전처리
        Object result = pjp.proceed(); // 실제 메서드 실행
        // 후처리
        return result; // 2-5번 실행 (후)
    }
    
    @AfterReturning("pointcut()") // 2-2번 실행
    public void afterReturning(JoinPoint jp) { }
    
    @After("pointcut()")  // 2-3번 실행
    public void after() { }
    
    @AfterThrowing
