## 📚 모범 답안 & 개념 정리

### 1. 🔧 문제 상황

# @Transactional 기본 동작 - 모범 답안

## 모범 답안

**@Transactional은 여러 DB 작업을 "all or nothing" 원칙으로 묶어주는 도구입니다.**

당신의 예시로 설명하면:

```java
@Transactional
public void purchaseProduct(Long productId, Long userId) {
    // 1) 재고 감소
    Product product = productRepository.findById(productId);
    product.decreaseStock();  // UPDATE
    
    // 2) 구매 기록 저장
    Purchase purchase = new Purchase(userId, productId);
    purchaseRepository.save(purchase);  // INSERT
}
```

**@Transactional이 해주는 일:**
- ✅ 1번과 2번이 모두 성공 → 둘 다 DB에 커밋(반영)
- ❌ 중간에 실패 → 둘 다 롤백(원래대로 돌리기)

서버가 중간에 꺼진다면?
- @Transactional 없음: 재고만 줄고 구매기록 못 씀 (데이터 불일치)
- @Transactional 있음: 자동 롤백으로 재고도 원래대로 돌림 (일관성 유지)

## 핵심 개념 정리

### 1️⃣ **Transaction의 4가지 특성 (ACID)**

| 특성 | 설명 | 예시 |
|------|------|------|
| **A (Atomicity)** | 원자성 - 전부 또는 전무 | 재고 감소 + 구매기록 저장 = 1개 단위 |
| **C (Consistency)** | 일관성 - 데이터 규칙 유지 | 재고 < 0 되지 않음 |
| **I (Isolation)** | 독립성 - 동시 작업 간섭 없음 | 동시 구매 시 중복 판매 방지 |
| **D (Durability)** | 지속성 - 저장되면 사라지지 않음 | 커밋된 데이터는 서버 다운해도 남음 |

### 2️⃣ **@Transactional 동작 원리**

```
메서드 시작
    ↓
[Transaction 시작] - savepoint 설정
    ↓
DB 작업 1, 2, 3...
    ↓
예외 발생? → [ROLLBACK] 모든 작업 취소 ❌
    ↓
정상 완료? → [COMMIT] 모든 작업 반영 ✅
    ↓
메서드 종료
```

### 3️⃣ **주요 옵션들**

```java
@Transactional(
    readOnly = false,           // 쓰기 작업 가능
    propagation = REQUIRED,     // 기존 TX 있으면 사용, 없으면 새로 생성
    isolation = READ_COMMITTED, // 트랜잭션 격리 수준
    timeout = 10,              // 10초 초과 시 자동 롤백
    rollbackFor = Exception.class  // 어떤 예외에서 롤백할지
)
public void process() { }
```

## 실무 포인트

### 🚨 **자주 하는 실수**

**❌ 실수 1: 클래스에만 붙이기**
```java
@Transactional  // 메서드에는 효과 없음
public class UserService {
    public void register(User user) { }  // TX 적용 안 됨!
}
```

**✅ 올바른 방법:**
```java
public class UserService {
    @Transactional  // 메서드에 붙여야 함
    public void register(User user) { }
}
```

---

**❌ 실수 2: 메서드 내 try-catch로 예외 잡기**
```java
@Transactional
public void purchase(Long productId) {
    decreaseStock(productId);
    
    try {
        savePurchase(productId);
    } catch (Exception e) {
        // 여기서 예외 처리하면
        e.printStackTrace();  // @Transactional이 예외를 못 본다!
                             // 롤백 안 됨!
    }
}
```

**✅ 올바른 방법:**
```java
@Transactional
public void purchase(Long productId) {
    decreaseStock(productId);
    savePurchase(productId);  // 예외가 밖으로 나가야 롤백됨
}
```

### 💡 **성능 최적화 팁**

**1. 읽기 전용 트랜잭션 (성능 향상)**
```java
@Transactional(readOnly = true)  // DB 락 불필요
public List<Product> getProducts() {
    return productRepository.findAll();
}
```

**2. 트랜잭션 범위 최소화**
```java
// ❌ 느림: 파일 다운로드까지 TX 유지
@Transactional
public byte[] downloadAndSave(Long productId) {
    Product product = productRepository.findById(productId);
    byte[] file = downloadFromExternal();  // 네트워크 지연!
    return file;
}

// ✅ 빠름: DB 작업만 TX
@Transactional
public Product getProduct(Long productId) {
    return productRepository.findById(productId);
}

public byte[] downloadFile(Product product) {
    return downloadFromExternal();
}
```

**3. 트랜잭션 타임아웃 설정**
```java
@Transactional(timeout = 5)  // 5초 이상 걸리면 자동 롤백
public void heavyOperation() {
    // 무한 대기 방지
}
```

### 🔍 **실제 시나리오**

**상황: 동시에 2명이 마지막 재고 1개를 구매**

```
시간  | 고객 A                    | 고객 B
-----|--------------------------|---------------------------
T1   | 재고 조회: 1개 ✅        |
T2   | 재고 1개 확인            | 재고 조회: 1개 ✅
T3   | 재고 0개로 업데이트      | 재고 1개 확인
T4   | 구매기록 저장            | 재고 0개로 업데이트
T5   | 커밋 ✅                 | 구매기록 저장
T6   |                          | 커밋 ✅ (잘못됨!)
```

→ **@Transactional + isolation 레벨 설정**으로 방지

```java
@Transactional(isolation = SERIALIZABLE)  // 가장 강한 격리
public void purchase(Long productId) {
    Product product = productRepository.findById(productId);
    if (product.getStock() <= 0) {
        throw new OutOfStockException();  // 롤백됨
    }
    product.decreaseStock();
    purchaseRepository.save(new Purchase(...));
}
```

### 📋 **체크리스트**

✅ @Transactional은 메서드에 붙인다  
✅ 예외가 밖으로 나가야만 롤백된다  
✅ 읽기만 하면 readOnly = true로 설정  
✅ 네트워크/파일 작업은 TX 범위 밖에  
✅ timeout을 설정해서 무한 대기 방지  
✅ 동시성 이슈는 isolation 레벨로 해결
