# GC의 필요성과 동작 원리 - 모범 답안

## 모범 답안

### 메인 질문: 메모리에 데이터를 계속 쌓아두면 생기는 문제

```
프로그램 실행 중 메모리 해제가 안 되면 "메모리 누수(Memory Leak)"가 발생합니다.

구체적인 문제점:
1. 힙 메모리 부족 → OutOfMemoryError 발생
2. 프로그램 성능 저하 → GC 부하 증가
3. 시스템 전체 불안정 → 서버 다운
```

**예시 시나리오:**
```java
// ❌ 문제가 있는 코드
while (true) {
    List<byte[]> list = new ArrayList<>();
    list.add(new byte[1024 * 1024]); // 1MB 계속 할당
    // 메모리 해제 안 됨 → 힙 메모리 고갈
}

// 실행 결과:
// 약 수백 MB 후 → Exception in thread "main" java.lang.OutOfMemoryError
```

### GC가 필요한 이유

| 항목 | 수동 관리 (C/C++) | GC 자동화 (Java) |
|------|------------------|-----------------|
| **개발 생산성** | 낮음 (매번 free() 호출) | 높음 (자동 해제) |
| **버그 위험** | 높음 (double free, 포인터 오류) | 낮음 (메모리 안정성) |
| **메모리 효율** | 수동으로 최적화 가능 | GC 오버헤드 있음 |

**결론:** GC는 프로그래머의 부담을 줄이고 메모리 안전성을 보장하기 위해 필수적

### 꼬리 질문: GC가 "필요 없는 데이터"를 판단하는 방법

GC는 **Reachability(도달 가능성)** 개념으로 판단합니다.

```
핵심 원리: "살아있는 객체가 참조하지 않는 모든 객체 = 쓸모없는 객체"
```

#### 🔍 객체 생명주기 판단 방식

**1) Reference Counting (참조 카운팅)**
```java
// 개념: 누가 이 객체를 가리키는가?
Person person = new Person("Kim"); // 참조 카운트 = 1
Person person2 = person;            // 참조 카운트 = 2

person = null;                       // 참조 카운트 = 1
person2 = null;                      // 참조 카운트 = 0 → GC 대상

// ❌ 문제점: Circular Reference (순환 참조)
// Person A가 Person B를 참조
// Person B가 Person A를 참조
// → 참조 카운트가 0이 아니어서 메모리 누수 발생!
```

**2) Mark and Sweep (표시하고 정리) - JVM의 실제 방식**
```
1단계: Mark (GC Root에서 도달 가능한 객체 표시)
  - GC Root: 스택 변수, 정적 변수, 클래스 로더 등
  - Root에서 시작해 참조 체인 따라 "살아있는 객체" 표시

2단계: Sweep (표시 안 된 객체 메모리 해제)
  - Mark 되지 않은 객체 = 죽은 객체 → 메모리 회수
  - 연속적 메모리 위치로 정렬 (Compact)
```

**시각화:**
```
Stack: [person ──┐
        list   ──┤
        temp   ──┤
                 │
Heap: ┌─────────────────────┐
      │ Person (Mark: ✓)    │◄─┘ GC Root가 도달 가능
      │ ArrayList (Mark: ✓) │◄─  살아있는 객체
      │ String (Mark: ✓)    │
      ├─────────────────────┤
      │ OldData (Mark: ✗)   │ ← 참조되지 않음 → GC 대상
      │ Cache (Mark: ✗)     │ ← 정리됨!
      └─────────────────────┘
```

## 핵심 개념 정리

### 📌 메모리 누수 관련

- **Memory Leak**: 더 이상 사용하지 않는 메모리가 해제되지 않는 현상
- **Dangling Pointer** (C언어): 해제된 메모리를 여전히 가리키는 포인터
- **Double Free**: 같은 메모리를 두 번 해제하는 오류

### 📌 GC의 동작 원리

| 개념 | 설명 |
|------|------|
| **Reachability** | 객체가 직접 또는 간접적으로 참조되고 있는 상태 |
| **GC Root** | GC 대상 판단의 시작점 (스택, 전역변수, JNI 참조) |
| **Mark** | 살아있는 객체를 표시하는 단계 |
| **Sweep** | 표시되지 않은 객체 메모리 회수 |
| **Compact** | 메모리 단편화 방지를 위한 메모리 정렬 |

### 📌 GC 종류

```
Generational GC (세대별 GC) - JVM의 표준
├─ Young Generation: 금방 죽는 객체가 대부분
│  ├─ Eden: 새 객체 할당
│  ├─ Survivor 0/1: 임시 보관
│  └─ Minor GC: 빠르고 자주 발생
│
└─ Old Generation: 오래 사는 객체
   └─ Major GC (Full GC): 느리고 드물게 발생
```

### 📌 GC의 장단점

**장점:**
- 메모리 안전성 (null pointer exception 방지)
- 개발 생산성 향상
- 메모리 누수 걱정 감소

**단점:**
- GC Pause (STW - Stop The World): 프로그램 일시 정지
- 예측 불가능한 성능 지연
- 메모리 오버헤드
