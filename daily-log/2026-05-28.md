### 1. 🎤 면접 질문

# 인터페이스 vs 추상클래스 - 결제 시스템 설계

## 모범 답안

### 1️⃣ 이 상황에서의 선택 기준

**결제 시스템에서는 `인터페이스`를 선택하겠습니다.**

**이유:**
- 결제 수단들은 `"결제"라는 행동만 공유`할 뿐, 공통 상태나 구현 로직이 없음
- 신용카드, 계좌이체, 휴대폰결제는 완전히 독립적인 결제 프로세스
- 향후 다양한 결제 수단이 추가될 가능성이 높음
- 각 결제 수단이 다른 여러 특성을 동시에 가질 수 있음 (예: "결제 + 환불 + 구독")

```java
// ✅ 인터페이스 사용
public interface PaymentMethod {
    boolean validate();
    void processPayment(BigDecimal amount);
    void refund(BigDecimal amount);
    PaymentResult getResult();
}

public class CreditCardPayment implements PaymentMethod {
    @Override
    public boolean validate() {
        // 카드번호 검증
        return validateCardNumber();
    }
    
    @Override
    public void processPayment(BigDecimal amount) {
        // 신용카드사 API 호출
        callCreditCardGateway(amount);
    }
    
    @Override
    public void refund(BigDecimal amount) {
        // 환불 처리
    }
    
    @Override
    public PaymentResult getResult() {
        return new PaymentResult();
    }
}

public class BankTransferPayment implements PaymentMethod {
    @Override
    public boolean validate() {
        // 계좌 검증
        return validateBankAccount();
    }
    
    @Override
    public void processPayment(BigDecimal amount) {
        // 은행 API 호출
        callBankAPI(amount);
    }
    
    @Override
    public void refund(BigDecimal amount) {
        // 환불 처리
    }
    
    @Override
    public PaymentResult getResult() {
        return new PaymentResult();
    }
}
```

### 2️⃣ 하드코딩하지 말아야 하는 이유

**현재 코드:**
```java
// ❌ 하드코딩 - 결제 로직이 산재됨
public void checkout(Order order, String paymentType) {
    if ("CREDIT_CARD".equals(paymentType)) {
        validateCardNumber(order.getCardNumber());
        callCreditCardGateway(order.getAmount());
        updateOrderStatus(order);
    } 
    else if ("BANK_TRANSFER".equals(paymentType)) {
        validateBankAccount(order.getAccount());
        callBankAPI(order.getAmount());
        updateOrderStatus(order);
    }
    // ... 계속 if-else 반복
}

// 문제점:
// 1. 새 결제 수단 추가 시마다 이 함수를 수정해야 함
// 2. 결제 로직이 여러 곳에 분산되어 유지보수 어려움
// 3. 테스트가 복잡함
// 4. 단일 책임 원칙 위반
```

**추상화 사용:**
```java
// ✅ 추상화 - 결제 로직이 캡슐화됨
public void checkout(Order order, PaymentMethod payment) {
    if (!payment.validate()) {
        throw new PaymentValidationException();
    }
    
    payment.processPayment(order.getAmount());
    updateOrderStatus(order);
    notifyPaymentResult(payment.getResult());
}

// 장점:
// 1. 새 결제 수단 추가 시 이 함수는 수정 안 함 (Open/Closed Principle)
// 2. 각 결제 방식의 로직이 캡슐화됨
// 3. 각 결제 수단을 독립적으로 테스트 가능
// 4. 결제 처리 로직이 한곳에 집중됨
```

### 3️⃣ 새로운 결제 수단 추가 시 코드 변경

**인터페이스 사용 시:**
```java
// 🎯 추가 코드만 작성 (기존 코드 무수정)
public class CryptoPayment implements PaymentMethod {
    @Override
    public boolean validate() {
        return validateWalletAddress();
    }
    
    @Override
    public void processPayment(BigDecimal amount) {
        callBlockchainAPI(amount);
    }
    
    @Override
    public void refund(BigDecimal amount) {
        // 스마트 컨트랙트 호출
    }
    
    @Override
    public PaymentResult getResult() {
        return new PaymentResult();
    }
}

// ✅ checkout() 메서드는 그대로! 다형성으로 자동 처리됨
PaymentMethod crypto = new CryptoPayment();
checkout(order, crypto);
```

**추상클래스 사용 시 (비추천 이유):**
```java
// ❌ 공통 구현이 없어서 이미 문제가 있음

}

// 하지만 만약 사용했다면:

    @Override

        callBlockchainAPI(amount);
    }

// 여전히 추가 코드만 필요하지만,
// 추상클래스는 단일 상속만 가능하다는 제약이 생김
```

## 핵심 개념 정리

### 📌 인터페이스 (Interface)
- **What (무엇을 하는가)**: 행동 계약서
- **특징**:
  - 구현 없음 (Java 8 이전)
  - 다중 구현 가능
  - 메서드 시그니처만 정의
  - `is-a` 관계보다 `can-do` 관계

### 📌 추상클래스 (Abstract Class)
- **How (어떻게 공통으로 하는가)**: 부분 구현
- **특징**:
  - 공통 구현 + 추상 메서드 혼재
  - 단일 상속만 가능
  - 멤버 변수 가능
  - `is-a` 관계 (상속 계층)

### 📌 선택 기준

| 상황 | 인터페이스 | 추상클래스 |
|------|----------|---------|
| 공통 구현 있음 | ❌ | ✅ |
| 공통 상태(필드) | ❌ | ✅ |
| 다중 구현 필요 | ✅ | ❌ |
| 행동 계약만 필요 | ✅ | ❌ |
| 변경 가능성 높음 | ✅ | ❌ |

### 📌 SOLID 원칙과의 관계
- **Open/Closed Principle**: 새 결제 수단 추가할 때 기존 코드 수정 없음
- **Dependency Inversion**: `PaymentMethod` 인터페이스에만 의존
- **Single Responsibility**: 각 결제 클래스는 하나의 결제 방식만 담당
