## 📚 모범 답안 & 개념 정리

### 1. 📖 개념 설명

# IoC와 DI — 왜 직접 객체를 만들지 않나

## 모범 답안

두 번째 방식(DI)이 훨씬 유연합니다. 그 이유는:

**첫 번째 방식(직접 의존성 생성)의 문제점:**
```java
// ❌ 강한 결합
class PizzaChef {
    private SamsungOven oven = new SamsungOven(); // 직원이 특정 오븐에만 의존
    
    public void makePizza() {
        oven.heat(200);
    }
}
```
- 새로운 오븐(LG, 독일제)으로 바꾸려면 코드 수정 필수
- 테스트할 때 실제 오븐을 사용해야 함 → 느리고 비쌈
- 직원(PizzaChef)이 특정 오븐(SamsungOven)에 종속됨

**두 번째 방식(DI)의 장점:**
```java
// ✅ 느슨한 결합
class PizzaChef {
    private Oven oven; // 추상화된 인터페이스만 의존
    
    // 의존성을 주입받음 (누가? 외부에서)
    public PizzaChef(Oven oven) {
        this.oven = oven;
    }
    
    public void makePizza() {
        oven.heat(200);
    }
}

// 사용처에서 결정
PizzaChef chef1 = new PizzaChef(new SamsungOven());
PizzaChef chef2 = new PizzaChef(new LGOven());
PizzaChef chef3 = new PizzaChef(new MockOven()); // 테스트용
```

**프로그래밍에서 DI가 좋은 이유:**

1. **변경에 대한 유연성** → 요구사항 변화에 빠르게 대응
2. **테스트 가능성** → Mock 객체로 단위테스트 용이
3. **재사용성** → 같은 코드가 다양한 환경에서 작동
4. **책임 분리** → 각 클래스는 "무엇을 하는가"만 집중
5. **확장성** → 새 기능 추가 시 기존 코드 수정 불필요

## 핵심 개념 정리

### IoC (Inversion of Control) - 제어의 역전
- **개념**: 객체 생성과 생명주기 관리의 책임을 외부(Spring Container 등)에 위임
- **효과**: 프로그래머가 "어떻게"가 아닌 "무엇을"에 집중

### DI (Dependency Injection) - 의존성 주입
- **의존성**: 클래스가 다른 클래스를 필요로 하는 관계
- **3가지 방식**:
  ```java
  // 1️⃣ 생성자 주입 (가장 권장)
  class PizzaChef {
      private final Oven oven;
      public PizzaChef(Oven oven) { this.oven = oven; }
  }
  
  // 2️⃣ 세터 주입 (선택적 의존성)
  class PizzaChef {
      private Oven oven;
      public void setOven(Oven oven) { this.oven = oven; }
  }
  
  // 3️⃣ 필드 주입 (Spring @Autowired - 편하지만 테스트 어려움)
  class PizzaChef {
      @Autowired
      private Oven oven;
  }
  ```

### 의존성 역전 원칙 (DIP - Dependency Inversion Principle)
- **하이레벨 모듈**은 로우레벨 모듈에 의존하면 안됨
- **둘 다 추상화(인터페이스)에 의존**해야 함

```java
// ❌ 나쁜 예
class PizzaChef {
    private SamsungOven oven; // 구체적인 클래스
}

// ✅ 좋은 예
interface Oven {
    void heat(int temp);
}

class PizzaChef {
    private Oven oven; // 추상화된 인터페이스
}
```

## 실무 포인트

### 1. Spring Framework에서의 실제 적용

```java
// @Component 등록 → IoC 컨테이너가 관리
@Service
public class PizzaService {
    private final OvenRepository ovenRepo;
    
    // 생성자 주입 (Spring이 자동으로 의존성 해결)
    public PizzaService(OvenRepository ovenRepo) {
        this.ovenRepo = ovenRepo;
    }
}

// 테스트 코드
@Test
public void testPizzaMaking() {
    // Mock 객체 주입 → 테스트만 전담
    Oven mockOven = mock(Oven.class);
    PizzaChef chef = new PizzaChef(mockOven);
    chef.makePizza();
    verify(mockOven).heat(200);
}
```

### 2. 언제 DI를 써야 하나?

| 상황 | 방식 |
|------|------|
| 필수 의존성 | **생성자 주입** (null 불가, 불변) |
| 선택적 의존성 | **세터 주입** |
| 변하는 의존성 | **메서드 주입** |
| 단순 유틸리티 | **직접 생성** (꼭 필요 없음) |

### 3. 실무 문제 해결 예시

**배경**: 결제 기능이 있는데 "Stripe"에서 "PayPal"로 변경

```java
// ❌ DI 없이
class PaymentProcessor {
    public void pay() {
        StripeAPI.charge(...); // 하드코딩
    }
}
// 변경하려면 모든 코드 수정 필요

// ✅ DI로
interface PaymentGateway {
    void charge(int amount);
}

class StripeGateway implements PaymentGateway { ... }
class PayPalGateway implements PaymentGateway { ... }

class PaymentProcessor {
    private PaymentGateway gateway;
    public PaymentProcessor(PaymentGateway gateway) {
        this.gateway = gateway;
    }
}

// 설정 파일에서만 변경
// gateway = new StripeGateway(); → gateway = new PayPalGateway();
```

### 4. 주의사항

- **과도한 추상화** → 프로젝트가 복잡해질 수 있음
- **생성자 주입 권장** → 필드 주입은 테스트 어려움
- **불변성 추구** → `private final` 사용

IoC/DI는 단순히 "코드 작성 편의"가 아니라 **유지보수성과 테스트 가능성의 핵심**입니다.
