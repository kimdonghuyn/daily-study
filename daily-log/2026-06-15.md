## 📚 모범 답안 & 개념 정리

### 1. 📖 개념 설명

# @Transactional 기본 동작

## 모범 답안

@Transactional은 **"All or Nothing"** 원칙으로 작동합니다.

당신의 송금 예시로 설명하면:

```java
@Transactional
public void transfer(Long fromAccountId, Long toAccountId, BigDecimal amount) {
    Account fromAccount = accountRepository.findById(fromAccountId);
    fromAccount.withdraw(amount);  // A 계좌에서 차감
    
    Account toAccount = accountRepository.findById(toAccountId);
    toAccount.deposit(amount);     // B 계좌에 입금
}
```

**@Transactional이 하는 일:**

1. **메서드 시작 시** → 트랜잭션 시작, 데이터베이스 커넥션 획득
2. **메서드 정상 종료** → COMMIT (A 차감 + B 입금 모두 영구 저장)
3. **도중 에러 발생** → ROLLBACK (A 차감도 취소, B 입금도 취소 → 원상복구)

**시스템이 꺼지는 중간에 에러가 발생하면:**
```
송금 시작
├─ A 계좌: -100만원 (메모리에만 반영)
├─ ❌ 오류 발생!
└─ ROLLBACK → A 계좌 -100만원도 취소됨
```

결국 **둘 다 실패하거나 둘 다 성공하는 상태만 존재**합니다.

## 핵심 개념 정리

### 🔑 ACID 원칙 (Atomicity)
- **원자성(Atomicity)**: 트랜잭션은 더 이상 쪼갤 수 없는 단위
- 모든 작업이 완료되거나, 모두 취소되거나 둘 중 하나만 가능

### 🔄 @Transactional의 동작 방식
```java
// Spring AOP를 이용한 프록시 객체 생성
@Transactional
public void businessLogic() {
    // 1. TransactionManager.begin() 자동 호출
    // 2. 실제 메서드 실행
    // 3. TransactionManager.commit() 또는 rollback() 자동 호출
}
```

### ⚙️ 중요한 설정값들
```java
@Transactional(
    propagation = Propagation.REQUIRED,    // 기본값: 기존 tx 사용, 없으면 생성
    isolation = Isolation.DEFAULT,         // DB의 기본 격리 수준
    readOnly = false,                      // 쓰기 작업 가능
    timeout = -1,                          // 무제한
    rollbackFor = Exception.class           // 어떤 예외에 롤백할지
)
```

### 🚨 예외에 따른 동작
```java
@Transactional
public void transfer() {
    // ✅ Checked Exception → 기본적으로 COMMIT (롤백 안함)
    // ❌ Unchecked Exception (RuntimeException) → 기본적으로 ROLLBACK
}

// 명시적으로 설정
@Transactional(rollbackFor = Exception.class)  // 모든 예외에 롤백
@Transactional(noRollbackFor = SomeException.class)
```

### 🔍 격리 수준(Isolation Level)
| 레벨 | 설명 | 사용처 |
|------|------|-------|
| READ_UNCOMMITTED | 가장 낮은 격리 (Dirty Read 가능) | 거의 사용 안함 |
| READ_COMMITTED | 커밋된 데이터만 읽음 (기본값) | 대부분의 DB |
| REPEATABLE_READ | 트랜잭션 중 일관된 읽음 | MySQL InnoDB |
| SERIALIZABLE | 가장 높은 격리 (성능 저하) | 매우 중요한 작업 |

## 실무 포인트

### ✅ DO: 올바른 사용

```java
// 1️⃣ 서비스 계층에 적용 (권장)
@Service
public class TransferService {
    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        // 비즈니스 로직
    }
}

// 2️⃣ 필요한 경우에만 readOnly 최적화
@Transactional(readOnly = true)
public Account getAccount(Long id) {
    return accountRepository.findById(id);
}

// 3️⃣ 명시적 롤백 조건 설정
@Transactional(rollbackFor = BusinessException.class)
public void criticalOperation() {
    // 체크 예외도 롤백 필요시
}
```

### ❌ DON'T: 흔한 실수

```java
// ❌ 1. Controller에 적용 (X)
@RestController
@Transactional  // 서비스까지 레이어를 거치면서 불필요하게 길어짐
public class AccountController {
    // ...
}

// ❌ 2. 무분별한 범위 설정
@Transactional
public void largeOperation() {
    // 100만 건 데이터 처리 → 메모리 부하, DB 락 유지 시간 증가
    for (int i = 0; i < 1000000; i++) {
        process(i);
    }
}

// ✅ 개선: 배치 처리
@Transactional
public void largeOperation() {
    List<Integer> batch = new ArrayList<>();
    for (int i = 0; i < 1000000; i++) {
        batch.add(i);
        if (batch.size() == 1000) {
            processBatch(batch);
            batch.clear();  // 주기적으로 커밋
        }
    }
}

// ❌ 3. Private 메서드 적용 (무시됨)
@Service
public class OrderService {
    @Transactional
    private void updateInventory() {  // 프록시 적용 안됨
        // ...
    }
}
```

### 🎯 실제 은행 송금 예제

```java
@Service
@RequiredArgsConstructor
public class BankTransferService {
    private final AccountRepository accountRepository;
    private final TransactionHistoryRepository historyRepository;
    
    @Transactional(isolation = Isolation.SERIALIZABLE)  // 동시성 제어 강화
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        // 1. 계좌 잠금 조회
        Account fromAccount = accountRepository.findByIdWithLock(fromId);
        Account toAccount = accountRepository.findById(toId);
        
        // 2. 유효성 검증
        if (fromAccount.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException();  // RuntimeException
        }
        
        // 3. 송금 처리
        fromAccount.withdraw(amount);
        toAccount.deposit(amount);
        
        // 4. 거래 기록 (같은 트랜잭션 내)
        historyRepository.save(
            TransactionHistory.of(fromId, toId, amount)
        );
        
        // 메서드 종료 → 자동 COMMIT
        // 에러 발생 시 → 자동 ROLLBACK (모든 변경사항 취소)
    }
}
```

### 📊 성능 고려사항

```java
// 읽기 전용 쿼리는 최적화
@Transactional(readOnly = true)  // 플러시 모드 OFF, 쓰기 스냅샷 미생성
public List<AccountDto> getAccountsByCustomer(Long customerId) {
    return accountRepository.findByCustomerId(customerId);
}
```
