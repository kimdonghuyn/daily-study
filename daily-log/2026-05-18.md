### 1. 🔧 문제 상황

# GC 알고리즘 선택 - 모범 답안

## 모범 답안

**당신의 상황에서는 G1 GC를 선택해야 합니다.** 그 이유는 명확합니다:

### 각 GC의 특성 비교

| 구분 | Serial GC | G1 GC | ZGC |
|------|-----------|-------|-----|
| **처리 모델** | Single Thread | Multi Thread | Multi Thread |
| **Pause Time** | 수백ms~수초 | 수십ms (예측 가능) | 1ms 이하 |
| **메모리** | 작음 | 중간 | 중간~많음 |
| **적합 상황** | 개발/테스트, CPU 1~2개 | 4GB 이상, 수십개 코어 | 초저지연 필요 (금융/거래소) |
| **복잡도** | 낮음 | 중간 | 높음 |

### 당신의 상황 분석

```
상황 요소:
✗ Serial GC → "응답 시간이 불규칙하게 튀는" = Full GC 때문에 STW(Stop-The-World) 발생
             → 동시 사용자 처리 불가
             
✓ G1 GC → Heap을 Region으로 분할하여 일부만 수집
        → Pause Time 예측 가능 (99.9%이 설정값 이내)
        → 동시성 좋음
        
? ZGC → 과도한 선택. 당신은 "초저지연"이 아니라 "안정적인 응답"이 필요
```

**결론: G1 GC 선택 + 튜닝으로 현재 문제 해결 가능**

## 핵심 개념 정리

### 1️⃣ Serial GC (구세대 - Java 8 이전 기본값)

```java
// Serial GC 동작 원리
Young Generation GC (Minor GC):
┌─────────────────────────┐
│  [STW 시작]             │
│  1. 모든 스레드 정지    │
│  2. Young Gen에서       │
│     Reachable 객체 추적 │
│  3. 살아남은 객체 이동  │
│  4. 모든 스레드 재개    │
│  [STW 종료]             │
└─────────────────────────┘
⏱️ Pause: 50-100ms (작은 heap)
```

**문제점:**
- Single-threaded collector
- STW(Stop-The-World) 시간이 김
- 멀티코어 활용 불가
- 동시 사용자 많을수록 악화

```bash
# Serial GC 활성화 (권장 안함)
java -XX:+UseSerialGC -Xmx4g MyApp
```

### 2️⃣ G1 GC (현재 표준 - Java 9+ 기본값)

```java
// G1 GC 동작 원리
Heap 구조:
┌──────────────────────────────────────┐
│  Region 0   Region 1   Region 2 ... │
│  [S]        [O]        [S]           │ S=Survivor, O=Old
│  Young Gen 영역                      │
└──────────────────────────────────────┘

GC 사이클:
1. Concurrent Marking (애플리케이션 함께 실행)
   - Live object 추적
   - STW 없음 ⭐

2. Concurrent Cleanup (병렬)
   - 비용 계산
   
3. Young Generation Collection (STW)
   - 모든 Young Region 수집
   - Pause: 20-50ms (설정 가능)

4. Mixed Collection (선택적)
   - Old Region 일부 수집
   - Pause: 50-100ms
```

**장점:**
- Pause Time 예측 가능 (`-XX:MaxGCPauseMillis=200`)
- 멀티 스레드 병렬 처리
- 대용량 heap (4GB~) 최적화
- 현대적인 멀티코어 활용

```bash
# G1 GC 설정 (권장)
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \  # 최대 pause 목표
     -Xmx8g \
     -XX:InitiatingHeapOccupancyPercent=35 \  # concurrent marking 시작
     MyApp
```

### 3️⃣ ZGC (최신 세대 - Java 11+)

```java
// ZGC 동작 원리 - STW 극소화
특징:
┌────────────────────────────────┐
│  • Concurrent GC (대부분)      │
│  • STW는 1ms 미만 ⚡          │
│  • Color Pointer 기술 사용     │
│  • Numa 지원                   │
│  • 16TB 이상 heap 지원         │
└────────────────────────────────┘

Timeline:
─────────────────────────────────────►
[앱 실행] [GC 마킹] [앱 실행] [STW:1ms] [앱]
         (동시)                       (짧음)
```

**특성:**
- 매우 낮은 Pause Time (< 1ms)
- 메모리 오버헤드 높음 (30~50%)
- 복잡한 튜닝
- 초저지연 필요한 금융/거래 시스템용

```bash
# ZGC 활성화
java -XX:+UseZGC \
     -Xmx10g \
     MyApp
```

## 실무 포인트

### 🎯 당신의 쇼핑몰에 G1을 선택하는 이유

```java
// 문제 상황 재분석
"응답 시간이 불규칙하게 튄다" 
→ Full GC의 장시간 STW (1~3초)
→ 결제 중 GC 발생 = 타임아웃 위험

// Serial GC가 원인
예: 4GB 힙에서 Full GC = 800ms~2초 멈춤
   100명 사용자 중 몇 명은 응답 대기
```

### 📊 성능 비교 실제 사례

```
부하: 500명 동시 사용자, 8GB Heap

Serial GC:
├─ Minor GC: 50ms
├─ Full GC: 1.2초 (25~30초마다)
├─ 응답 시간: 100~200ms (평균 50ms)
├─ P99: 2000ms ⚠️ (Full GC 때문에 사용자 이탈)
└─ 처리량: 1,200 req/s

G1 GC (MaxPauseMillis=200):
├─ Young GC: 40ms
├─ Mixed GC: 150ms (병렬로 자주 실행)
├─ 응답 시간: 60~150ms
├─ P99: 300ms ✅
└─ 처리량: 2,800 req/s (2배 이상)
```

### 🔧 적용 단계

```bash
# Step 1: 현재 GC 상태 분석
java -XX:+PrintGCDetails \
     -XX:+PrintGCDateStamps \
     -XX:+PrintGCApplicationStoppedTime \
     -Xloggc:gc.log \
     MyApp

# gc.log에서 확인 사항
# - Full GC 빈도 (30초마다?)
# - Pause Time (1초 이상?)
# - Heap 사용률 패턴

# Step 2: G1 GC로 변경 (보수적 설정)

     -XX:MaxGCPauseMillis=200 \
     -XX:+PrintGCDetails \
