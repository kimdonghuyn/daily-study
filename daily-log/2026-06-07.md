### 1. 📖 개념 설명

# Spring Bean 생명주기 관리의 필요성

## 모범 답안

Spring이 Bean의 생명주기를 관리하는 이유는 **일관된 객체 상태 보장**, **리소스 효율성**, **의존성 주입의 신뢰성** 때문입니다.

만약 개발자가 직접 `new`로 객체를 생성했다면:

1. **리소스 누수 문제**
   - Database Connection, Thread Pool, File Handle 등이 제때 정리되지 않음
   - 메모리 누수 발생으로 OOM 에러 유발

2. **초기화/정리 코드의 산재**
   ```java
   // ❌ 문제 있는 방식
   UserRepository repo = new UserRepository();
   repo.connect(); // 매번 연결
   // ... 사용
   repo.close(); // 명시적으로 호출해야 함 (개발자가 까먹을 수 있음)
   ```

3. **일관되지 않은 객체 상태**
   - 같은 클래스의 인스턴스가 여러 개 생성되면서 싱글톤 패턴이 깨짐
   - 공유 리소스를 여러 인스턴스가 접근하면서 동시성 문제 발생

4. **의존성 주입 불가능**
   - 각 클래스에서 필요한 의존성을 직접 생성해야 함
   - 강한 결합도로 인해 테스트 어려움

5. **중복된 객체 생성**
   ```java
   // ❌ 비효율적
   UserRepository repo1 = new UserRepository();
   UserRepository repo2 = new UserRepository();
   UserRepository repo3 = new UserRepository();
   // 같은 객체를 여러 번 생성, 리소스 낭비
   ```

**Spring의 해결책:**
```java
// ✅ Spring이 관리
@Component
public class UserRepository {
    @PostConstruct  // Bean 생성 후 초기화
    public void init() {
        this.connect();
    }
    
    @PreDestroy     // Bean 소멸 전 정리
    public void cleanup() {
        this.close();
    }
}

// 싱글톤으로 관리되며, 생명주기도 자동 관리
@Autowired
private UserRepository repo;  // 항상 동일한 인스턴스
```

## 핵심 개념 정리

### 1. **Bean 생명주기 단계**
- **인스턴스화(Instantiation)**: `new`를 통한 객체 생성
- **의존성 주입(Dependency Injection)**: `@Autowired` 설정
- **초기화(Initialization)**: `@PostConstruct`, `InitializingBean.afterPropertiesSet()`
- **사용(In Use)**: 빈이 애플리케이션에서 활용
- **소멸(Destruction)**: `@PreDestroy`, `DisposableBean.destroy()`

### 2. **Bean 스코프 (중요한 4가지)**

| 스코프 | 설명 | 사용 시기 |
|--------|------|---------|
| **Singleton** | 애플리케이션 전체에서 1개만 생성 (기본값) | 상태가 없는 서비스 |
| **Prototype** | 매번 요청할 때마다 새로운 인스턴스 생성 | 상태가 있는 객체 |
| **Request** | HTTP 요청마다 새로운 빈 생성 | 웹 요청별 독립적 처리 |
| **Session** | HTTP 세션마다 새로운 빈 생성 | 사용자 정보 저장 |

```java
// Singleton (기본)
@Component
public class UserService { }

// Prototype
@Component
@Scope("prototype")
public class UserDTO { }

// Request
@Component
@Scope("request")
public class RequestContext { }
```

### 3. **리소스 관리 방식**

```java
// 방식 1: @PostConstruct, @PreDestroy (권장)
@Component
public class DataSource {
    @PostConstruct
    public void init() { connection.open(); }
    
    @PreDestroy
    public void cleanup() { connection.close(); }
}

// 방식 2: InitializingBean, DisposableBean
@Component
public class DataSource implements InitializingBean, DisposableBean {
    @Override
    public void afterPropertiesSet() { }
    
    @Override
    public void destroy() { }
}

// 방식 3: 빈 설정에서 init-method, destroy-method 지정
@Bean(initMethod = "init", destroyMethod = "cleanup")
public DataSource dataSource() { }
```

### 4. **왜 Singleton이 안전한가?**
- 한 번만 생성되므로 초기화 비용 절감
- 메모리 효율적 (공유 인스턴스)
- 스레드 안전성만 보장하면 안정적
- 상태가 없는 서비스에 이상적

## 실무 포인트

### 1️⃣ **DB Connection Pool 관리 예제**
```java
@Component
public class ConnectionPool {
    private HikariDataSource dataSource;
    
    @PostConstruct
    public void init() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");
        config.setMaximumPoolSize(10);
        this.dataSource = new HikariDataSource(config);
        // ✅ 애플리케이션 시작 시 한 번만 초기화
    }
    
    @PreDestroy
    public void cleanup() {
        if (dataSource != null) {
            dataSource.close();
            // ✅ 애플리케이션 종료 시 정리
        }
    }
    
    public DataSource getDataSource() {
        return dataSource;
    }
}
```

### 2️⃣ **Singleton vs Prototype 선택 기준**
```java
// ✅ Singleton: 상태가 없는 서비스
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository;
    
    public User getUser(Long id) {
        return userRepository.findById(id);
    }
    // 메서드만 있고 상태(인스턴스 변수)가 없음
}

// ✅ Prototype: 상태가 있는 요청별 객체
@Component
@Scope("prototype")
public class UserContext {
    private Long userId;      // 상태 보유
    private LocalDateTime requestTime;
    // 매 요청마다 새로운 인스턴스 필요
}
```

### 3️⃣ **스프링 부트에서의 자동 관리**
```java
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        // ✅ Spring이 자동으로 모든 Bean 생명주기 관리
        SpringApplication.run(Application.class, args);
        // 종료 시에도 모든 @PreDestroy 메서드 호출
    }
}
```

### 4️⃣ **실전 체크리스트**
```
☑️ 리소스(DB, 파일, 스트림)를 사용하는 Bean은 @PreDestroy로 정리
☑️ 상태가 없는 객체는 Singleton으로 (메모리 절약)
☑️ HTTP 요청별로 격리해야 할 데이터는 Request 스코프 사용
☑️ 초기화가 필요한 경우 @PostConstruct 활용
☑️ 순환 의존성이 있으면 ObjectProvider나 Lazy 사용
☑️ Prototype Bean을 Singleton에 주입할 때는 ObjectProvider 사용
```
